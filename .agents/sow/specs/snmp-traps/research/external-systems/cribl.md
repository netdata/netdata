## 1. System Overview & Lineage

Cribl Stream (formerly LogStream — rename announced by Cribl on 2022-09-21 at Cribl.Live; see `https://cribl.io/press-releases/cribl-renames-logstream-to-stream-and-positions-itself-as-the-data-engine-for-it-and-security/`, Vendor-documented) is the flagship product of Cribl, Inc. It is a **vendor-neutral observability pipeline** sitting between data producers (agents, applications, network devices) and observability/analytics destinations (Splunk, Elasticsearch, S3, Kafka, Datadog, Snowflake, and so on). The product's defining promise is to decouple "where data comes from" from "where data ends up" — operators can reshape, enrich, filter, redact, sample, and reroute data without changing producers or consumers.

Cribl Stream is closed-source. There is no public source code repository for the daemon itself (the public `criblio` GitHub organisation publishes SDKs, terraform providers, helm charts, packs, and tooling — none of which contain the running engine). What is public, and what this spec consumes:

1. The **OpenAPI specification** at `.speakeasy/out.openapi.yaml` in the public Go SDK repository. Cribl uses Speakeasy to auto-generate the SDK from its product OpenAPI, so this YAML is the literal control-plane contract.
2. The **generated SDK code and per-type Markdown** for Go, Python, and TypeScript.
3. The **operator-facing documentation** at `https://docs.cribl.io/stream/`.

Where SNMP traps fit in Cribl's worldview:

- Cribl treats SNMP traps as **just another kind of event-shaped input** in the broader observability-pipeline catalogue. The Source SNMP-Trap input is one of ~90 Source types (alongside Splunk TCP/HEC, Syslog, Kafka, S3, OpenTelemetry, Prometheus, AWS CloudWatch, Azure Event Hub, NetFlow, etc., per the OpenAPI `InputCreateExamples*` entries near line 49249-49368).
- Cribl treats trap **emission** as just another kind of output. The Destination SNMP output is one of ~70 Destination types (per `OutputCreateExamples*` near line 49656; "snmp" is one element in the enumerated output-type discriminator at line 42703).
- A dedicated **pipeline function**, `snmp_trap_serialize`, exists to re-shape an in-pipeline event back into a wire-compatible trap; this is required when the pipeline has modified the original trap and the destination is another SNMP listener (the function is documented at `https://docs.cribl.io/stream/snmp-trap-serialize-function/`).

The single most important architectural fact: **Cribl Stream is the pipeline tier; it is not an NMS, a SIEM, or a metrics database.** It does not store traps beyond its persistent queue (which exists only as a backpressure cushion for the next outbound hop). It does not alert on trap content. It does not visualize trap content. It does not maintain a trap event index, ack/clear lifecycle, severity model, or correlation engine. All of those concerns belong to whichever destination the operator picks (Splunk, Elasticsearch, Datadog, Cribl Lake, an SIEM, an NMS, another Cribl Stream sink, etc.). This is the inverse of OpenNMS/Zenoss/Centreon/Zabbix, which couple trap reception tightly to alert and storage subsystems.

Relationship to upstream tools:

- **No embedded MIB compiler**. Unlike Telegraf (`gosmi`), Logstash (`SNMP::TrapListener` + `smidump`), or OpenNMS (in-house `snmpinfo`), Cribl Stream does not ship a MIB parser. OID-to-name resolution is offloaded to either (a) operator-supplied CSV lookup files consumed by the Cribl **Lookup** pipeline function, or (b) the community `cribl-snmp-trap-varbind-translation` pack (see `https://packs.cribl.io/packs/cribl-snmp-trap-varbind-translation`), which ships ~168 CSV lookups derived from ~15,000 upstream MIBs.
- **No reliance on Net-SNMP `snmptrapd`**. Cribl's trap listener is its own first-party UDP/162 server (terminating decode and SNMPv3 USM in the Worker Process), in contrast to Nagios/SNMPTT or Sensu's `snmptrapd2sensu` bridge, which both shell out to Net-SNMP's `snmptrapd`.
- **No SNMPTT, no syslog bridge**. The SNMP trap path is direct UDP-to-event.
- **Cribl Edge** (sister product) ships the same SNMP Trap Source surface as Cribl Stream (per `https://docs.cribl.io/edge/sources-snmp-traps`); the difference is positioning — Edge is the lightweight per-host agent, Stream is the central pipeline tier.

## 2. Trap-Subsystem Architecture

There is no separate "trap subsystem"; the trap surfaces are three first-class components woven into the same Worker Process event loop that handles every other input/function/output. The end-to-end architecture is:

```
   +-----------------------------------------------------+        the Worker often runs unprivileged)
```

Deployment model (Vendor-documented):

- **Single-process per Worker Node, multi-process within**. Each Worker Node hosts one **API Process** (talks to the Leader) and N **Worker Processes** (one per CPU core by default, all running the data pipeline). This is documented in the Worker architecture: `https://docs.cribl.io/stream/deploy-distributed`.
- **Distributed deployment** is the production shape: one Leader + N Worker Nodes (in Cribl-managed Cribl.Cloud, hybrid customer-managed, or fully self-hosted). Workers poll the Leader every 10 s for config. SNMP Trap is a **Push-type Source** (per `https://docs.cribl.io/stream/sources-snmp-traps`); each Worker Process listens for UDP traffic on the configured port.
- **No documented data-plane HA on UDP**. No Cribl documentation describes native UDP HA, leader-elected UDP listener, or automatic fan-out for the SNMP Trap Source; the Known Issues page's recommendation of external LBs (cited below) is the only guidance on the topic.
- **Leader HA exists for the control plane** (primary/standby Leaders, per `https://docs.cribl.io/stream/deploy-add-second-leader`) but does NOT affect trap UDP ingestion — Workers continue to process traps even if the Leader is down, but configuration changes cannot be made.
- **UDP HA is operator's responsibility**: docs explicitly recommend "deploy a hybrid Stream Worker Group with customer-managed Worker Nodes as close as possible to the UDP senders" as the workaround for UDP loss (per the Known Issues page at `https://docs.cribl.io/stream/4.3/known-issues`). An external L4 load balancer (AWS NLB, F5, keepalived VIP, anycast) in front of multiple Workers is the canonical pattern.
- **Cribl.Cloud SaaS** runs the same daemon in Cribl-managed infrastructure with automatic Leader HA. Cribl.Cloud's documented Known Issue is that **high-volume UDP can drop in the Cloud tier** (per the same Known Issues page), so customer-managed Workers are recommended for UDP-heavy workloads.

Languages and key technical choices (**Inferred** — daemon source is closed; the items below are independent inference from observable behaviour, NOT Cribl-documented):

- **Runtime**: likely Node.js. Independent inference from (a) JavaScript-syntax filter expressions evaluated against events (Routes filters, Eval Functions, `keyExpr` arguments — `https://docs.cribl.io/stream/routes`), (b) the `__filename` field on `FunctionSnmpTrapSerialize` (`.speakeasy/out.openapi.yaml:5564`), and (c) the deployment doc's references to per-process behaviour consistent with Node clusters. Cribl's public engineering material and job postings mention Node.js, supporting but not proving the inference. **Confidence: moderate**, not vendor-confirmed in product docs.
- **SNMP library**: not publicly identified. Speculation: given the Node.js inference, the library is likely `net-snmp` (`https://www.npmjs.com/package/net-snmp`) or a fork — but this is **unverifiable** without daemon source access. The API surface (auth/priv enum names: `none`, `md5`, `sha`, `sha224`, `sha256`, `sha384`, `sha512` and `des`/`aes`/`aes256b`/`aes256r`) is consistent with `net-snmp`'s capability set, but several other Node.js SNMP libs also expose these names. **Confidence: low.**
- **Multi-Worker-Process UDP fan-out**: on Linux, when multiple processes bind the same UDP port, the kernel uses `SO_REUSEPORT` to round-robin datagrams across listeners. Whether Cribl Stream's Worker Processes actually bind the same UDP socket with `SO_REUSEPORT`, or whether one Process owns the socket and dispatches internally, is **NOT** stated in any public doc. The architecture diagram below shows "Worker Process(es) bound to UDP/162" as the documented endpoint; the per-packet dispatch mechanism is unverified. **Confidence: low.**
- **Inter-component IPC**: events flow inside the same Worker Process between Source → Routes → Pipeline → Destination. Cross-process communication is the Leader-Worker config channel (HTTPS REST, polling), not the event path. This is implied by the deployment doc but not literally stated.

## 3. Trap Reception (UDP/162 Ingress)

Trap reception is implemented by the SNMP Trap **Source** (`type: snmp` in OpenAPI). It is a UDP listener that decodes the PDU and emits one Cribl event per trap.

Bind and transport:

- `host` is the bind address. Documented default `0.0.0.0` (IPv4 all) per the SDK doc: `inputsnmpinput.md:18` ("Address to bind on. For IPv4 (all addresses), use the default '0.0.0.0'. For IPv6, enter '::' (all addresses)…"). The OpenAPI schema confirms `host` is required (line 20084-20085, 20129-20132).
- `port` is the UDP listening port; required field. The schema declares `maximum: 65535` (no minimum) and the doc string says "UDP port to receive SNMP traps on. Defaults to 162." (`.speakeasy/out.openapi.yaml:20133-20137`). NOTE: the OpenAPI `InputCreateExamplesSnmp` snippet at line 49249-49257 sets `port: 161` — this is an upstream documentation bug (161/UDP is the SNMP polling port; 162/UDP is the trap port). The narrative doc is correct (162); the example is wrong. The stock pre-shipped Source on on-prem installations listens on UDP/9162 (per `https://docs.cribl.io/stream/sources-snmp-traps`) because Workers commonly run unprivileged.
- `udpSocketRxBufSize` sets `SO_RCVBUF` directly (`inputsnmpinput.md:24`, `.speakeasy/out.openapi.yaml:20191-20196` — range 256 to 4,294,967,295 bytes). This is the standard kernel-receive-buffer knob; sizing it well is the primary defence against burst-driven UDP drops.
- `maxBufferSize` caps the in-process per-Source event buffer (`inputsnmpinput.md:21`, `.speakeasy/out.openapi.yaml:20176-20180`). When downstream Functions or the Routes table back up, this is the high-water mark before the Source either drops or engages PQ.

UDP socket semantics:

- The exact socket-API used by the Source is **not publicly documented**. Independent inference (consistent with the Node.js runtime hypothesis in §2) is that Cribl uses Node's `dgram` API; this is unverified. The per-worker fan-out mechanism (single-listener-with-internal-dispatch vs `SO_REUSEPORT` round-robin) is likewise unverified — see §2. The only Cribl-level reception filter exposed in the schema is `ipWhitelistRegex` (a regex on the source IP, no schema default; the doc string at `inputsnmpinput.md:22` and `.speakeasy/out.openapi.yaml:20181-20184` does not declare one — operators commonly use `.*` as an effective wide-open setting).
- **TLS is not available** on the Source (`https://docs.cribl.io/stream/sources-snmp-traps` lists `TLS Support: No`, Vendor-documented). This is correct for UDP/162 (SNMP-over-TLS via DTLS RFC 5953 is rarely deployed; almost no production traps use it).

Decode and trap-shape:

- The Source decodes all three SNMP versions (v1, v2c, v3) — the OpenAPI schema does not include an explicit version selector; version is auto-detected from the PDU header (consistent with how every SNMP library handles it). The decoded version is surfaced on the output event as `__snmpVersion` (integer 0/2/3, per `https://docs.cribl.io/stream/sources-snmp-traps`).
- **SNMPv1 trap-PDU specifics** (enterprise OID, generic-trap, specific-trap, agent-address) are not visible as discrete top-level fields in the documented event schema. It is plausible that Cribl performs the standard v1-to-v2c bridge (RFC 3584 §3) so these PDU fields become varbinds in the output, but this is **NOT explicitly documented** at `https://docs.cribl.io/stream/sources-snmp-traps` or in the SDK markdown. The `__snmpRaw` byte buffer preserves the original PDU losslessly for any operator-side decoding. **Confidence: low** (inferred from standard SNMP practice, not Cribl-confirmed).
- **SNMPv2c InformRequest**: the protocol requires an InformResponse acknowledgement from the receiver to the sender. The OpenAPI schema exposes **no knob** to disable, tune, or even acknowledge configuration of InformRequest handling. Cribl docs do not state whether the listener responds with InformResponse. Two possibilities cannot be distinguished from public material: (a) the listener acknowledges unconditionally; (b) the listener silently consumes the InformRequest without responding. The claim that "InformRequest is decoded the same as Trap-PDU and emitted as an event" is **inferred** from the schema's lack of a separate inform path. **Confidence: low**; cannot be verified without source.
- **SNMPv3 USM**: per-user records (`InputSnmpV3User` with `name`, `authProtocol`, `authKey`, `privProtocol`, `privKey` — see `inputsnmpv3user.md`; the outer wrapper `InputSnmpSNMPv3Authentication` carries `v3AuthEnabled`, `allowUnmatchedTrap`, `v3Users`, per `inputsnmpsnmpv3authentication.md`). Multi-user (`minItems: 1`, `.speakeasy/out.openapi.yaml:20156`) so several USM users can be configured at once on the same Source. The `allowUnmatchedTrap` field controls whether v3 traps that don't match any configured user are passed through (without decryption) or dropped (per `.speakeasy/out.openapi.yaml:20148-20151`); the OpenAPI schema does not encode a default, but the Source docs page at `https://docs.cribl.io/stream/sources-snmp-traps` documents the default as **off** (Vendor-documented — unmatched v3 traps are dropped). Cribl's `snmpV3Auth` description recommends "Set the log level to debug if you are experiencing authentication or decryption issues" (`.speakeasy/out.openapi.yaml:20141`), implying operator-side validation via Worker debug logs.
- **SNMPv3 context engine ID / context name / local engine ID — NOT supported**. The `InputSnmp` schema (`.speakeasy/out.openapi.yaml:20080-20219`) and `InputSnmpV3User` schema contain no `contextEngineId`, `contextName`, `localEngineId`, or equivalent fields. Per RFC 3411, USM operates within a context-scoped engine ID; Cribl's v3 surface appears limited to USM user-level auth/priv without per-context or fixed-engine-ID configuration. This is a real comparability gap versus Logstash v4.1.0+ (which exposes `context_engine_id`, `context_name`) and Datadog Agent (which derives an engine ID from hostname via FNV-128). **Confidence: high** (schema-verified absence).

Varbind parsing modes:

- `varbindsWithTypes` (boolean). The OpenAPI schema does not declare a default (`.speakeasy/out.openapi.yaml:20197`); the public Source docs page at `https://docs.cribl.io/stream/sources-snmp-traps` states the default is **off** (Vendor-documented). When **on**, the `varbinds` field is an array of `{oid, value, type}` objects where `type` is the BER tag number (numeric — e.g. 67 for TimeTicks, 6 for OID, 64 for IpAddress per RFC 1155/1213 BER ASN.1 tags). When **off** (default), `varbinds` is a flat object map of `oid -> value`, losing per-varbind type info. The SNMP Trap Serialize pipeline function **requires** `varbindsWithTypes: true` (per `https://docs.cribl.io/stream/snmp-trap-serialize-function/`, Vendor-documented), so any pipeline that needs to emit traps must turn this on at the Source.
- `bestEffortParsing` (boolean). No default declared in the OpenAPI schema (`.speakeasy/out.openapi.yaml:20201`) or in `inputsnmpinput.md:26`. Per `inputsnmpinput.md:26`: "If enabled, the parser will attempt to parse varbind octet strings as UTF-8, first, otherwise will fallback to other methods" (SDK-verified). Note: the characterization that this addresses "the perennial trap-payload ambiguity" (vendor MIBs declare `OCTET STRING` for both human-readable text and packed binary) is **independent analysis**, not a Cribl-sourced claim. The Cribl-documented behaviour is "UTF-8 first, fallback to other methods"; the "other methods" are not enumerated. **Confidence: low** on the precise fallback chain.

Per-event emitted fields (after Source decode). The Cribl Source docs page at `https://docs.cribl.io/stream/sources-snmp-traps` enumerates the **internal `__`-prefixed meta-fields** the Source attaches — these are Vendor-documented. The non-internal top-level fields (`host`, `community`, `source`, `varbinds`) are not enumerated by the Source page as a list but are **observable in events** emitted from the Source and are universally referenced in Cribl tutorials and pack documentation; they are best treated as Vendor-documented via the broader Cribl event-model docs rather than the SNMP Source page specifically.

Vendor-documented internal meta-fields (`__`-prefixed, Source docs page):

- `__snmpRaw` — opaque buffer of the original wire PDU. Crucial: this is what enables the "forward verbatim to another SNMP listener" semantics.
- `__snmpVersion` — integer; 0 = v1, 2 = v2c, 3 = v3. (Note: SNMP "v2u" and "v2p" historic variants are skipped; this is consistent with industry practice and is not unique to Cribl.)
- `__srcIpPort` — `"${srcIP}|${srcPort}"` formatted string.
- `__didDecrypt` — true only on a v3 trap that was successfully decrypted using a configured user (false for v3 traps that passed through via `allowUnmatchedTrap` or for v1/v2c traps).
- `__inputId` — the Source's own id; useful for Routes-table filters that distinguish multiple SNMP Sources by identifier (`__inputId == 'in_snmp_trap_<source-id>'`).
- `__final` — Cribl internal flag used by the Routes table to mark events that have been finalised by a `Final=on` Route. Documented on the Source page as one of the internal fields present on emitted SNMP-trap events.
- `_time` — event timestamp.

Vendor-documented (Source page) varbind shape:

- `varbinds` — either a flat object map `{oid: value}` (when `varbindsWithTypes` is off) or an array `[{oid, value, type}]` (when `varbindsWithTypes` is on). The shape is determined by the `varbindsWithTypes` toggle described above.

Observable / generic event fields (present on emitted events but NOT enumerated on the Source page; framing is **inferred** from generic Cribl event-model behaviour and from blog/pack examples):

- `host` — source host (in practice the device IP that sent the trap; Cribl's generic event model uses `host` for sender identity).
- `community` — SNMP community string from the PDU (v1/v2c only, meaningless for v3). Operators routinely filter on `community` in Routes; the field's existence is presumed from blog/pack examples and the Lookup function's use of it. **Confidence: moderate** — universally referenced but not Source-page-enumerated.
- `source` — Cribl's generic event-source identifier (present on events from many Source types, not SNMP-specific).

Internal queueing:

- A bounded in-memory buffer (`maxBufferSize`). The OpenAPI schema (`.speakeasy/out.openapi.yaml:20176-20180`) declares no default; the Source docs page at `https://docs.cribl.io/stream/sources-snmp-traps` states the documented default is **1000 events** (Vendor-documented).
- A **Persistent Queue (PQ)** can be enabled on the Source (`pqEnabled` boolean, with optional `pq` subobject of `PqType`; per `inputsnmpinput.md:14, :17`). PQ is **strongly recommended** for UDP sources because UDP senders cannot backoff. Mode "Always On" (the default since v4.1) writes every event to disk before downstream; mode "Smart" engages only on backpressure. Cribl.Cloud forces Always On with a fixed 1 GB per Source per Worker Process and no compression (per `https://docs.cribl.io/stream/persistent-queues-sources`).

Reception filters:

- `ipWhitelistRegex` — JavaScript regex matched against the source IP literal. The OpenAPI schema does not declare a default (`.speakeasy/out.openapi.yaml:20181`); the Source docs page at `https://docs.cribl.io/stream/sources-snmp-traps` states the default is **`.*`** (match all IPs) — Vendor-documented. This is the **only access control on inbound traps for v1/v2c** (which have no auth besides the community string); for v3, USM additionally enforces auth+priv.
- `disabled` — boolean to suspend the entire Source.
- `environment` — Git-branch-scoping (a Cribl GitOps mechanism), so the Source can be active only in certain config branches.

Listener lifecycle:

- The Source belongs to a Worker Group; the Leader pushes Source config; the Worker Processes start/stop the listener atomically on config reload. No explicit listener-reload knob is exposed.
- There is no "global trap-rate cap" knob on the Source — backpressure is via downstream buffering and PQ, not by dropping at the wire.

## 4. MIB Management

Cribl Stream **does not embed a MIB compiler or any built-in OID-to-symbolic-name resolution**. This is its single biggest gap relative to NMS-class systems (OpenNMS, Zenoss, Centreon, LibreNMS, Zabbix) and even relative to lighter-weight peers (Telegraf via `gosmi`, Logstash via the Ruby `snmp` gem + `smidump`).

What ships in-product:

- Nothing. The `__snmpRaw` buffer is exposed losslessly, and the `varbinds` array carries numeric OIDs and primitive-typed values. No symbolic names. No MIB store. No `smilint`-style compile pipeline.

What the operator does instead:

1. **Cribl Lookup files** (CSV). Cribl Stream has a `Lookup` pipeline function that consumes a CSV with a key column and N value columns, matched on a configurable field of the event. Operators precompute the CSV externally (from MIBs) and import it.
2. **The community pack `cribl-snmp-trap-varbind-translation`** (per `https://packs.cribl.io/packs/cribl-snmp-trap-varbind-translation`, Vendor-documented; pack-page metrics retrieved 2026-05-22):
   - Ships **168 CSV lookup files** parsed from over **15,000 upstream MIBs** containing **~92,000 trap OIDs**, compressed to ~10 MB. (The pack page metric for "OID count" has fluctuated between roughly 92K and 97K across pack versions; the value used in this analysis is the 2026-05-22 retrieval. Pack maintainers do not publish a per-version OID-count changelog.)
   - Schema: `oid, oidName, mibName, objects` where `objects` is a pipe-delimited, OID-ordered list of varbind names that should be hoisted as `trap_*` fields.
   - Pipeline shape: a **Lookup** function (match incoming trap OID against the CSV) followed by a **Code** function (a JS-based equivalent of Eval) that derives `trap_*` fields from the matched `objects` list and the original `varbinds` array.
   - Operators add missing MIBs **manually** by extending the CSVs in the same format. The pack page is explicit: "While Cribl does not currently support automatic MIB parsing, missing MIBs can be manually parsed and added to the lookups following the format in the other files."
   - The pack documentation calls out **two external tools** for converting fresh MIBs into CSV form: `snmptranslate` (from Net-SNMP) and `pysnmp` (the Python SNMP library). Neither is shipped by Cribl; the operator wires them up themselves.
3. **The Cribl blog post** at `https://cribl.io/blog/unravelling-snmp-trap-varbind-translation-with-cribl/` documents the recommended workflow: dump MIBs via `snmptranslate` or `pysnmp`, hand-build a CSV per MIB family, drop it into the pack's lookup directory, restart the pipeline.

The implication is severe for unattended deployments:

- Out of the box, traps are decodable but **opaque**. Operators see numeric OIDs like `1.3.6.1.4.1.9.9.41.1.2.3.1.5` and must memorise what they mean.
- "Just bring your MIBs" is not a workflow — Cribl deliberately offloads MIB compilation to external tooling.
- The 168-CSV pack is the practical bridge, but it is community-maintained and lags upstream MIB updates.

Comparison points:

- vs **Telegraf**: Telegraf runs `gosmi` in-process and parses standard `.mib` files at startup. Adding a MIB is a file-copy + restart. Cribl requires a CSV pre-derivation step.
- vs **Logstash**: Logstash's `snmptrap` input uses `smidump` (a libsmi CLI) to compile MIBs at startup. Same UX as Telegraf.
- vs **OpenNMS / Zenoss / Centreon**: each ships a full MIB management UI/store with thousands of MIBs preloaded. Cribl ships zero.
- vs **Splunk SC4SNMP**: SC4SNMP preloads ~500 standard MIBs and runs `pysnmp` MIB compilation in-cluster. The dependency on `pysnmp` is shared with Cribl's recommended workflow; the difference is that SC4SNMP automates the compilation step.

The design is internally consistent: Cribl Stream's job is to be the pipeline tier. Its job is not to be authoritative about device semantics. The MIB-management problem is pushed downstream to whichever destination is the actual NMS, or upstream to the CSV-prep workflow.

## 5. Trap Processing Pipeline

The processing pipeline is the heart of Cribl Stream and is what makes its trap support fundamentally different from a Net-SNMP-style trap daemon. After the Source decodes a trap into a Cribl event, the event flows through:

1. **(Optional) pre-processing Pipeline attached to the Source.** A first pass of Functions that runs before the Routes table sees the event. Typical use: hoist `__snmpRaw` semantics, mark trap origin, drop malformed traps early.
2. **The Routes table** (`https://docs.cribl.io/stream/routes`). A first-match-wins list of `{filter, pipeline, output, final}` rows; the filter is a JS-syntax expression evaluated against the event. SNMP-trap events typically match on `__inputId == 'in_snmp_trap'` or on a varbind OID. The `final` toggle (default on) stops further Route evaluation; turning it off forks a clone to subsequent Routes.
3. **The chosen processing Pipeline.** A sequence of Functions. Functions Cribl ships that are relevant to trap processing include:
   - **Eval** — add/remove/rename fields, set values from JS expressions; the workhorse for SNMP MIB-derived enrichment via lookup tables.
   - **Lookup** — match a field against a CSV/JSON file; the workhorse for OID-to-name resolution per §4.
   - **Drop** — discard events matching an expression; the storm-handling primitive.
   - **Suppress** — drop duplicates within a time window keyed on an expression (`https://docs.cribl.io/stream/4.x` documents `Suppress`); the dedup primitive (see §10).
   - **Regex Extract** — pattern-extract from a string field; used to parse human-readable OCTET STRING varbinds.
   - **Parser** — structured parsing for known formats; rarely applicable to traps directly but useful if a varbind embeds JSON/XML.
   - **Code / JavaScript** — full JS expression block for complex logic.
   - **snmp_trap_serialize** — re-serialise the in-pipeline event back into an SNMP-wire-format PDU. Required when the pipeline has modified varbinds and a downstream SNMP destination is expected to receive a real trap (rather than a JSON dump of one). See §5.2 below.
4. **(Optional) post-processing Pipeline attached to the Destination.** Last-mile shaping (compression hints, header injection, batch keys).
5. **Destination.** Either the **SNMP Trap Destination** (verbatim forwarding of `__snmpRaw`, or re-serialised PDU from `snmp_trap_serialize`) or any of ~70 non-SNMP destinations (Splunk, Elastic, Kafka, S3, Datadog, Syslog, HTTP, …). Non-SNMP destinations receive the event as JSON-like records, not as wire-format traps.

### 5.1 The `snmp_trap_serialize` pipeline function — strict & permissive modes

Cribl ships a first-party, in-pipeline trap **re-serializer** function. Scope and constraints (per `https://docs.cribl.io/stream/snmp-trap-serialize-function/` and `https://docs.cribl.io/stream/sources-snmp-traps`, Vendor-documented):

- The function works on events that **originated from an SNMP Trap Source** and have been modified by the pipeline; it re-builds the wire-format PDU from the in-event fields. It is not a general-purpose "any event to SNMP trap" emitter.
- The Source must be configured with `varbindsWithTypes: true` so each varbind carries an explicit BER type — the serializer cannot infer types from values.
- The SNMP Trap **Destination** can only receive events that have a valid `__snmpRaw` buffer (verbatim path) or have been processed by `snmp_trap_serialize` (re-serialised path). Cribl docs explicitly state **non-SNMP input cannot be sent to SNMP Destinations** — the Destination has no synthesis path for events without SNMP provenance.

The schema is `FunctionConfSchemaSnmpTrapSerialize` in `.speakeasy/out.openapi.yaml:5525-5560`:

- `strict` (bool, default true on the docs page): "Prevent event serialization if any required fields are missing. When disabled, @{product} will attempt to serialize the event even if required fields are missing, which could cause unexpected behavior at the downstream receiver." (`.speakeasy/out.openapi.yaml:5528-5531`).
- `dropFailedEvents` (bool, default true on the docs page): "When disabled, `snmpSerializeErrors` will be set on the event, and the `__snmpRaw` field will be removed to prevent @{product} from sending the event from the SNMP Trap Destination." (`.speakeasy/out.openapi.yaml:5532-5535`).
- `v3User` (object, optional) — used when the function is emitting a v3 trap. Discriminated union by `authProtocol`:
  - `authProtocol == "none"` → `SnmpTrapSerializeV3UserAuthProtocolNone` (no key fields required).
  - `authProtocol ∈ {"md5", "sha", "sha224", "sha256", "sha384", "sha512"}` → `SnmpTrapSerializeV3UserAuthProtocolNotNone` (requires `authKey`, `name`; carries a nested `privProtocol` discriminator).
  - When `privProtocol == "none"`, no priv key is required; when `privProtocol ∈ {"des", "aes", "aes256b", "aes256r"}`, `privKey` is required (per `.speakeasy/out.openapi.yaml:744-759`, `760-796`).

Required event-side fields for `snmp_trap_serialize` to succeed in strict mode (per `https://docs.cribl.io/stream/snmp-trap-serialize-function/`):

| Varbind name | OID | BER type | Notes |
|--------------|-----|----------|-------|
| `sysUpTime`   | `1.3.6.1.2.1.1.3.0`     | `67` (TimeTicks) | RFC 3416 §4.2.6 mandate for v2c notifications |
| `snmpTrapOid` | `1.3.6.1.6.3.1.1.4.1.0` | `6` (OID)        | RFC 1905 §3.1.2 mandate |

Plus the event fields the Source set at decode time (see §3 for the documented set). The exact set of fields beyond `sysUpTime` and `snmpTrapOid` that strict mode demands is **not exhaustively enumerated** in the public Cribl docs; operators identify missing fields by running the function in non-strict mode and inspecting `snmpSerializeErrors`. Without the required varbinds, strict mode rejects the event and either drops it (`dropFailedEvents: true`) or passes it through with `snmpSerializeErrors` populated and `__snmpRaw` stripped (`dropFailedEvents: false`).

**Failure interaction with the SNMP Trap Destination**: when `dropFailedEvents: false`, the function strips `__snmpRaw` and sets `snmpSerializeErrors`. The Destination is documented as a verbatim forwarder of `__snmpRaw` (Vendor-documented at `https://docs.cribl.io/stream/destinations-snmp-traps`); a failed event therefore reaches the Destination without a wire PDU to replay. The Destination's behaviour on a missing `__snmpRaw` is **not documented**: it may drop the event, log an error, or hold it in the output queue. Operators relying on `dropFailedEvents: false` for visibility should route the failed events to a non-SNMP destination (e.g. Splunk, Elastic, S3) where `snmpSerializeErrors` can be queried. **Confidence: low** on the Destination's exact failure path.

### 5.2 The "preserve original source IP" idiom

A subtle pipeline pattern: when traps are forwarded to a downstream SNMP destination, the destination by default sees Cribl Stream's IP, not the original device's IP. The trap-serialise documentation page recommends a pre-serialise Eval Function that appends:

```javascript
{oid: '.1.3.6.1.6.3.18.1.3', type: 64, value: __e.host}
```

— that is, the `snmpTrapAddress` OID (RFC 3584 §3.1, public standard) with BER type 64 (IpAddress, RFC 1155 APPLICATION 0) and value taken from `__e.host` (the source IP). Cribl's specific endorsement of this exact idiom is documented at `https://docs.cribl.io/stream/snmp-trap-serialize-function/` (**Vendor-documented**, not SDK-schema-verified — the OpenAPI does not list `snmpTrapAddress` as a recognised output field). Without this idiom, the original source IP is lost to the downstream listener.

### 5.3 v3 outbound re-shaping — `msgFlags` and `msgSecurityParams.msgUserName`

In addition to the `v3User` knob in the `snmp_trap_serialize` function configuration, the Vendor-documented Serialize function page at `https://docs.cribl.io/stream/snmp-trap-serialize-function/` describes two **event-level fields** that operators can modify via an Eval Function before serialization, to re-shape the outbound v3 trap:

- `msgFlags` — a top-level event string with three documented values: `"00"` (noAuthNoPriv), `"01"` (authNoPriv), `"03"` (authPriv). Setting `msgFlags` before serialization changes the security level of the outbound trap independently of the `v3User` configured on the function. Useful when forwarding traps from a high-security source environment to a lower-security downstream listener (or vice versa) without re-configuring the function.
- `msgSecurityParams.msgUserName` — a nested event field controlling the SNMPv3 username on the outbound trap. Setting `msgSecurityParams.msgUserName = 'newUser'` via Eval before serialization changes the username carried in the outbound trap, useful for multi-tenant trap forwarding where the downstream receiver uses a different user database.

These are not exposed in the OpenAPI schema (which describes the function's static config, not the per-event mutable fields). **Vendor-documented**, not SDK-schema-verified.

### 5.4 v2c dual-location `sysUpTime` / `snmpTrapOID` precedence

For v2c trap events, the Source emits the `sysUpTime` (OID `1.3.6.1.2.1.1.3.0`) and `snmpTrapOID` (OID `1.3.6.1.6.3.1.1.4.1.0`) values **both at the top level of the event and inside `varbinds`**. This dual location is operationally significant because operators can mistakenly assume that modifying the top-level value will affect the serialised PDU. Per the Vendor-documented Serialize function page: **the values inside `varbinds` take precedence** during serialization (they reflect the event's generation time semantics); modifying the top-level fields does NOT affect what the wire PDU carries. Operators wanting to actually change `sysUpTime` or `snmpTrapOID` on an outbound trap must edit the matching entries inside the `varbinds` array.

### 5.5 What the pipeline never does

- **No alert firing**. Cribl Stream has no internal "if trap OID == X, raise alarm Y" engine.
- **No trap correlation**. There is no "link this trap to a related trap via 5-minute window" engine.
- **No severity inference**. Cribl events have no built-in severity field for traps; if the operator wants severity, they map it from an OID via Eval/Lookup.
- **No ack/clear lifecycle**. Cribl does not track per-trap acknowledgement state.
- **No persistent dedup window across restarts**. The Suppress Function uses in-memory state.

These are all destination-tier responsibilities.

## 6. Data Model & Persistent Storage

Cribl Stream's data model is **the Cribl event** — a JSON object with arbitrary fields, plus reserved `__`-prefixed internal fields and `_time` for the timestamp. The trap-specific event shape is documented in §3.

Persistence: **Cribl Stream does not persistently store events**. The only durable storage in the trap path is:

- **Persistent Queue (PQ)** — disk-backed buffer engaged on backpressure. PQ is not a queryable store; it's a forward-only WAL whose only purpose is to replay events into the downstream destination when the destination recovers. Documented at `https://docs.cribl.io/stream/persistent-queues`.
  - **Source-side PQ** on the SNMP Trap Source (engaged when the Routes table or Functions back up). The `InputSnmp` schema exposes `pqEnabled` and `pq` (per `inputsnmpinput.md:14, :17`); operators are explicitly encouraged to enable this for UDP sources (`https://docs.cribl.io/stream/persistent-queues-sources`).
  - **Destination-side PQ is NOT supported** for the SNMP Trap Destination. Per `https://docs.cribl.io/stream/destinations-snmp-traps`, the SNMP Trap Destination's reference table lists `PQ Support: No`; the `OutputSnmp` schema (`.speakeasy/out.openapi.yaml:34064-34135`) contains no `pqEnabled`, `pq`, or equivalent fields. This is a real limitation: backpressure relief on outbound traps exists only at the Source PQ; if the downstream listener is unreachable, in-flight events cannot be queued at the Destination boundary, and the operator must either accept loss or insert a non-SNMP intermediate destination (Kafka, S3) with its own PQ.
  - In Cribl.Cloud: Source PQ is fixed 1 GB per Source per Worker Process, no compression, mode forced to Always On (`https://docs.cribl.io/stream/persistent-queues-sources`).
  - On-prem: Source PQ up to 1 TB, configurable compression (None or Gzip), configurable mode (Always On / Smart), configurable buffer/file/queue size limits.
- **Local Worker disk** — the Source PQ file path is `$CRIBL_HOME/state/queues` by default.

Notably absent:

- **No trap index**. There is no searchable archive of past traps maintained by Cribl Stream itself.
- **No `events` table, no Postgres, no SQLite, no Elasticsearch internal index** — the only databases Cribl runs internally are the embedded ones for its own configuration state and operational metrics (not exposed for trap querying).
- **No retention policy in the trap sense**. Retention is the downstream destination's responsibility.

Cribl's adjacent product, **Cribl Lake**, is a separate optional storage product (S3-backed lake with Cribl Search overlaid). If an operator wants Cribl to durably store traps, they configure a Destination pointing at Cribl Lake or at their own object storage. Cribl Lake is **out of scope** of Cribl Stream proper for this analysis — Stream does not store; Lake stores.

What this means operationally:

- The "queryable history of all traps" capability that LibreNMS, Zenoss, OpenNMS, and Centreon ship in their core product **does not exist** in Cribl Stream. The closest substitute is "forward to Splunk/Elastic/Loki/Lake and query there."
- A pipeline restart (config push, Worker crash, Cribl version upgrade) flushes the PQ but does not flush durable storage (because there isn't any). Events caught in PQ replay on resume.
- This is by deliberate design: Cribl is the pipeline, not the database.

## 7. Configuration UX

Cribl Stream's configuration model is **API-first, UI-second, file-as-implementation-detail**. The OpenAPI spec at `.speakeasy/out.openapi.yaml` is the canonical configuration contract; the operator UI, the REST API, the SDKs (Go, Python, TypeScript), and the Terraform provider all build on this spec.

Configuration surfaces:

- **Web UI**: visual configuration of Sources, Pipelines (with drag-and-drop Function chain editing), Routes (filter editor with autocomplete), Destinations. The UI is the primary supported surface for human operators.
- **REST API**: the OpenAPI spec at `.speakeasy/out.openapi.yaml` documents endpoints to create, list, update, and delete every config object. SNMP-specific endpoints (per the SDK doc tree) include the create-input, create-output, and update-input/update-output flows for type `snmp`.
- **SDKs**: auto-generated by Speakeasy from the OpenAPI spec, for Go, Python, and TypeScript. The SDK doc files under `docs/models/components/` and `docs/models/operations/` are the per-language operator manual.
- **Terraform provider** (`terraform-provider-criblio`, in the mirror): declarative GitOps for Cribl config.
- **Git mode**: Cribl Stream supports versioning the live configuration in Git (the `environment` field on every config object — confirmed on `InputSnmp` at `.speakeasy/out.openapi.yaml:20106` — scopes that object to a specific Git branch). The broader Git-integration workflow is Vendor-documented at `https://docs.cribl.io/stream/version-control`.

SNMP Source UI knobs (per `InputSnmpInput` and `inputsnmpinput.md`):

```

- Type (= "snmp", fixed)
- Disabled (bool)
- Pipeline (id of an attached pre-processing pipeline)
- Send to Routes (bool — if false, the Source goes through QuickConnect direct to one or more Destinations)
- Environment (Git branch scope)
- PQ Enabled (bool)

- Connections (QuickConnect direct-to-Destination shortcuts)
```

Note the `Template*` fields (`__template_environment`, `__template_streamtags`, `__template_host`, `__template_port` in OpenAPI). These bind a variable ID into a config field, so the resolved value comes from a Cribl variable system at runtime rather than from the literal config. This enables, for example, "this Source binds to the variable `cribl.snmp_port`" — useful when the same config is reused across Worker Groups that listen on different ports.

SNMP Destination UI knobs (per `OutputSnmp` and `outputsnmp.md`):

Note the **list-of-hosts model** for the Destination — Cribl can fan out one outbound trap to N destinations atomically. This is uncommon in the cohort; most peers require one Destination per receiver. The Destination supports `dnsResolvePeriodSec` (range 0-86400 seconds; `.speakeasy/out.openapi.yaml:34124-34129`), controlling how often hostnames in `hosts[]` are re-resolved — `0` means resolve on every outbound trap, which adds DNS-latency per trap at high volume and is unsuitable for storms unless paired with a fast local resolver.

```
- Disabled
- Final (stop the pipeline here on match)
- Conf:
    - Strict (bool)
    - DropFailedEvents (bool)
```

Configuration declarative-ness: every config object is declarative JSON/YAML. The UI generates the JSON; the API consumes it directly; the Terraform provider produces it; the Git mode versions it. No imperative DSL, no procedural setup.

Operator-friendliness:

- Cribl ships a first-party visual pipeline builder (drag-and-drop Function chain editor) and JSON-schema-validated configuration. The UI's market reception is positive (frequently cited in industry analyst reports and community discussions) but this is anecdotal — not SDK-verifiable. The structural facts are: the UI is auto-generated from the same OpenAPI that drives the SDKs, every field has a description string in the schema, and the UI shows the same field constraints (`maximum`, `minimum`, `pattern`) as the API.
- The SDK docs are auto-generated and machine-readable, which makes integration in IaC pipelines straightforward.
- The downside: every change is a config push from the Leader to all Workers; there is no per-Worker hot-swap of just one Source.

## 8. Integration with Other Signals

Cribl Stream's integration model is **first-class for all signals**. SNMP traps are one of ~90 inputs and ~70 outputs; everything else (logs, metrics, traces, events, NetFlow, etc.) flows through the same Routes/Pipelines/Destinations machinery. The subsections below follow the SOW's §8.1-§8.5 partition.

### 8.1 Metrics

- Traps are NOT auto-converted to metrics. Cribl Stream emits operational metrics about its own pipelines (events/sec, bytes/sec, queue depth — visible in the Monitoring UI) but does **not** synthesise a "trap count by OID" metric stream from incoming traps.
- Traps can be **manually converted** to metrics by an operator-built Pipeline using a Function chain: e.g. `Aggregate` rolling traps by `(host, varbinds[0].oid)` into a counter, then routing to a Prometheus/StatsD/InfluxDB destination. There is no built-in template for this.
- Traps are NOT used as annotations on metric dashboards within Cribl — Cribl has no metric dashboards for trap content. Operators wanting annotations must forward traps to a destination that has its own annotation system (Grafana, Datadog).

### 8.2 Alerting / Notifications

- **Cribl Stream does not generate alerts from trap content**. Cribl's built-in Notifications system (`https://docs.cribl.io/stream/monitoring`, Vendor-documented) covers operational concerns — data-volume thresholds, node-offline, license usage — not trap-payload semantics.
- Operators wanting "alert when trap OID X arrives" must forward the trap to a destination with its own alerting (Splunk, Datadog, PagerDuty webhook, ServiceNow, etc.) and define the alert there.
- There is no ack/clear lifecycle in Cribl for individual traps. Acknowledgement is purely a destination-side concern.
- Alert routing, escalation, deduplication policies: NONE in Cribl beyond the generic pipeline dedup primitive (Suppress function, see §10).

### 8.3 Topology

- **No native topology model**. Cribl Stream has no concept of "node", "interface", "link", or "vendor model" — a trap is an event with a source IP. There is no L2/L3/L7 graph maintained by Cribl.
- **No topology-aware suppression**. Suppress dedups on an arbitrary JS expression, not on a topology relation. An operator could approximate topology suppression by building a Lookup table that resolves `source IP -> rack/site` and keying Suppress on that resolved field, but the topology itself must be supplied externally.

### 8.4 Logs / Events

- Traps become events in the Cribl event stream alongside Syslog, JSON logs, OTel events, NetFlow, etc. All event types flow through the same Routes/Pipelines/Destinations machinery — a **strong distinguishing capability** versus systems where trap pipelines are siloed (Nagios SNMPTT → Nagios, OpenNMS trap engine → OpenNMS alarms).
- **Searchability and retention live in the destination**. Cribl Stream itself has no event index; queryable trap history is the responsibility of whichever destination the operator picks (Splunk index, Elastic ILM-managed index, Cribl Lake, S3+Athena, etc.).
- **Schema**: Cribl events are JSON objects with arbitrary fields plus `__`-prefixed internal fields and `_time`. Trap-specific shape is enumerated in §3.
- **Cross-signal enrichment via Lookup**: a trap can be enriched with the CMDB owner of the source IP by joining against a CSV lookup loaded by the Lookup function; the same lookup is reusable across syslog, log, and metric pipelines — a real cross-signal benefit.
- **Cross-signal correlation is NOT done in Cribl Stream**. Cribl can route and enrich the trap, but the actual "this trap belongs to the same incident as that log line" logic happens in the destination (Splunk Enterprise Security, Elastic SIEM, Datadog, ServiceNow ITOM, etc.). Cribl's adjacent products **Cribl Search** (S3-query overlay) and **Cribl Lake** (S3-backed storage) can query trap data once stored, but those are separate products.

### 8.5 Northbound Forwarding

- **Verbatim SNMP forwarding**: the SNMP Trap Destination (`OutputSnmp`) replays the original `__snmpRaw` PDU to one or more downstream SNMP listeners (`hosts[]` list). This is the canonical "forward to my NMS" path.
- **Re-serialised SNMP forwarding**: the `snmp_trap_serialize` pipeline function rebuilds a wire-format PDU from the in-event fields (after pipeline modification). Required when varbinds have been changed; mandates `varbindsWithTypes: true` on the Source. **Important constraint** (Vendor-documented): only events originating from an SNMP Trap Source can be sent to the SNMP Trap Destination — non-SNMP input cannot be synthesised into a trap.
- **Non-SNMP northbound**: traps can be forwarded to ~70 other destinations (Splunk, Elastic, Kafka, S3, Datadog, Syslog, HTTP/REST, OTLP via the OpenTelemetry destination). These receive the event as JSON-like records, not as wire-format traps.
- **Atomic fan-out**: Routes with `Final=off` clone the event; one trap can simultaneously go to an SNMP NMS (verbatim) and to Splunk (JSON) and to S3 (cold storage).

This is structurally similar to Logstash and Telegraf: be the pipeline, let the destination do the rest. The difference: Logstash's UI/UX is configuration files only, Telegraf has no UI at all, Cribl ships a UI-driven visual pipeline builder plus a first-party trap-serialiser function.

## 9. Severity Model

**Cribl Stream has no built-in severity model for traps.** Traps emerge from the Source with their raw varbinds; there is no automatic mapping from generic-trap-type (cold-start, warm-start, link-down, link-up, authentication-failure, egp-neighbor-loss, enterprise-specific) to a severity ladder.

If the operator wants severity, they build it themselves in the Pipeline:

- A typical Eval function pattern:
  ```javascript
  severity = (snmpTrapOID == '1.3.6.1.6.3.1.1.5.3') ? 'critical'   // linkDown
           : (snmpTrapOID == '1.3.6.1.6.3.1.1.5.4') ? 'info'       // linkUp
           : (snmpTrapOID == '1.3.6.1.6.3.1.1.5.1') ? 'warning'    // coldStart
           : 'unknown';
  ```
- A Lookup function pattern: a CSV with `oid, severity` columns, matched on the trap OID.
- A community-pack pattern: the `cribl-snmp-trap-varbind-translation` pack does not assign severity directly (its job is OID translation), but the resulting `trap_*` fields make a downstream Eval trivial.

What this means operationally:

- An out-of-the-box Cribl Stream installation **does not classify traps by severity at all**. Traps flow as events; the destination is expected to do severity classification (or none at all, in which case all traps look the same to the human operator).
- This is structurally different from OpenNMS/Zenoss/Centreon, where the severity model is part of the trap definition (every event has a severity by config); and from Splunk SC4SNMP, where the YAML map specifies severity per OID.
- vs **Logstash**, **Telegraf**: same — no severity model; same workaround.

The implication for Netdata's design: shipping a default severity model (with curated OID → severity mappings for a handful of standard traps) is something Cribl explicitly **does not do**. This is a possible differentiation axis: Netdata can ship out-of-the-box severity mappings for at minimum the IETF generic traps (RFC 1215) and a curated subset of common vendor MIBs (Cisco, Juniper, Arista, HP, Dell), without forcing operators to build severity tables.

## 10. Storm / Volume Handling

Trap storms (one device emitting hundreds of traps per second, or a fleet emitting ten thousand traps per second across a backbone outage) are the perennial NMS killer. Cribl Stream's posture on storms:

**Reception-side**:

- The Source has `udpSocketRxBufSize` to tune `SO_RCVBUF`; sizing this generously is the first line of defence against burst loss.
- `maxBufferSize` caps in-process event accumulation; once exceeded, the Source either drops or engages PQ.
- IP allowlist regex can hard-reject specific noisy senders.
- **Cribl.Cloud explicitly cannot handle high-volume UDP** — the Known Issues page (`https://docs.cribl.io/stream/4.1/known-issues`, `https://docs.cribl.io/stream/4.3/known-issues`) calls out UDP drop under load and the recommended workaround is hybrid Worker Groups on customer infrastructure close to the senders.

**Pipeline-side dedup**:

- The **Suppress** function (per `PipelineExamplesSuppress` at `.speakeasy/out.openapi.yaml:50870-50890`) drops duplicate events based on a JS-expression key over a configurable time window. Sample shape:
  ```yaml
  - id: suppress
    filter: "true"
    conf:
      keyExpr: "`${ip}:${port}`"
      allow: 1
      suppressPeriodSec: 30
      dropEventsMode: true
      maxCacheSize: 50000
      cacheIdleTimeoutPeriods: 2
      numEventsIdleTimeoutTrigger: 10000
  ```
  Adapted for traps: `keyExpr: `${host}:${varbinds[0].oid}`` with `suppressPeriodSec: 60` would dedupe identical source+OID combinations within 60s. This is the primary storm-rate-limit primitive.
- The **Drop** function with a JS filter drops matching events outright.
- **Sample** function reduces volume to a fraction (1-in-N, or rate-limited).
- **Aggregate** function can roll up bursts into summarised events.

**Backpressure**:

- The SNMP Trap Destination has **no Destination-side PQ** (per `https://docs.cribl.io/stream/destinations-snmp-traps`). Any backpressure relief for outbound SNMP traps must come from the Source PQ or from inserting a non-SNMP intermediate destination (Kafka, S3, Splunk) that has its own PQ.
- If the downstream SNMP receiver is unreachable or slow, in-flight events propagate backpressure upstream into the Pipeline and ultimately to the Source.
- If the Source PQ fills (or is not enabled), the Source drops on the wire (UDP). Non-UDP sources can apply TCP-like flow control; UDP cannot.

What Cribl does NOT do for storms:

- **No deduplication keyed on trap semantics** out of the box. Suppress is a generic primitive; the operator wires it up.
- **No "storm-of-similar-traps becomes one summarised trap" engine**. Aggregate exists, but it has no trap-semantics awareness.
- **No per-source rate limiting at the Source level**. There is no `maxTrapsPerSourceIpPerSecond`; rate limiting is a pipeline-side decision via Throttle (a generic Function).
- **No automatic trap-rate alarms on the Source itself**. Volume alerts exist on the Source overall, not "this IP just exceeded N traps/sec".

Operator burden:

- Out-of-the-box, a trap storm will fill PQ and either backpressure the Source (causing UDP drop) or drain to the destination (loading the destination). The operator MUST build storm handling explicitly in the Pipeline.
- vs **OpenNMS**: built-in `eventd` dedup with per-OID configurable thresholds. Cribl has nothing similar by default.
- vs **Zenoss**: built-in event deduplication and de-escalation per event-class. Cribl has nothing similar by default.
- vs **Centreon**: similar story to OpenNMS — built-in storm handling. Cribl has nothing similar by default.
- vs **Telegraf**, **Logstash**: same as Cribl — generic primitives, no trap-aware semantics.

The implication for Netdata's design: Cribl's "ship the primitives, operator builds the storm handler" is the Logstash/Telegraf pattern. The NMS-class systems ship storm semantics. Either is defensible; Netdata's choice should be explicit and documented.

## 11. Security

**Inbound trap path** (Source-side):

- **No TLS** on the SNMP Trap Source (UDP/162 is plaintext; DTLS RFC 5953 is not supported, per `https://docs.cribl.io/stream/sources-snmp-traps`). For v1/v2c, the only "auth" is the community string in the PDU — universally known to be inadequate. For v3, USM provides per-user keyed auth (HMAC-MD5/SHA family) and priv (DES/AES variants).
- **IP allowlist regex** is the primary access control for v1/v2c (`ipWhitelistRegex` field). The Source docs page documents the default as `.*` (Vendor-documented; not in OpenAPI schema) — i.e., no filter; operators are expected to lock down.
- **SNMPv3 USM**:
  - Auth protocols (per `AuthenticationProtocolOptionsV3User` enum at `.speakeasy/out.openapi.yaml:45658-45676`): `none`, `md5`, `sha` (= SHA-1), `sha224`, `sha256`, `sha384`, `sha512`. The SHA-2 family (224-512) is significant because RFC 7860 (HMAC-SHA-2) is from 2016 — products that pre-date that won't have it.
  - Priv protocols (per `PrivacyProtocolOptionsSnmpTrapSerializeV3UserAuthProtocolNotNone` enum at `.speakeasy/out.openapi.yaml:45583-45598`): `none`, `des`, `aes` (= AES-128, RFC 3826), `aes256b` (Blumenthal — RFC draft, not standardised), `aes256r` (Reeder — RFC draft, not standardised). The presence of both Blumenthal and Reeder AES-256 is unusual and a positive — most products ship only one variant or none.
  - **Discrepancy alert**: the SDK schema lists `des` as a valid `privProtocol`, but the public docs page at `https://docs.cribl.io/stream/sources-snmp-traps` (and the Serialize function docs page, both verified during this analysis) **list only AES variants — DES is absent from the UI selection**. The SDK is generated from the OpenAPI; if DES is in OpenAPI, the **control-plane schema models DES** as a valid configuration value (API caller can submit it). Whether the closed-source daemon actually negotiates a DES session at runtime is a separate question that cannot be verified from public materials — schema modelling and runtime behaviour are not the same thing. The docs page may be (a) hiding deprecated DES from the UI dropdown while the API still accepts it (graceful deprecation), (b) genuinely up-to-date showing DES has been removed from the UI only, or (c) DES is in the schema for legacy compatibility but no longer functional at runtime. Either way, **DES support is at best deprecated and at worst schema-accepted but non-functional** — operators should not rely on it. Note also a separate docs typo: the privacy protocol list on the Cribl docs page renders one entry as `AES2556r (Reeder)` (extra `5`); the correct token is `AES256r`. **Confidence: moderate.**
  - `allowUnmatchedTrap`: when on, v3 traps not matching any configured user pass through (undecoded). When off (default), they are dropped. The "off" default is the safer choice; "on" exists for diagnostic and noisy-environment cases.
- **No DTLS**, no SSH-based transport, no SSL VPN integration on the trap path. Operators wanting wire encryption beyond v3-USM-priv must terminate elsewhere (IPSec, WireGuard, etc.).

**Outbound trap path** (Destination-side):

- The Destination is a verbatim UDP forwarder — it replays `__snmpRaw` to `(host, port)`. No TLS, no DTLS, no authentication beyond what was in the original PDU.
- The `snmp_trap_serialize` function (when re-emitting after pipeline modification) supports the **same v3 USM matrix** for the outbound side — operator provides `v3User` with `name`, `authProtocol`, `authKey`, optional `privProtocol`/`privKey`.

**Control plane security**:

- Leader-Worker comms over HTTPS REST (per `https://docs.cribl.io/stream/deploy-distributed`).
- RBAC is **Enterprise/Cribl.Cloud only** (per `https://docs.cribl.io/stream/roles`); the Free tier has only the `admin` role. Built-in roles: `admin`, `stream_admin`, `stream_editor`, `stream_reader`, `stream_user`. Worker-group-scoped policies: `GroupRead`, `GroupEdit`, `GroupFull`, `*`. SNMP Source/Destination/Pipeline edits sit under `GroupEdit`.
- **No mTLS Cribl-to-Cribl** for the trap data path (UDP); Worker-Worker traffic uses HTTPS for config.

**Secret handling**:

- SNMPv3 `authKey` and `privKey` are modelled as plain `type: string` fields in the OpenAPI (`.speakeasy/out.openapi.yaml:20168-20175`, `inputsnmpv3user.md:10`, `:12`) — there is no `$ref` to a Secret-type wrapper. This is one of the rougher edges: per-user pre-shared keys are critical secrets and storing them as plaintext in the YAML/JSON config (and the Git-versioned config tree, if Git mode is on) is a significant exposure surface.
- Cribl Stream does ship a **Secrets** subsystem for credentials to other destinations (AWS, Splunk HEC, Kafka SASL, etc.), Vendor-documented at `https://docs.cribl.io/stream/secrets/`. The SNMP-USM keys do not appear to use the Secrets subsystem in the OpenAPI — they are inline string fields. **Confidence: moderate**; cannot verify whether the operator UI offers a Secrets-backed entry path that the OpenAPI surfaces as a plain string. Operators should treat this as a real risk and consider storing Cribl configs outside Git, or using sealed-secret / git-crypt overlays.

**Audit**:

- The Leader's monitoring console captures operational events but does not, per the public docs, log per-trap reception details. There is no built-in trap audit log; that responsibility is pushed to the destination.

## 12. Trap Simulation & Testing (in-source evidence)

Because the daemon is closed-source, there is **no in-source test suite available for inspection**. What is publicly available:

- The OpenAPI **examples** at `.speakeasy/out.openapi.yaml`:
  - `InputCreateExamplesSnmp` (line 49249-49257) — minimal Source create payload.
  - `OutputCreateExamplesSnmp` (line 49656-49663) — minimal Destination create payload.
  - `PipelineExamplesSNMPTrapSerialize` (line 50854-50869) — minimal Pipeline with the serialize function.

- Public-blog example pipelines (per `https://cribl.io/blog/unravelling-snmp-trap-varbind-translation-with-cribl/`) — narrative walk-throughs, not test artefacts.

What is **not** publicly available:

- The unit and integration tests for the SNMP Trap Source decoder.
- The conformance tests against malformed PDUs, fragmented UDP, oversized packets, replay attacks, USM time-window violations, unknown engineID handling.
- Performance benchmarks of trap-decode-per-second per Worker Process.
- Specific behaviour under UDP receive-buffer pressure (the docs say drops happen; no specific rate numbers are published).

What can be inferred from the docs:

- The Source documents `__didDecrypt` for v3 (suggesting test coverage of decryption-failure paths).
- The Serialize function documents specific error fields (`snmpSerializeErrors`) and required varbinds (`sysUpTime` OID `1.3.6.1.2.1.1.3.0` and `snmpTrapOid` OID `1.3.6.1.6.3.1.1.4.1.0`) — these correspond to RFC 3416 §4.2.6 and RFC 1905 §3.1.2 requirements, suggesting standards conformance was a design goal.
- Cribl.Cloud's documented UDP-drop limitation (per the Known Issues page) is itself test evidence: this is a behaviour Cribl has tested under load and documents as an operational constraint.

**Confidence in test-coverage assessment: low** — "low" here means the assessment cannot be made, not that coverage is known to be low. This is the central honesty point of this entire spec: Cribl Stream's SNMP trap support is a black box. The shape of the API is public; the behaviour under stress, edge cases, and malicious input is not auditable.

**Operator-facing validation surfaces are nonetheless documented**, separate from internal tests:

- **Live Data** (per `https://docs.cribl.io/stream/sources-snmp-traps` and the Sources reference at `https://docs.cribl.io/stream/sources`): per-Source live tail of incoming events in the UI; immediate sanity check that traps are being decoded.
- **Logs** (per `https://docs.cribl.io/stream/monitoring`): Worker debug logs surface SNMPv3 auth/decrypt failures (the `snmpV3Auth` field description at `.speakeasy/out.openapi.yaml:20141` explicitly directs operators to enable debug logging on auth issues).
- **Monitoring dashboards**: Source-level events/sec, bytes/sec, errors per the Monitoring page.
- **Data Preview**: pipeline-level "what does the Function chain output for this input?" panel; documented for the `snmp_trap_serialize` function at `https://docs.cribl.io/stream/snmp-trap-serialize-function/`. Lets the operator dry-run serialization against a captured event.
- **External fixture generation**: `snmptrap` (Net-SNMP CLI), `snmpsend` (Wireshark-friendly), and crafted-PDU Python scripts using `pysnmp` are the standard practitioner tools for trap simulation against a live Cribl Source. Cribl does not ship its own simulator.

The distinction "no source tests available" vs "operator validation surfaces available" matters for risk assessment: an evaluator can confirm Cribl decodes a given trap correctly, but cannot confirm Cribl decodes every malformed trap safely.

- **Telegraf**, **Logstash**, **Sensu**, **Centreon**, **Zabbix**, **OpenNMS**, **Zenoss**, **LibreNMS**, **Nagios/SNMPTT** — all open-source; their tests are file-system-grep-able and citable line-by-line.

## 14. User Customization Surface

What an operator can customise in Cribl Stream's trap path:

- **Decode**: bind addr, UDP port, `SO_RCVBUF`, max buffer, IP allowlist, varbind parsing mode, best-effort UTF-8, v3 USM user list, allowUnmatchedTrap.
- **Identity**: per-Source metadata fields (`metadata` list, injecting fixed fields like `env=prod`), streamtags, description, Git-branch scoping (`environment`).
- **Pipeline**: any sequence of Functions, with full JS-expression filtering at each step. Practical effect: arbitrary trap transformation in a sandboxed JS runtime.
- **Routes**: arbitrary JS filter expressions, fan-out via Final=off cloning, default catch-all route.
- **Destinations**: zero or more SNMP destinations (forwarding verbatim or re-serialised); zero or more non-SNMP destinations (Splunk, Elastic, Kafka, S3, …); per-Destination post-processing pipeline.
- **OID lookups**: arbitrary CSVs uploaded via the Cribl Lookup file infrastructure; consumed by Lookup function.
- **Packs**: install the community `cribl-snmp-trap-varbind-translation` pack or build your own.

What an operator cannot customise (or cannot easily customise):

- **The trap decoder itself**. Cribl Stream's PDU decoder is internal; there is no plugin point to insert a different decoder.
- **The wire emitter for `snmp_trap_serialize`**. The serialiser is internal; the operator can affect *what* event fields go in, but not *how* the binary PDU is constructed.
- **The `__snmp*` reserved field set**. These are fixed by the Source code (closed); the operator cannot add new internal fields beyond what Cribl ships.
- **Bind-level UDP handling**. The kernel `SO_REUSEPORT` semantics are fixed; the operator cannot configure (for example) a Cribl-internal connection tracker that ties one Worker Process to one device IP.

Extensibility model:

- Cribl Stream has no native "write a new Source plugin in language X" mechanism for end-users. Pipeline Functions can use JavaScript and call out to external services (HTTP, Redis, DBs) but the Source/Destination types are fixed to the catalogue Cribl ships.
- This is materially different from Logstash (plugin gem ecosystem), Telegraf (Go-built plugins), and Sensu (script handlers). Cribl is closer to a SaaS-style closed plugin catalogue.

Customisation budget summary:

- **High** for transformation (pipelines, lookups, packs).
- **Medium** for ingress shape (binding, ACL, multi-user v3).
- **Zero** for decoder internals.
- **Zero** for emitter PDU internals.
- **Zero** for adding new Source types.

## 15. End-User Value Analysis

Through the lens of the SOW's foundational spec at `.agents/sow/specs/snmp-traps/research/domain/snmp-traps-in-observability.md`:

**Strengths from the operator perspective**:

1. **Bidirectional first-party trap support**. Cribl Stream is uncommon in the cohort in shipping both inbound trap reception AND outbound trap forwarding as first-party features. Logstash also has both, but its outbound emitter is a community plugin (`logstash-output-snmptrap`); Telegraf has reception but no emission at all; OpenNMS/Zenoss/Centreon focus on reception with limited northbound. Cribl's outbound path has a documented constraint — the `snmp_trap_serialize` function only operates on events originating from an SNMP Trap Source — so it is best described as a first-party **re-serializer** for SNMP-source-derived compliant events, not a general event-to-trap emitter.
2. **Pipeline composition over hard-coded behaviour**. An operator who wants "drop all link-up traps, rate-limit link-down to 1/min per source, enrich with CMDB data, forward unchanged to OpenNMS and JSON-form to Splunk" can express that in a Cribl pipeline cleanly. Doing the same in OpenNMS requires editing eventd configs; doing it in Logstash requires writing a Ruby filter; doing it in Telegraf is not possible at all without an external co-process.
3. **Broad v3 USM support**. Auth protocols cover the full SHA-2 ladder per RFC 7860 (`md5`, `sha`=SHA-1, `sha224`, `sha256`, `sha384`, `sha512`). Priv protocols include DES (in API; not in UI dropdown), AES-128 (RFC 3826), and both AES-256 Blumenthal-draft and Reeder-draft variants. Among pipeline-tier and NMS-tier peers examined to date, Cribl's v3 priv surface is the widest (most peers ship AES-128 only, or AES-128 + one AES-256 draft).
4. **Multi-user v3**. The Source's USM user list is `minItems: 1` with no documented upper bound; multiple device-classes can authenticate with distinct credentials against the same Source.
5. **Fan-out from one trap to many destinations**. A trap can be routed simultaneously to its OpenNMS NMS sink (verbatim), to Splunk (JSON), and to S3 (cold storage) by enabling all three with Routes Final=off cloning. This atomic fan-out is core to Cribl's value prop.
6. **GitOps-friendly config**. The OpenAPI-defined config tree, Terraform provider, and Git branch scoping support modern declarative ops.

**Weaknesses from the operator perspective**:

1. **No batteries included**. Out of the box, traps are decoded but unenrichable, unprioritised, undashboarded, unalerted. Operators bootstrapping from zero face a non-trivial pipeline-build phase. NMS-class systems give immediate value; Cribl gives infrastructure.
2. **MIB management is external**. Cribl ships no MIB compiler; the recommended workflow involves `snmptranslate` or `pysnmp` runs by the operator, generating CSV lookups. Compared to Telegraf (`gosmi` in-process), this is a regression in operational simplicity.
3. **No content-level alerting**. Cribl Notifications are operational only. "Alert me when this trap arrives" requires forwarding to a destination that has alerting.
4. **No persistent trap history in Cribl**. PQ is forward-only. "Show me all traps from device X yesterday" requires the destination.
5. **DES privacy is in the OpenAPI but absent from the UI page**. Documentation discrepancy; operators trusting either side may misconfigure.
6. **Plaintext USM keys in config**. SNMPv3 `authKey`/`privKey` appear as inline strings in the config JSON; if the config is Git-versioned, those keys are in Git. No documented Secrets-backed alternative for these fields in the OpenAPI.
7. **No native UDP/162 HA**. Operators must bring their own load balancer or VIP solution. Other UDP-listener products (Splunk Universal Forwarder, Net-SNMP `snmptrapd`) have similar constraints, but several NMS-class systems (Zenoss, OpenNMS) ship active/passive guidance.
8. **Cribl.Cloud documented UDP drop**. The Known Issues page explicitly states that Cribl.Cloud may drop UDP under load. For UDP-heavy SNMP-trap workloads, the recommended deployment is hybrid (customer-managed Workers near the senders) — meaning the SaaS-tier convenience is not actually available for this workload.
9. **Closed-source**. Operators cannot patch bugs, audit security behaviour, build new Source plugins, or fork.

**Net value posture**:

Cribl Stream's value is highest when the operator already has a powerful destination (Splunk, Elastic SIEM, ServiceNow, Datadog, etc.) and wants to feed it cleanly. Cribl is the vendor-supported, UI-driven, scriptable pipeline that operators sometimes describe as "what they wished Logstash had been". Cribl Stream's value is lowest when the operator needs an out-of-box NMS — that is a category mistake, and Cribl is not the product for that need.

## 16. Strengths

In one place, the standout strengths (evidence-grounded; comparative claims are bounded by the cohort actually examined to date, not absolute):

1. **Bidirectional first-party trap support** — inbound Source + outbound Destination + in-pipeline re-serializer function (`snmp_trap_serialize` is scoped to SNMP-source-derived events). Uncommon in the cohort; only Cribl and Logstash (via community plugin) ship both inbound and outbound trap support.
2. **Pipeline-function composition** — broad generic transformation surface; any sequence of Functions (Eval, Lookup, Drop, Suppress, Regex Extract, Code, snmp_trap_serialize, ...) can be applied to a trap event. UI-driven visual pipeline builder; field validity and constraints are enforced from the same OpenAPI schema that drives the SDKs.
3. **Broad SNMPv3 USM matrix** — auth: `md5`, `sha` (SHA-1), `sha224`, `sha256`, `sha384`, `sha512`; priv: `des`, `aes` (AES-128), `aes256b` (Blumenthal), `aes256r` (Reeder). Among pipeline-tier and NMS-tier peers examined to date, the priv ladder is the widest.
4. **Multi-user v3 on one Source** — `v3Users` is a list; multiple per-device-class credentials can coexist.
5. **Bound `__snmpRaw` for verbatim re-emission** — operators can route untouched traps to legacy receivers (OpenNMS, vendor NMS) while also routing a transformed JSON copy to modern systems (Splunk, Elastic) atomically.
6. **Atomic fan-out to many destinations** — Routes with `Final=off` cloning is a clean primitive.
7. **OpenAPI-first config** with auto-generated SDKs and a Terraform provider — modern declarative-ops story.
8. **First-party non-SNMP destinations** — traps can go to S3, Kafka, Snowflake, Splunk, Elastic, OpenTelemetry, etc., not just SNMP listeners.
9. **Persistent Queue with explicit recommendation for UDP sources** — Cribl actively documents that UDP needs Source PQ.
10. **Cribl Edge parity** — the same SNMP Trap Source surface runs on the lightweight Edge agent (per `https://docs.cribl.io/edge/sources-snmp-traps`, Vendor-documented), enabling edge-of-network trap reception without a heavyweight central deployment. (Edge-vs-Stream differences in PQ sizing, fleet management scoping, and Worker Group model are not enumerated in this analysis — deferred as additional surface area; not blocking.)

## 17. Weaknesses / Gaps

The honest list:

1. **Closed-source.** Behaviour under stress, malicious input, edge cases is unauditable. No community can patch bugs.
2. **No embedded MIB compiler.** OID-to-name resolution is operator-supplied CSV. Burden shifted out of the product.
3. **Zero out-of-box vendor MIB coverage.** Compared to NMS-class systems shipping thousands of MIBs preloaded, Cribl ships zero.
4. **No severity model.** Every operator builds their own.
5. **No alerting on trap content.** Notifications are operational, not semantic.
6. **No trap dashboards.** Monitoring dashboards visualise throughput, not trap payloads.
7. **No durable trap history in Cribl.** PQ is forward-only; queryable history is offloaded to the destination.
8. **No native UDP HA.** Operator brings the LB.
9. **Cribl.Cloud UDP drop is a documented limitation** for the SaaS-tier of Cribl Stream itself, which is the vendor's primary commercial offering. The official workaround pushes operators back to hybrid/on-prem — meaning the SaaS convenience is not actually delivered for the SNMP-trap workload.
10. **SNMPv3 keys in plaintext** in config (per the schema modelling these as plain strings). No documented Secrets-backed binding.
11. **DES privacy docs/SDK discrepancy.** API accepts `des`; docs imply it isn't offered. Either deprecated-but-still-API-accepting or under-documented.
12. **No way to author new Source types.** The Source catalogue is fixed; an operator with an unusual trap source (BGMP traps, RMON-2 traps with custom encoding, or any vendor that ships off-spec PDUs) cannot extend the decoder.
13. **Per-trap cost in fan-out**. Routes Final=off clones the event; very large fan-out (10+ destinations) duplicates the event N times in memory and N times in PQ. Not unique to Cribl, but worth noting.
14. **No first-party InformRequest acknowledgement documentation.** The schema does not expose an inform-ack tuning knob; whether v2c InformRequest is acknowledged at all is undocumented in the public material.
15. **Documentation example sets UDP port to 161 for the SNMP Source.** This is a real bug in the OpenAPI examples block (`.speakeasy/out.openapi.yaml:49255` for `InputCreateExamplesSnmp` and `.speakeasy/out.openapi.yaml:49663` for `OutputCreateExamplesSnmp`); a copy-paste error from the polling port. Operators following the example would bind the listener to the polling port, missing all traps.
16. **No Destination-side PQ for the SNMP Trap Destination.** Per `https://docs.cribl.io/stream/destinations-snmp-traps`, the Destination has `PQ Support: No`. Source PQ is the only line of defence; if the downstream SNMP listener is unreachable, in-flight events propagate backpressure straight back to UDP receive — and UDP cannot apply flow control.
17. **No SNMPv3 context engine ID / context name / local engine ID.** Schema-verified absence in `InputSnmp` and `InputSnmpV3User`. Peers exposing these knobs (Logstash v4.1.0+, Datadog Agent) can integrate with fixed-engine-ID device configurations and per-context scoping; Cribl cannot.
18. **`x-speakeasy-unknown-values: allow` on auth/priv protocol enums.** Both `AuthenticationProtocolOptionsV3User` (`.speakeasy/out.openapi.yaml:45677`) and `PrivacyProtocolOptionsSnmpTrapSerializeV3UserAuthProtocolNotNone` (`.speakeasy/out.openapi.yaml:45598`) declare this Speakeasy SDK-generation directive. The flag instructs **generated SDK clients** (Go, Python, TypeScript) to deserialise responses containing unknown enum values rather than crash — i.e., forward-compatibility for new server-side enum additions. It does NOT directly prove whether the Cribl server validates submitted enum strings strictly. Whether the daemon rejects an unknown auth/priv protocol is not source-verifiable. Operators using the OpenAPI schema for client-side config validation should be aware the schema alone does not enforce strict enum acceptance on the client and the daemon's strictness is unknown.
19. **No native severity-aware storm dampening.** Suppress is generic; "suppress only repeated link-flap traps from the same interface OID within 30 s" requires an operator-built `keyExpr` that knows the OID semantics — and an operator who has already loaded the right Lookup tables. This is the same gap as Telegraf and Logstash, but worth noting alongside the storm-handling discussion in §10 because operators coming from OpenNMS/Zenoss/Centreon expect this to "just work".
20. **Docs typo `AES2556r`.** The Cribl Source docs page renders the AES-256r Reeder option as `AES2556r (Reeder)` (extra digit) at `https://docs.cribl.io/stream/sources-snmp-traps`. Cosmetic; the OpenAPI and SDK markdown use the correct `aes256r` token.
21. **No documented retry / timeout / backoff knobs on the SNMP Trap Destination.** The `OutputSnmp` schema (`.speakeasy/out.openapi.yaml:34064-34135`) exposes `hosts[]`, `dnsResolvePeriodSec`, `pipeline`, `systemFields`, `environment`, `streamtags`, `description` — and no `retries`, `timeoutMs`, `backoffMs`, `failureMode`, or equivalent. Combined with the absence of Destination-side PQ (item 16), outbound trap reliability is single-shot UDP with no documented retransmission. **Confidence: high** for the schema-verified absence; the closed-source daemon may have hardcoded retry behaviour, but it is not exposed.

### 18.9 Suppress function for storm handling (adapted from OpenAPI pipeline example)


## 20. Evidence Confidence

This is the single most important section to read honestly. Cribl Stream is one of four closed-source / docs-only systems in this comparative pass (alongside SolarWinds, Dynatrace, LogicMonitor per the SOW Locked Scope). Within that subset, Cribl is distinctive because its public OpenAPI + Speakeasy-generated SDKs expose the **control-plane configuration schema** at machine precision — every Source/Destination/Function field, type, requirement, and discriminator. That is more verifiable than what SolarWinds/Dynatrace/LogicMonitor offer publicly. But it is still only the configuration contract, not the daemon's runtime behaviour. The runtime decoder, USM negotiation, PDU emission, and error paths are unauditable.

**High confidence (verifiable from public, auto-generated artefacts)**:

- **API surface (control-plane contract)**: every Source/Destination/Function config field, type, requirement, default-where-documented, and discriminator. The OpenAPI spec is auto-generated from Cribl's production code and is the literal control-plane contract that the Cribl API server validates inbound configuration requests against. Note: "the schema models X" is NOT the same claim as "the daemon, at runtime, successfully negotiates X" — particularly for SNMPv3 priv where DES is in the schema but absent from the docs UI; the runtime acceptance of DES negotiation is unverified.
- **Configuration UX**: the OpenAPI fields map 1:1 onto UI knobs and SDK constructors. The shape of the operator experience is fully determined by the schema.
- **Cribl public docs page content**: high confidence, because the same fields are cross-referenced between the docs page and the OpenAPI (port, descriptions match). Field descriptions are coherent. Note: the docs page documents several defaults the OpenAPI schema does NOT encode (e.g., `maxBufferSize` default 1000, `ipWhitelistRegex` default `.*`, `varbindsWithTypes` default false, `allowUnmatchedTrap` default off, `v3AuthEnabled` default off, `pqEnabled` default off) — these are vendor-documented but not schema-encoded.

**Moderate confidence (inferred from public artefacts but not directly verified)**:

- **Runtime language: likely Node.js**. Inferred from JS-syntax filter expressions, the `__filename` field on Function objects (`.speakeasy/out.openapi.yaml:5564`), and the Speakeasy SDK regen process — supported by Cribl public engineering material and job postings; not literally proven without daemon source.
- **SNMP library**: not publicly identified. Speculation: `net-snmp` Node.js package or a fork, given the Node.js inference. Unverified.
- **Per-Worker-Process UDP fan-out mechanism**. Whether multiple Worker Processes bind the same UDP port with `SO_REUSEPORT` and let the kernel round-robin, or whether one process binds and dispatches internally, is not stated in any public doc.
- **The DES privacy actual support status**. The control-plane schema models `des` at `.speakeasy/out.openapi.yaml:45583-45598`; the public docs UI page lists only AES variants. The most likely explanation is "schema-modelled-but-UI-hidden", but neither documentation path explicitly states whether the runtime daemon actually negotiates DES.
- **Handling of SNMPv2c `InformRequest` acknowledgement**. The schema has no inform-ack knob; the docs do not say explicitly. Best guess: ack is always-on or not-implemented.
(The previous bullet about `maxBufferSize` default has been promoted to high-confidence above — the docs page does document the default as 1000 events.)
- **The SNMPv1-to-v2c bridge (RFC 3584)**. Plausible Cribl performs this for v1 traps so v1 PDU fields (enterprise OID, generic-trap, specific-trap, agent-address) become varbinds in the output event, but not Cribl-documented.

**Low confidence (closed-source means unverifiable)**:

- Behaviour under malformed/malicious PDUs. The system's robustness against crafted SNMP packets is unauditable.
- Test coverage of v3 USM edge cases (engineID discovery, time-window enforcement, replay window, unknown user, decryption failure paths).
- Performance: traps-per-second per Worker Process. Cribl publishes no benchmark for the SNMP Source specifically.
- The internal data structures used for the `varbinds` decode (whether it preserves SEQUENCE/SEQUENCE-OF nesting, how it handles MIB Counter64 wraparound, etc.).
- The actual semantic of `bestEffortParsing` — the docs describe it ("attempt UTF-8 first, fallback to other methods") but the "other methods" are not enumerated.

## Appendix B. Key Insights for Netdata Trap Design

Stress-tested observations from the Cribl Stream analysis, against Netdata's hub architecture (per `.agents/sow/specs/snmp-traps/netdata-snmp-hub-architecture.md`):

1. **Bidirectional first-party trap support is a genuine differentiator** when present. Cribl is one of the very few systems with both inbound and outbound traps shipped as first-class features. Netdata should consider whether the SNMP hub model is bi-directional in its target use (Netdata-as-NMS-aggregator vs Netdata-as-trap-bridge) and design accordingly. Netdata's primary use is **reception** into the hub; northbound trap emission is rare in observability platforms but could be the bridge to legacy NMS estates (OpenNMS, Centreon).

2. **Pipeline-function composition over hard-coded trap semantics is a workable architecture, but it shifts burden to the operator.** Cribl proves the pattern works for sophisticated operators with strong destination infrastructure (Splunk, Elastic). For Netdata's "fit for purpose" approach (DevOps/SREs), the trade-off is the opposite: Netdata should ship more out-of-the-box semantics (severity mapping, common-MIB enrichment, basic storm handling) so that "install and it works" is a default experience. Cribl deliberately does the opposite; Netdata should not.

3. **MIB management is the recurring pain point in the entire cohort.** Cribl pushes MIB compilation entirely outside the product (CSV lookups). Telegraf and Logstash embed compilers but require operators to find and load MIBs. NMS-class systems ship thousands of MIBs preloaded — and their operators still complain about it. Netdata's choice: ship a curated set of standard + top-vendor MIBs preloaded (RFC 1213, IF-MIB, ENTITY-MIB, plus Cisco/Juniper/Arista/HP/Dell common families) with an explicit MIB-add workflow. Anything less ships an unfinished feature.

4. **The "no severity model out of the box" pattern (Cribl, Telegraf, Logstash) is a defensible choice but a poor user experience.** Operators do not want to write OID-to-severity tables for the IETF-generic traps — those mappings are well-known and stable for 25+ years. Netdata should ship default severity for at least the RFC 1215 generic traps (cold-start, warm-start, link-down, link-up, authentication-failure, egp-neighbor-loss) and a curated subset of common vendor traps, with operator override.

5. **The DES/AES discrepancy is a cautionary tale about API-vs-UI consistency.** Cribl's OpenAPI lists `des`; its public docs page lists only AES variants. Either DES is silently accepted-but-hidden, or the docs are out of date — and operators have no way to know without source. Netdata's API and UI must be in lock-step; the truth source must be one document, ideally machine-checked.

6. **Plaintext USM keys in versionable config is a real risk.** Cribl's schema models `authKey`/`privKey` as plain strings; if the config is Git-versioned (and Cribl encourages this), those keys are in Git. Netdata's hub should treat SNMPv3 credentials as Secrets from day one, with explicit Secrets binding rather than inline strings.

7. **The "no native UDP HA" pattern is the cohort default — but it is also the cohort weakness.** Almost no system in this pass ships active/passive UDP listener failover; operators are uniformly expected to bring an external LB or VIP. For Netdata's hub-per-site architecture (one Netdata Agent per site is the SNMP hub), this is mitigated: site-level HA is two-Agent active/passive with VIP. But Netdata should document this pattern explicitly because operators coming from NMS-class systems may assume it ships.

8. **Pipeline fan-out (one trap, many destinations) is a genuinely useful primitive.** Cribl's Routes Final=off cloning lets an operator simultaneously route a trap to OpenNMS (verbatim, as a real PDU) and to Splunk (as JSON) for the migration period. Netdata's hub should consider whether it can be the "trap proxy" for an estate migrating from a legacy NMS to a modern destination — supporting both verbatim re-emission and JSON-shape forwarding atomically.

9. **The closed-source nature of Cribl Stream means its trap behaviour is not auditable.** Among the 13 tier-1 open-source systems analysed in this pass, the trap decoder behaviour is grep-able and citable line-by-line; among the four docs-only systems (Cribl, SolarWinds, Dynatrace, LogicMonitor), it is not. Netdata's open-source posture is a genuine differentiator versus Cribl in regulated/security-sensitive environments where auditability of the trap reception path is a procurement requirement.

10. **First-class non-SNMP destinations matter.** Cribl's strongest pitch is "your traps can go to any of 70 destinations." For Netdata, integrating the hub trap path with Netdata's existing alert/metric ecosystem (so traps become alerts, become metrics) is the equivalent move — making the trap not "another data type" but part of the unified observability surface.
