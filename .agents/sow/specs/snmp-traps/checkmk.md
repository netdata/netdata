## 1. System Overview & Lineage

CheckMK is a GPLv2 monitoring platform descended from the Nagios family (originally `check_mk`, an extension to Nagios; later replaced its core with the Checkmk Micro Core in commercial editions). Primary audience is sysadmins / SREs running on-prem infrastructure (data centres, manufacturing/OT, MSPs). It is multi-process, mostly Python on top of C/C++ daemons (CMC core, Livestatus, helpers like `mkeventd_open514`). The Event Console (EC) is its event-correlation subsystem; in source it lives as the `cmk-ec` Bazel package at `packages/cmk-ec/cmk/ec/` and ships its daemon binary as `mkeventd`.

SNMP traps are **one event source among several** in EC — alongside syslog (UDP/TCP), an internal `events` named pipe (`tmp/run/mkeventd/events`), a Unix socket (`tmp/run/mkeventd/eventsocket`), and a spool directory (`var/mkeventd/spool/`). They share a single processing pipeline, a single rule engine, a single status / history backend, and a single set of GUI views. The Event Console is therefore architecturally a *generic event pipeline* in which SNMP traps are normalised into the same internal `Event` record as syslog messages and then matched against the same regex/glob rules. This is the central architectural choice of CheckMK's trap support and the chief difference from systems with a dedicated trap data model (OpenNMS `Trapd`, Zenoss `zentrap`).

- The daemon entry-point is `bin/mkeventd.py:9-12`, a one-line shim that calls `cmk.ec.main.main(cmk.utils.paths.omd_root, sys.argv)` (`bin/mkeventd.py`).
- Trap reception happens inside the **same** `EventServer` thread that receives syslog: `packages/cmk-ec/cmk/ec/main.py:362-429`, with all socket-opening (`open_syslog_udp`, `open_syslog_tcp`, `open_snmptrap`) and the `SNMPTrapParser` constructed in `EventServer.__init__`.
- SNMP parsing uses **PySNMP** directly inside the EC process (`packages/cmk-ec/cmk/ec/snmp.py:1-251`); CheckMK does **not** delegate to Net-SNMP's `snmptrapd`. The user docs are explicit: "Since the Event Console has its own built-in SNMP engine, setting the reception of SNMP traps is very simple. You do not need `snmptrapd` from the operating system! If you already have this running, stop it." (`checkmk-docs/src/onprem/en/ec.asciidoc:1254-1256`).
- MIB compilation uses **PySMI** (also direct, in-process), both at install time for the bundled MIBs (`omd/packages/net-snmp/compile_mibs.py`) and at runtime for operator-uploaded MIBs (`packages/cmk-ec/cmk/ec/snmp.py:378-407` `compile_mib(...)` and `cmk/gui/mkeventd/wato.py:3686-3712` `_validate_and_compile_mib(...)`).
- Net-SNMP is bundled in the OMD package only for its CLI tools (`snmptrap`, `snmpwalk`, `snmptranslate`, …) and the bundled MIB ASN.1 source files, not for the trapd daemon (`omd/packages/net-snmp/BUILD:10-30` enumerates `bin_paths` — `snmptrap`, `snmpwalk`, etc.; no `snmptrapd`).

A small community-treasures script exists if an operator *wants* to use the system's `snmptrapd` for some reason and pipe its output to the EC: `doc/treasures/Event_Console/snmptd_mkevent.py` (49 lines). It is explicitly a community-contributed wrapper, not the supported path. The supported path is the in-product PySNMP engine.

The SNMP `traphost` resource at `netapp-ontap-cmk/netapp_ontap/resources/snmp_traphost.py` is **NetApp ONTAP**'s autogenerated Python client (REST resource for registering a trap host on a NetApp filer); it is bundled into the CheckMK mirror because CheckMK can use the ONTAP API as a special agent, but it is *NetApp's outbound trap-destination configuration*, not CheckMK's inbound trap subsystem. It is mentioned here only to disambiguate the SOW hint; it is not part of CheckMK's trap implementation.

### Deployment models

- **Single-site (default)**: one `mkeventd` per OMD site; the Apache/CGI GUI reads its live status via the Unix socket `tmp/run/mkeventd/status` and accesses the history files locally (`checkmk-docs/src/onprem/en/ec.asciidoc:1791-1810`).
- **Distributed CheckMK with central EC**: remote sites run no EC; monitored hosts at remote sites send syslog/traps to the central site; remote sites still use `MessageForwarder` (`packages/cmk-ec/cmk/ec/forwarder.py:36-68`) to push *logwatch-derived* messages to the central EC over TCP. This is in-product but is for log forwarding, not for *trap forwarding*. For traps, the operator just retargets the device's trap destination to the central site.
- **Distributed CheckMK with per-site EC** ("Distributed Event Console"): each site runs its own `mkeventd`, captures traps locally, and the central GUI fetches event status/history from each site via Livestatus. Configuration is replicated from the central site via WATO when the connection has *Replicate Event Console configuration to this site* enabled. Documented in `checkmk-docs/src/onprem/en/distributed_monitoring.asciidoc:655-672`.
- **Active/standby replication** (available in both Checkmk Raw and commercial on-prem editions per `Checkmk/checkmk-docs :: src/onprem/en/distributed_monitoring.asciidoc:663`): the `replication` config block (`packages/cmk-ec/cmk/ec/config.py:88-94` `ReplicationBase`/`Replication`) lets a slave EC pull master state at the configured `replication["interval"]` seconds (GUI default **10s** at `cmk/gui/mkeventd/wato.py:4042-4050`; runtime use at `packages/cmk-ec/cmk/ec/main.py:2536-2539`). Note: `retention_interval` (default 60s, `packages/cmk-ec/cmk/ec/defaults.py:54`) is a separate setting — it controls how often the master flushes its in-RAM status to `var/mkeventd/status`, not how often the slave pulls. Slave can take over if master is down. There is **no Active/Active**: only one EC binds UDP/162 on a given host because of kernel port-exclusivity (`checkmk-docs/src/onprem/en/ec.asciidoc:90-92`, `:1262`).
- **HA**: no native cluster-coordinated trap reception at the `mkeventd` daemon level. The Checkmk Appliance HA documentation (`Checkmk/checkmk-docs :: src/onprem/en/appliance_cluster.asciidoc:176, :198`) describes the operator pattern: send traps to the cluster VIP, with active/standby failover handled by the Appliance / OS cluster, not by `mkeventd` itself.
- **Docker / Kubernetes**: CheckMK ships official container images (see `docker_image/Dockerfile`). The `mkeventd_open514` helper inside the container relies on the same packaging logic (capabilities-first, setuid-fallback per `omd/debian/postinst.in:76-87`). In container runtimes, granting `cap_net_bind_service` to the container (`--cap-add=NET_BIND_SERVICE` for Docker, the equivalent securityContext for Kubernetes) is the standard way to allow the helper to bind UDP/162 without root — this is a general Linux/container observation; the exact recommended container configuration was not source-verified in the Checkmk repo in this pass.

### Languages and key libraries

- **Python 3.13** in the analysed commit (`pyproject.toml:167` sets `python_version = "3.13"`) for the entire `mkeventd` daemon (`packages/cmk-ec/cmk/ec/main.py:1-3631`) and the WATO GUI (`cmk/gui/mkeventd/*.py`).
- **PySNMP** for SNMP PDU parsing, USM, MIB resolution (`packages/cmk-ec/cmk/ec/snmp.py:15-27`).
- **PySMI** for compiling user-uploaded MIBs at runtime (`packages/cmk-ec/cmk/ec/snmp.py:29-37, :378-407`) and the bundled MIBs at install time (`omd/packages/net-snmp/compile_mibs.py`).
- **C++** for one tiny helper: `bin/mkeventd_open514.cc` (162 lines, normally installed with `cap_net_bind_service+ep` file capabilities, SUID root only as fallback when `setcap` fails or capabilities are unsupported; see `omd/omd.spec.in:71` for RPM caps spec and `omd/debian/postinst.in:76-87` for Debian capability-first / setuid-fallback logic).
- **SQLite** (Python stdlib) for one of the three history backends (`packages/cmk-ec/cmk/ec/history_sqlite.py:1-388`).
- **PyMongo** for the optional MongoDB history backend (`packages/cmk-ec/cmk/ec/history_mongo.py:1-234`).
- **No SNMP4J / Net-SNMP daemon dependency** at runtime for trap reception; PySNMP supplies everything.

### Inter-component IPC

- **Unix status socket** (`tmp/run/mkeventd/status`): clients send a simplified Livestatus query (`GET events…`, `GET history…`, `GET status…`) or a one-word `COMMAND` (`RELOAD`, `FLUSH`, `SHUTDOWN`, `DELETE`, `UPDATE`, `CHANGESTATE`, `ACTION`, …) via `unixcat`. Protocol restrictions: only `Filter:` and `OutputFormat:` headers allowed, no `KeepAlive`, one request per connection (`checkmk-docs/src/onprem/en/ec.asciidoc:1796-1834`).
- **Named pipe** (`tmp/run/mkeventd/events`): single-writer event-injection FIFO. Anything written here is parsed as a syslog-shaped message and run through the same pipeline. This is how `cmk --logwatch` and the `MessageForwarder` (for spooled fallback) deliver messages.
- **Event socket** (`tmp/run/mkeventd/eventsocket`): same purpose as the pipe but supports concurrent writers; mode 0660. Wired from `EventServer.open_eventsocket()`.
- **In-process call chain** for traps: `_snmp_trap_socket.recvfrom(65535)` → `create_events_from_trap(...)` (a generator) → `SNMPTrapParser(...).parse(...)` → `SNMPTrapTranslator(...).translate(...)` → `create_event_from_trap(...)` → `process_potential_event_instrumented(...)` → `process_potential_event(...)`. All in the same thread (`packages/cmk-ec/cmk/ec/main.py:754-760, :770-779, :1266`). No inter-thread queue, no async dispatcher.
- **History writer**: no dedicated thread. Trap-ingestion history writes (`add(event, "NEW", ...)` etc.) happen synchronously inside the `EventServer` thread (`main.py:810, :930, :1085, :1112, :1384`). Operator-command history writes (acknowledge, change-state, delete, archive) happen synchronously inside the `StatusServer` thread (`main.py:2318, :2340`). The chosen backend (`file` | `sqlite` | `mongodb`) is a Python dispatcher behind the `History` ABC (`packages/cmk-ec/cmk/ec/history.py:37-50`); the backend itself does not spawn worker threads.
- **Notification IPC**: `do_notify(...)` (`packages/cmk-ec/cmk/ec/actions.py:325-440`) builds a `CMK_NOTIFY_*` env block and invokes `cmk --notify` via `subprocess.run`. This is how an event becomes a CheckMK notification routed through the standard notification rules.

## 3. Trap Reception (UDP/162 Ingress)

### Listener implementation

CheckMK opens its **own** UDP socket directly (no `snmptrapd`). Key lines from `packages/cmk-ec/cmk/ec/main.py:604-644` (`EventServer.open_snmptrap`):

- Three branches: (a) inherited file descriptor from the SUID helper (`endpoint.value` is a `FileDescriptor`) — `socket.fromfd(...)`; (b) IPv6 dual-stack bind on `("::", port)`; (c) fallback to IPv4 `("0.0.0.0", port)`.
- `setsockopt(SOL_SOCKET, SO_REUSEADDR, 1)` is set on the freshly-created socket (`main.py:626, :640`). On IPv6 it also tries to `setsockopt(IPPROTO_IPV6, IPV6_V6ONLY, 0)` for dualstack (`main.py:627-635`).
- The socket is then `recvfrom(65535)` inside the `select()` loop at `main.py:754-760`.
- The actual PDU decode is handed off to `SNMPTrapParser.parse(...)` (`packages/cmk-ec/cmk/ec/snmp.py:170-185`), which feeds the bytes back into PySNMP's `message_dispatcher.receive_message(...)` so the framework's standard `NotificationReceiver` callback (`_handle_snmptrap`) fires.

### SNMP version support

- **SNMPv1**: full PDU support — see `SNMPTrapParser._ECNotificationReceiver.SUPPORTED_PDU_TYPES` enumerating `pysnmp.proto.api.v1.TrapPDU.tagSet` (`packages/cmk-ec/cmk/ec/snmp.py:50-53`).
- **SNMPv2c**: notifications (`pysnmp.proto.api.v2c.SNMPv2TrapPDU.tagSet`, same lines).
- **SNMPv3**: USM auth/priv with the following protocol matrix (`packages/cmk-ec/cmk/ec/snmp.py:83-115`):
  - Auth: `md5`, `sha`, `SHA-224`, `SHA-256`, `SHA-384`, `SHA-512` (six options; the legacy short names `md5`/`sha` map to HMAC-MD5/HMAC-SHA-96).
  - Privacy: `DES`, `3DES-EDE`, `AES`, `AES-192`, `AES-256`, `AES-192-Blumenthal`, `AES-256-Blumenthal` (seven options).
  - EngineId is recognised (`add_v3_user(..., securityEngineId=OctetString(hexValue=engine_id))`, `snmp.py:160-168`) and the GUI documents it as: "As the engine ID is used for the encryption of SNMPv3 traps sent by the devices, Checkmk needs to know the engine ID to be able to decrypt the SNMP traps." (`cmk/gui/mkeventd/wato.py:4626-4631`).
  - **Engine IDs are effectively required for SNMPv3 reception** — the v3-user registration loop iterates `for engine_id in spec.get("engine_ids", [])` (`snmp.py:152-168`). If a credential entry has an empty `engine_ids` list, *no* v3 user is registered for that credential at all, so a matching SNMPv3 trap arriving will fail authentication. The GUI does not make `engine_ids` a required field (it is `optional_keys=["engine_ids"]` at `wato.py:4636`), but in practice an SNMPv3 credential with no engine IDs is non-functional.
  - Multiple engine IDs per user are allowed: the user-id is registered once for each engine_id in the list.
- **SNMPv3 INFORM**: **NOT supported**. The receiver class explicitly restricts itself to `TrapPDU` and `SNMPv2TrapPDU`: `SUPPORTED_PDU_TYPES` (`packages/cmk-ec/cmk/ec/snmp.py:50-53`). The class-level comment is candid: `# Disable receiving of SNMPv3 INFORM messages. We do not support them (yet)`. SNMPv2c INFORM is also therefore not handled (no `pysnmp.proto.api.v2c.InformRequestPDU.tagSet` is registered). This is a real and signalled gap.
- **DTLS / TLS-TM (RFC 5953/6353/9456)**: **NOT supported**. The transport is a hard-coded UDP socket; PySNMP's TLS/DTLS transports are not referenced.
- **IPv6**: yes, dual-stack with IPv4-mapped fallback (`main.py:622-641`). IPv4-mapped IPv6 source addresses are normalised back to IPv4 via `unmap_ipv4_address(...)` (`main.py:266-273`) so a trap from `::ffff:192.0.2.5` appears in events as `192.0.2.5`.
- **SNMPv3 contexts**: the receiver captures `contextEngineId` and `contextName` from the PDU and logs them at VERBOSE (`snmp.py:213-216`), but **does not** use them for access control, VACM-style filtering, or event routing. All contexts are treated identically by the rule engine. Operators wanting context-based isolation must implement it at a different layer (e.g., per-VRF UDP forwarders).

### Concurrency model

- Single-thread reception. The `EventServer` thread `select()`s across `(pipe, syslog_udp, snmp_trap_socket)` (`main.py:669, :706`).
- One trap PDU is parsed, translated, matched, rewritten, and dispatched **fully synchronously** before the next packet is read. There is no queue, no worker pool, no async dispatcher.
- Performance design assumption: rule matching is "a couple of ms" (`packages/cmk-ec/cmk/ec/actions.py:140`: `# Rule actions are currently done synchronously. Actions should not hang for more than a couple of ms.`).
- The end-user docs are open about the scale limit: "As soon as the number of current events exceeds about 5 000, the performance begins to deteriorate noticeably." (`checkmk-docs/src/onprem/en/ec.asciidoc:1667`). And: "The Event Console always keeps all current events in RAM. These are written to the `~/var/mkeventd/status` file once per minute (adjustable) and on a clean completion of processing. If this becomes very large (e.g. over 50 megabytes), this process also becomes slower and slower." (`ec.asciidoc:1672-1676`).
- The Python stdlib UDP socket has no userland buffer beyond what the kernel allocates; back-pressure is "drop in kernel when SO_RCVBUF full." There is no per-source rate limit at the listener layer; protection is at the *event count* layer downstream (see §10).

### Privileged-port handling

Two-stage approach with **file-capability-preferred** installation:

1. **`bin/mkeventd_open514`** (`bin/mkeventd_open514.cc`, 162 lines, GPLv2): a privileged-port-binder C++ helper that parses `--snmptrap`, `--snmptrap-fd <N>`, `--syslog`, `--syslog-fd <N>`, `--syslog-tcp`, `--syslog-tcp-fd <N>`. For each enabled service it calls `getaddrinfo(nullptr, "snmptrap", …)` to resolve the well-known port (162), opens a UDP socket, `setsockopt(SO_REUSEADDR)`, `bind()`, and then `dup2`s the FD to the requested number (3 for syslog UDP, 4 for syslog TCP, 5 for SNMP trap). It then does `seteuid(getuid())` to drop privileges (only meaningful if running setuid; harmless otherwise) and transfers control to `mkeventd` (`mkeventd_open514.cc:153-160`). The packaging tries `setcap cap_net_bind_service+ep` first (RPM: `omd/omd.spec.in:71` declares `%caps(cap_net_bind_service+ep) .../bin/mkeventd_open514`; Debian: `omd/debian/postinst.in:76-87` runs `setcap` and falls back to `chmod 4750` only if `setcap` fails or returns non-zero). On Linux kernels with capability support and a filesystem that honours xattrs, the binary is NOT setuid root — it has only the file capability to bind privileged ports. The legacy SUID-root mode is a packaging fallback.
2. **`mkeventd`** sees the inherited FDs via `--snmptrap-fd 5` (etc.) and uses `socket.fromfd(...)` to wrap them as Python sockets (`main.py:608-619`).

Init wiring at `omd/packages/check_mk/skel/etc/init.d/mkeventd:50-84`: when `MKEVENTD_SYSLOG`/`MKEVENTD_SYSLOG_TCP`/`MKEVENTD_SNMPTRAP` are `on`, the script prepends `--syslog --syslog-fd 3 --syslog-tcp --syslog-tcp-fd 4 --snmptrap --snmptrap-fd 5` and invokes `${DAEMON}_open514`. Otherwise it just runs `mkeventd` directly without privilege escalation, and SNMP trap reception is disabled.

The `omd config` hook controls this — `omd/packages/check_mk/MKEVENTD_SNMPTRAP`:
- default: `off`
- choices: `on`/`off`
- depends on `CONFIG_MKEVENTD=on` (`MKEVENTD_SNMPTRAP:20-22`)

Practical implication: **out-of-the-box on a freshly-installed CheckMK site, SNMP trap reception is OFF**. The operator must `omd config set MKEVENTD_SNMPTRAP on` (a deliberate `omd stop; omd config …; omd start` sequence). This is documented at `checkmk-docs/src/onprem/en/ec.asciidoc:1252-1276`. Once turned on, `omd start` prints `Starting mkeventd (builtin: syslog-udp,snmptrap)...OK` (`ec.asciidoc:1268`).

### Horizontal scaling pattern

There is **no built-in horizontal scaling** for trap reception. The recommended pattern is per-site EC (Distributed Event Console), with each site collecting traps from a defined set of devices. Cross-site dedup is not implemented; each site's EC is independent (`distributed_monitoring.asciidoc:655-672`).

### HA / clustering

- Active/standby via the `replication` config block (`packages/cmk-ec/cmk/ec/config.py:88-94`). The slave pulls the master's full state every `replication["interval"]` seconds (GUI default 10s at `cmk/gui/mkeventd/wato.py:4042-4050`; runtime at `main.py:2536-2539`). `is_replication_slave(...)` (`main.py:792`) gates event processing on the slave when sync mode is active.
- No native cluster coordination at the UDP layer at the EC daemon level. Operators put a VIP in front; the failover happens at the network layer, not in EC. **Note on Checkmk Appliance HA**: the official Checkmk Appliance documentation (`Checkmk/checkmk-docs :: src/onprem/en/appliance_cluster.asciidoc:176, :198`) does state that incoming SNMP traps to a cluster should be sent to the cluster VIP. So the VIP pattern *is* documented at the Appliance / OS-cluster level — just not at the `mkeventd` daemon level.

## 4. MIB Management

### MIB store location and layout

Five directory candidates are searched. The `AnnotatedPath` definitions live in `packages/cmk-ec/cmk/ec/settings.py:80-88`, but the actual runtime search order (the order in which `MibBuilder.add_mib_sources(...)` is called) is at `packages/cmk-ec/cmk/ec/snmp.py:282-288`:

| Order | Path (relative to OMD site root) | `AnnotatedPath` description | Role |
|---|---|---|---|
| 1 | `local/share/snmp/mibs/` | "custom MIB directory" | Operator's hand-uploaded ASN.1 sources |
| 2 | `share/snmp/mibs/` | "Checkmk MIB directory" | Bundled-with-CheckMK ASN.1 sources |
| 3 | `/usr/share/snmp/mibs/` | "system MIB directory" | Net-SNMP system-wide ASN.1 sources |
| 4 | `local/share/check_mk/compiled_mibs/` | "compiled MIBs directory" | Per-site compiled Python output (write target for user uploads) |
| 5 | `share/snmp/compiled_mibs/` | "Checkmk Compiled MIB directory" | Bundled-with-CheckMK compiled Python output |

The `compile_mibs` Bazel rule (`omd/packages/net-snmp/snmp-rules.bzl:1-37`) processes `@net-snmp//:mibs` (the ASN.1 source tree shipped with the upstream Net-SNMP tarball) at install time and writes Python modules into `share/snmp/compiled_mibs/`. The `compile_mibs.py` script (`omd/packages/net-snmp/compile_mibs.py:1-63`) uses PySMI's `MibCompiler` with `SmiV1CompatParser` + `PySnmpCodeGen` and the standard `StubSearcher(*PySnmpCodeGen.baseMibs)` to avoid recompiling MACRO-bearing roots.

### Compilation pipeline (user-uploaded MIBs)

`packages/cmk-ec/cmk/ec/snmp.py:378-407` (`compile_mib(...)`): the operator-uploaded text is fed through a `CallbackReader` (returns the content only for the matching MIB name) plus `FileReader`s for the source directories enumerated above. The result is written to `destination_dir` (`local/share/check_mk/compiled_mibs/`). Errors are tolerated (`ignoreErrors=True`).

The GUI binding is `cmk/gui/mkeventd/wato.py:3686-3712` (`ModeEventConsoleUploadMIBs._validate_and_compile_mib`). For each uploaded module the code reports per-dependency status: `failed`, `missing`, or success; the user message lists `<dep_name> - Dependency missing` for each missing dependency.

### Bundled MIBs out-of-the-box (vendor coverage)

The bundled-with-CheckMK MIB set is **only what comes with upstream Net-SNMP** — the standard MIBs (`SNMPv2-MIB`, `IF-MIB`, `IP-MIB`, `RFC1213-MIB`, `IANAifType-MIB`, `HOST-RESOURCES-MIB`, etc.) plus those Net-SNMP ships for its own agent. No third-party vendor MIBs are bundled.

The user manual discloses the gap directly: "Regrettably, the advantages of open source have not yet spread to the authors of MIB files, and so we from the {CMK} project are unfortunately not able to deliver vendor specific MIB files. Only a small collection of free basic MIBs is preinstalled, which for example provides for a translation of `sysUpTime`." (`checkmk-docs/src/onprem/en/ec.asciidoc:1357`). The screenshot at `checkmk-docs/images/ec_mibs_for_translation.jpg` shows an example with Netgear Smart Switches MIBs hand-uploaded.

This is a stark contrast with OpenNMS, which ships hundreds of example `.events.xml` files covering many vendors (see the comparison-matrix and the OpenNMS analysis at `.agents/sow/specs/snmp-traps/opennms.md:191-194` for the exact corpus counts). CheckMK's design philosophy is: "you supply vendor MIBs; we give you a translator."

### User workflow for adding/updating MIBs

- **UI** (recommended): WATO → Setup → Event Console → *SNMP MIBs for trap translation* → *Add one or multiple MIBs* (`cmk/gui/mkeventd/wato.py:3352-3544`). Accepts single `.mib` files or `.zip` collections (`wato.py:3608-3637`, `_is_zipfile`/`_process_uploaded_zip_file`). `.tar`/`.gz`/`.tgz` is explicitly rejected: `raise Exception(_("Sorry, uploading TAR/GZ files is not yet implemented."))` (`wato.py:3618`).
- **CLI**: copy ASN.1 files directly into `~/local/share/snmp/mibs/`. The end-user docs explicitly comment this out as an alternative (`Checkmk/checkmk-docs :: src/onprem/en/ec.asciidoc:1366-1367`), because the runtime resolver only loads *compiled* MIBs. Runtime compilation is wired only through the WATO upload pipeline (`cmk/gui/mkeventd/wato.py:3690-3716`). Operators who place ASN.1 files in `local/share/snmp/mibs/` must therefore separately run `pysmi-mib2pysnmp` to produce compiled PySNMP modules under `local/share/check_mk/compiled_mibs/`, or re-upload via the GUI to get the compile step. There is no auto-compile-on-start mechanism for files dropped into the MIB directories.
- **Dependency resolution**: PySMI walks the search dirs; if a dependency is missing, the upload reports `<dep> - Dependency missing` in the GUI flash message (`wato.py:3701-3704`). The operator must then upload the missing MIB. There is no automatic fetch from any registry.
- **Versioning**: none. The file at `local/share/snmp/mibs/<name>.mib` is overwritten on re-upload; compiled output at `local/share/check_mk/compiled_mibs/<NAME>.py` is regenerated. No history is kept. No mapping to firmware versions.

### Loading at daemon start

`SNMPTrapTranslator._construct_resolver(...)` (`packages/cmk-ec/cmk/ec/snmp.py:273-317`):

- Creates a `pysnmp.smi.builder.MibBuilder()`.
- Adds all five `DirMibSource` directories above (in precedence order).
- Calls `builder.load_module(module_name)` for **every** module file found in **every** source dir. Each module's load wrapped in try/except `pysnmp.smi.error.MibNotFoundError` and `pysnmp.smi.error.SmiError`; on failure logs `Skipping MIB module %s: missing dependency (%s)` or `Failed to load MIB module %s` and **continues**. Final tally: `logger.info("Loaded %d SNMP MIB modules", len(loaded_mib_module_names))`.
- If translation is disabled (`translate_snmptraps = False`), no resolver is built (`snmp.py:256-264`).
- Two translation modes:
  - `(True, {})`: translate OID → name, no description text (`load_texts=False`, `snmp.py:260-264`).
  - `(True, {"add_description": True})`: translate **and** append the MIB-documented unit / description text to each varbind value (`snmp.py:265-269`). The end-user docs warn: "the whole thing gets much more detailed -- and more confusing" (`ec.asciidoc:1349-1350`).

### Fallback behaviour for unknown OIDs

Without translation (default — see §13): `_translate_simple` (`packages/cmk-ec/cmk/ec/snmp.py:319-332`) returns `(str(oid), value.prettyPrint())`. The OID stays numeric. One special case: `sysUpTime.0` (OID `1.3.6.1.2.1.1.3.0`) is renamed to the literal `"Uptime"` for prettier display (`snmp.py:326`); `TimeTicks` values are formatted via `_approx_age` (e.g. `"3.2 d"`, `"45 m"`, lines 412-442).

With translation but unresolvable OID: `_translate_binding_via_mibs` (`snmp.py:359-375`) raises `SmiError` or `ValueConstraintError` for that varbind only; the outer loop in `_translate_via_mibs` (`snmp.py:334-357`) catches it, logs a warning, and falls back to `(str(oid), str(value))` for *that varbind only* — translation of the others continues. This is graceful degradation per-varbind, not per-trap.

## 5. Trap Processing Pipeline

End-to-end (single `mkeventd` thread):

### Step 1: UDP receive

`main.py:754-760`. `recvfrom(65535)`. The full datagram plus the sender address tuple `(host, port)` arrives at the next step.

### Step 2: PDU decode and authentication (PySNMP)

`SNMPTrapParser.parse(data, sender_address)` at `snmp.py:170-185`:

- Sets `sender_address` on the SNMP engine's user context so the trap handler can recover the source IP.
- Calls `snmp_engine.message_dispatcher.receive_message(...)` with the wholeMsg.
- PySNMP dispatches based on SNMP version: v1 → `MPv1`, v2c → `MPv2c`, v3 → `MPv3` with USM-based authentication. If authentication fails (unknown community for v1/v2c, unknown user/wrong password/wrong engineId for v3), the `_handle_unauthenticated_snmptrap` observer fires (`snmp.py:222-250`) and the trap is **silently dropped** — only a `VERBOSE` log line is emitted: `Trap (v%d) dropped from %s: %s`. There is no failure counter that increments to operator visibility.

### Step 3: Varbind extraction and OID-to-name resolution

`SNMPTrapParser._handle_snmptrap(...)` fires (`snmp.py:187-201`) on a successfully authenticated PDU:

- Retrieves the sender IP from the engine's user context.
- Logs verbose details: `Trap accepted from %s (ContextEngineId "...", SNMPContext "...")` followed by one `%-40s = %s` line per varbind (`snmp.py:203-220`).
- Hands the varbinds to `SNMPTrapTranslator.translate(ipaddress, var_binds)` which dispatches to one of three methods chosen at construction time (`snmp.py:256-271`):
  - `_translate_simple` (no translation, default — see §13).
  - `_translate_via_mibs` (MIB-based, names only).
  - `_translate_via_mibs` (MIB-based, names + descriptions).

The result is `tuple[Iterable[tuple[str, str]], str]` — a list of `(name_or_oid, formatted_value)` pairs plus the sender IP.

### Step 4: Build the internal `Event` record

`create_event_from_trap(trap, ipaddress)` (`main.py:1778-1794`):

```
def create_event_from_trap(trap, ipaddress_):
    """New event with the trap OID as the application."""
    trapOIDs, other = partition(
        lambda binding: binding[0] in {"1.3.6.1.6.3.1.1.4.1.0", "SNMPv2-MIB::snmpTrapOID.0"}, trap
    )
    return Event(
        time=time.time(),
        host=HostAddress(scrub_string(ipaddress_)),
        ipaddress=scrub_string(ipaddress_),
        priority=5,    # notice
        facility=31,   # not used by syslog -> we use this for all traps
        application=scrub_string(trapOIDs[0][1] if trapOIDs else ""),
        text=scrub_string(", ".join(f"{oid}: {value}" for oid, value in other)),
        core_host=None,
        host_in_downtime=False,
    )
```

This is the heart of the trap-to-event normalisation. Three points worth highlighting:

- **The trap OID (`snmpTrapOID.0`) becomes the event's `application` field.** Every other varbind concatenated as `oid: value, oid: value, ...` becomes the `text` field. The structural consequence: rules with `match_application` filter by trap OID type; rules with `match` (regex on `text`) filter by varbind content. The full structured varbind set is not preserved as a list — it is flattened into a single string at this step, which limits what kinds of cross-varbind logic a rule can express (see §17 weakness on flat schema).
- **The host field starts as the sender's IP.** Hostname resolution (via the monitored-hosts cache, see `HostConfig` below) is performed later in `do_translate_hostname` / `_add_core_host_to_new_event`.
- **SNMPv1 `agent-addr` is NOT used for host identity.** RFC 1157 SNMPv1 traps carry an `agent-addr` field that names the originating device (which can differ from the UDP packet's source IP when a proxy or NAT is in the path). CheckMK does not read this — `_handle_snmptrap` (`snmp.py:196-201`) takes the sender from the SNMP engine's user context (set from the UDP `recvfrom` address at `snmp.py:177-178`), and `create_event_from_trap` stores that UDP source as both `host` and `ipaddress`. The `agent-addr` value, if present, is just one varbind among many in the rendered `text` field. For NAT/proxy/VRF trap paths, this means CheckMK's host identification will reflect the *last hop* (the proxy), not the *origin*.
- **Syslog facility is hard-coded to 31 ("snmptrap").** This is officially out of the RFC 5424 range (0-23) — the code comment is explicit: `# everything above that is for internal use. see: https://datatracker.ietf.org/doc/html/rfc5424#section-6.2.1` (`packages/cmk-ec/cmk/ec/syslog.py:64-65`). Priority is hard-coded to 5 ("notice"). This unifies traps with syslog: rules can match `match_facility=31` to scope to traps only, but a trap-only rule must match facility 31, not anything more specific. Severity has to come from the rule action (`rewrite_event`'s `state` setting), not from any vendor-supplied severity field in the trap itself — because there is no such field in SNMPv1/v2c/v3 traps anyway.

### Step 5: Source identification — IP → monitored-host mapping

`do_translate_hostname(event)` (`main.py:1522-1527`) applies the `hostname_translation` regex rules; `_add_core_host_to_new_event(event)` (called inside `process_potential_event`, `main.py:1356`) consults `HostConfig`:

`HostConfig` (`packages/cmk-ec/cmk/ec/host_config.py`):
- Fetches all monitored hosts from the local monitoring core via Livestatus on first access.
- Builds two dicts: `_hosts_by_name` (canonical host name) and `_hosts_by_designation` (`address` lowercased, `alias` lowercased, name lowercased, all pointing back at the canonical name).
- `get_canonical_name(event_host_name)` returns the canonical name if any of the three designations matches; otherwise `None`.
- Refresh policy: re-fetch only when the monitoring core's `status_program_start` timestamp advances (`_update_cache_after_core_restart`, lines 52-66). Renames between core restarts are stale.

Net effect: a trap from `192.168.1.10` becomes `core_host="switch-1"` if a host has `address=192.168.1.10` (or matching alias) in the monitoring core's host database. If no match, `core_host=None` and the event still goes through but is unattached to any monitored host. The end-user docs note: "Here the IP address of the sender is compared with the cached IP addresses of the hosts in the monitoring -- i.e. as soon as regular active checks are available, such as the accessibility check of the Telnet or SSH port at a switch, status messages from this device sent via SNMP will be assigned to the correct host." (`ec.asciidoc:323-324`).

### Step 6: Rule matching

`process_potential_event(event)` (`main.py:1266-1473`):

- If `rule_optimizer` is enabled (default: True per `defaults.py:41`), pick candidates from `self._rule_hash[event["facility"]][event["priority"]]` — a precomputed list of rules that could possibly match this `(facility, priority)` combination. Built by `hash_rule` / `hash_rule_facility` (`main.py:1212-1244`); see §15 for the design lesson. **For traps that means `_rule_hash[31][5]`** — the bucket contains: (a) non-inverted rules without an explicit `match_facility` (they hash to *all* facilities), (b) non-inverted rules with `match_facility=31`, and (c) **all inverted rules** (`invert_matching=True` causes `hash_rule` to hash the rule across all 32 facilities regardless of `match_facility` — `main.py:1213-1218`); the actual facility test then fires in `RuleMatcher.event_rule_matches_facility` (`rule_matcher.py:348-350`) with the inversion applied by `event_rule_matches` (`rule_matcher.py:216-228`).
- For each candidate `rule`, call `event_rule_matches(rule, event)` (`packages/cmk-ec/cmk/ec/rule_matcher.py:216-228`). This evaluates in order:
  - `event_rule_matches_site` (`SiteId` whitelist) — `rule_matcher.py:330-334`.
  - `event_rule_matches_host` (regex on `event.host`) — `rule_matcher.py:336-342`.
  - `event_rule_matches_ip` (CIDR check on `event.ipaddress` via `ipaddress.ip_network(...)`) — `rule_matcher.py:344-351`.
  - `event_rule_matches_facility` (integer equality) — `rule_matcher.py:353-357`.
  - `event_rule_matches_service_level` (range) — `rule_matcher.py:359-368`.
  - `event_rule_matches_timeperiod` (calls back into the monitoring core for `is_active_time_period`) — `rule_matcher.py:370-374`.
  - `event_rule_matches_syslog_application` (regex on `event.application` — i.e. *on the trap OID string*) — `rule_matcher.py:392-413`.
  - `event_rule_matches_message` (regex on `event.text` — i.e. *the joined varbind string*) — `rule_matcher.py:415-435`.
- A successful match returns `MatchSuccess(cancelling, match_groups)`. The matcher distinguishes "match-as-new-event" from "match-as-canceling-event" via `cancel_priority`/`cancel_application`/`match_ok` (the latter cancels open events whose text matches the OK pattern) — `rule_matcher.py:222-282`.
- Inverted matching is supported via `invert_matching: True` on a rule (`rule_matcher.py:226-238`).

Regex compilation is lazy and pattern-stripping is applied: a pattern that starts with `.*` has those two chars dropped, because `.*` is "redundant and dramatically destroys performance when doing an infix search" (`rule_matcher.py:43-69` `compile_matching_value`; the function's docstring includes a benchmark table). Patterns lacking regex metacharacters are kept as lowercase strings and matched with `in` (substring).

### Step 7: Rule outcome — rewrite, count, open, action

- **Drop**: `rule.get("drop")` (`main.py:1310-1316`). Two flavours: `True` (drop event entirely), `"skip_pack"` (skip the rest of this rule pack, useful as an early gate).
- **Cancelling**: if the match is a cancelling one, all currently-open events that share the matched groups are closed with action `CANCELLED` and history is appended.
- **Rewriting**: `rewrite_event(rule, event, match_groups)` (`main.py:1475-1521`):
  - Sets `event["state"]` from the rule (-1 means "derive from priority": prio ≤ 3 → CRIT, prio == 4 → WARN, else OK; or a `("text_pattern", {...})` tuple lets the rule classify by text regex).
  - Substitutes `set_text`, `set_host`, `set_application`, `set_contact`, `set_comment` using `replace_groups(template, original, match_groups)` (positional `\1`, `\2`, … from the regex).
- **Counting** (volumetric aggregation): if `"count"` in rule, the matching event is folded into an existing counting event (`_event_status.count_event`); when the count threshold is reached, the event transitions from `counting` to `open` and actions fire.
- **Action execution**: `event_has_opened(...)` (`actions.py:72-109`) iterates the rule's `actions` list:
  - `"@NOTIFY"`: build the notification context and invoke `cmk --notify` via `subprocess.run`.
  - Otherwise look up the action by ID in `config["action"]` and call `do_event_action`, which dispatches to:
    - `_do_email_action`: runs `mail -S sendcharsets=utf-8 -s <subject> <to>` via `subprocess.run` (`actions.py:223-256`).
    - `_do_script_action` (thin wrapper at `actions.py:189-205`) → `_execute_script` (`actions.py:258-280`): runs `/bin/bash` with the script body as stdin and the event fields injected as `CMK_*` env vars (see `_get_event_tags` at `actions.py:284-307`).

### Step 8: Persistence

The event is appended to in-RAM `EventStatus` (`main.py:2583+`); a `NEW` row is written to the chosen history backend (file / sqlite / mongodb); the in-RAM dump is flushed to `var/mkeventd/status` once per minute or at clean shutdown (default `retention_interval=60`, `defaults.py:54`).

### Error handling

- Malformed PDU (any exception in `SNMPTrapParser.parse`): logged at DEBUG (`main.py:771-776`: `self._logger.debug("skipping unparsable SNMP trap, reason: %s", e)`). The packet is silently dropped. There is no malformed-PDU counter at the perfcounter layer.
- Unauthenticated PDU: `_handle_unauthenticated_snmptrap` logs at VERBOSE only (`snmp.py:222-250`). Two specific error codes are decoded:
  - `unknownCommunityName` → `"Unknown community (<name>)"`.
  - `unknownSecurityName` → `"Unknown credentials (msgUserName: <name>)"`.
- Translation failure (per-varbind): falls back to numeric OID + raw value, warns once per varbind (`snmp.py:334-357`).
- Rule-matching exception: caught in `process_potential_event` (`main.py:1283-1289`), logged as exception, treated as `MatchFailure(reason="Rule would match, but due to inverted matching does not. {e}")`. The trap is *not* dropped; matching continues with the next rule. Note that the inverted-match label is a quirk of error message reuse — the inversion is the only place this reason is otherwise used — and unrelated to whether the rule actually had `invert_matching=True`.
- Action execution exception: caught in `do_event_action` (`actions.py:155-170`); logged but the event itself is opened normally.

## 6. Data Model and Persistent Storage

CheckMK does **not** use a relational database for the EC; configuration is Python source files on disk, runtime state is one big in-RAM dict, and history is one of three backends.

| Item | Where it lives | Format / schema | Retention |
|---|---|---|---|
| Configured rules and rule packs | `etc/check_mk/mkeventd.d/wato/rules.mk` (and `**/*.mk` under the same dir tree) | Python `rule_packs = [{...ECRulePackSpec...}]` (PPRINTed); loaded by `exec()` (`packages/cmk-ec/cmk/ec/rule_packs.py:105-180`) | Forever; changes go through WATO and git-style change tracking |
| Global EC config (translate_snmptraps, snmp_credentials, history_lifetime, …) | `etc/check_mk/mkeventd.d/wato/global.mk` | Python keyword bindings | Forever |
| Active (post-distribution) config | `var/mkeventd/active_config/` (`settings.py:60-62`) | Same as above, filtered for site-relevance | Replaced on each activation |
| Currently-open events (the working set) | `var/mkeventd/status` | Python `repr(...)` dict, deserialized via `ast.literal_eval` | Flushed every `retention_interval` (60s) and on clean shutdown |
| Per-trap SNMP credentials | inside `snmp_credentials` in `global.mk` | `Collection[SNMPCredential]` per `config.py:310-316` (description + credentials + engine_ids) | Forever |
| Trap history | `var/mkeventd/history/<unix-ts>.log` OR `var/mkeventd/history.sqlite` OR MongoDB (per `archive_mode`) | TSV (file) / one row per HistoryWhat (sqlite) / one BSON doc per event (mongo) | `history_lifetime` days, default **365** (`defaults.py:59`) |
| Raw trap bytes | **not persisted** | n/a — only the decoded `Event` record (with `application=trap-OID` and `text=joined-varbinds`) is kept | n/a |
| MIB ASN.1 sources (user-uploaded) | `local/share/snmp/mibs/<file>.mib` | Plain text ASN.1 | Forever (until manually deleted via WATO) |
| MIB ASN.1 sources (bundled) | `share/snmp/mibs/` | Plain text ASN.1 | Replaced on package upgrade |
| Compiled MIBs (PySNMP) | `local/share/check_mk/compiled_mibs/<NAME>.py` (user) and `share/snmp/compiled_mibs/` (bundled) | Python source from PySMI codegen | Replaced on upload / package upgrade |
| Rule hit counters | `EventStatus` in-RAM (`event_status.count_rule_match(...)`) | dict[rule_id] = int | Until restart or `RESETCOUNTERS` command |
| Slave-mode status | `var/mkeventd/slave_status` | Python dict | Always present in replication slave |
| Event-pipe spool overflow | `var/mkeventd/spool/` | one file per chunk | Until processed or rotated by `_load_spooled_messages` (`forwarder.py:271-335`) |

### History backend choice — `archive_mode` (default: `sqlite`)

`packages/cmk-ec/cmk/ec/defaults.py:67`: `archive_mode="sqlite"`.

- **`file`** (`history_file.py`, 358 lines): one log file per rotation period (daily or weekly per `history_rotation`, default `daily` — `defaults.py:60`). One line per history event, fields separated by `\t`. Search requires reading every file in the time range and substring-matching. The end-user docs put it plainly: "Text files are hard to beat in terms of performance when *writing* data … However, this is at the expense of read accesses. Since text files do not have an index, a complete read is necessary when searching in the files." (`ec.asciidoc:1746-1749`).
- **`sqlite`** (`history_sqlite.py`, 388 lines): WAL-mode SQLite at `var/mkeventd/history/history.sqlite`. PRAGMAs: `journal_mode=WAL`, `synchronous=NORMAL`, `busy_timeout=2000` (`history_sqlite.py:68-72`). Indexes on `time`, `id`, `host` (`history_sqlite.py:62-66, :74-76`). Periodic housekeeping deletes rows older than `history_lifetime` days, then `VACUUM`s when free-list crosses `sqlite_freelist_size` bytes (`history_sqlite.py:316-388`, defaults: `sqlite_housekeeping_interval=3600`s, `sqlite_freelist_size=50 MiB` from `defaults.py:56-57`).
- **`mongodb`** (`history_mongo.py`, 234 lines): per-event document with TTL index of `history_lifetime * 86400` seconds (`history_mongo.py:207-220`). Connection details from `mongodb_config_file` (`settings.py:89`).

### Database-backed config?

No equivalent in CheckMK. **All EC configuration is Python source files on disk**, executed via `exec(compile(file.read(), path, "exec"), global_context)` at start-up and on each reload (`rule_packs.py:105-122`). This is a deliberate choice that makes the config diff-friendly (git, WATO change tracking) but also limits what kind of mass-validation can happen at write time.

### Migration / upgrade

Schema migrations happen in-Python at config load time, not via a database migration tool. Examples in `rule_packs.py:_load_config`:
- Lines 124-130: livetime field upconverted from int to `(int, ["open"])` tuple.
- Lines 132-137: legacy bare rules upconverted into a single default rule pack.
- Lines 139-149: contact group format and service level format upgrades.
- Lines 152-163: log level format upgrade from pre-multi-component layout.
- Lines 172-173: `translate_snmptraps = True` → `(True, {})` (pre-1.6.0 → current).

The SQLite history backend has no schema versioning; the column set is hard-coded in `TABLE_COLUMNS` (`history_sqlite.py:29-60`) and a schema mismatch on upgrade would be a manual fix.

## 7. Configuration UX

CheckMK's EC has a *full* configuration surface in five locations:

### 1. `omd config` (site-level switch)

`omd/packages/check_mk/MKEVENTD` (the EC daemon itself) and `omd/packages/check_mk/MKEVENTD_SNMPTRAP` (the SNMP trap socket) are OMD hook scripts (`MKEVENTD:11-34`, `MKEVENTD_SNMPTRAP:12-23`). Operator runs:

```
omd stop
omd config set MKEVENTD_SNMPTRAP on
omd start
```

This re-generates the `mkeventd` init script invocation (with `--snmptrap --snmptrap-fd 5`) and toggles the SUID-helper path.

### 2. WATO web GUI — Setup → Event Console

`cmk/gui/mkeventd/wato.py` (5,479 lines). The GUI is the operator's primary surface.

- **Rule Packs**: `ModeEventConsoleRulePacks` (`wato.py:1744+`). Drag-and-drop ordering, expand/collapse, copy, export-as-MKP (CheckMK Extension Package).
- **Rules within a pack**: `ModeEventConsoleRules` (`wato.py:2258+`). One rule per row, with toggles for enable/disable, drop, drop-skip-pack, count, expect, livetime.
- **Edit a single rule**: `ModeEventConsoleEditRule` (`wato.py:2858+`). Includes all the matching fields (Match text, Match host, Match IP, Match application, Match priority, Match facility, Match time period, Match service level), and the outcome/action fields (state, set_text, set_host, set_application, contacts, actions list with `@NOTIFY`, count, expect, livetime, drop).
- **Status**: `ModeEventConsoleStatus` (`wato.py:3066+`). Shows daemon state, perf counters.

### Discoverability and defaults

The defaults at `packages/cmk-ec/cmk/ec/defaults.py:34-89`:
- `translate_snmptraps=False` — translation OFF.
- `snmp_credentials=[{description: '"public" default for receiving SNMPv1/v2 traps', credentials: "public"}]` — a single v2c credential with community `public` is pre-seeded.
- No SNMPv3 user is pre-seeded.

The consequence: out-of-the-box, an SNMPv2c `public`-community trap *will* be decoded and processed, but the rule list is **empty by default** (`packages/cmk-ec/cmk/ec/defaults.py:24-31` — `default_rule_pack(rules=[])` returns `ECRulePackSpec(id="default", title="Default rule pack", rules=[], disabled=False)`). So day-1 result is "trap is received, syslog facility 31 priority 5, application=<trap OID>, text=<varbinds>, but no rule matches, so the event is dropped" — *unless* the operator has set `archive_orphans=True` (`defaults.py:66` — default `False`). The end-user docs label this behaviour: action type `ARCHIVED` ("The event was automatically archived because no rule was applied and *Force message archiving* was enabled in the global settings", `ec.asciidoc:1498`).

### Live reload vs restart

- **Live reload** of WATO config: send `COMMAND RELOAD` via `unixcat tmp/run/mkeventd/status`. The reload path calls `EventServer.reload_configuration(config, history)` (`packages/cmk-ec/cmk/ec/main.py:1128-1141`), which:
  - reassigns `self._config = config` and `self._history = history`;
  - **re-constructs `self._snmp_trap_parser = SNMPTrapParser(self.settings, self._config, ...)` at `main.py:1130-1133`** — so SNMP credentials, engine IDs, and the `translate_snmptraps` setting from the freshly-read `global.mk` are picked up on reload;
  - re-loads MIBs via the new `SNMPTrapTranslator(...)` constructor — calling `_construct_resolver(...)` (`snmp.py:273-317`) — so newly-uploaded MIBs become effective on reload;
  - calls `compile_rules(...)` to rebuild the rule hash;
  - re-instantiates `HostConfig` and `RuleMatcher`.
- **Restart**: required only for changes that the in-process reload cannot pick up:
  - port enable/disable (the listener sockets are created once in `EventServer.__init__` at `main.py:421-426` and not re-created on reload).
  - the SUID helper / system-level `mkeventd_open514` privilege drop.
  - the Python code itself (package upgrade).
  - the bundled-MIB tree at `share/snmp/mibs/` (since these go through the same compiled-MIB resolver as user MIBs and are reloaded by `_construct_resolver`, in principle they too pick up on `COMMAND RELOAD`; however bundled MIBs only change during a package upgrade, which itself requires a restart).
- **MIB uploads via WATO**: the upload writes ASN.1 source to `local/share/snmp/mibs/` and compiled output to `local/share/check_mk/compiled_mibs/` (`cmk/gui/mkeventd/wato.py:3690-3712`). Becomes effective on the next WATO change activation, which issues `COMMAND RELOAD` to the EC daemon.

### Multi-tenancy / RBAC

- Permissions are managed via the standard CheckMK RBAC: the EC adds `mkeventd.config`, `mkeventd.edit`, `mkeventd.activate`, `mkeventd.delete`, `mkeventd.update`, `mkeventd.update_comment`, `mkeventd.update_contact`, `mkeventd.archive_events_of_hosts`, `mkeventd.actions`, `mkeventd.seeall`, `mkeventd.seeunrelated`, `mkeventd.changestate`, `mkeventd.switchmode`, `mkeventd.allowall` (the permission section file: `cmk/gui/mkeventd/permission_section.py`).
- In Managed Services Edition (CME), the `customer` field on rule packs and rules (`config.py:161, :197`) ties each pack/rule to a customer for visibility-isolation. This is GUI-only — runtime EC ignores `customer`.

## 8. Integration with Other Signals

### 8.1 Metrics

Traps are NOT converted to metrics in any first-class way. There is no "trap-to-counter" or "OID-to-graph" pipeline. The closest CheckMK has is the per-EC performance check (`cmk/plugins/checkmk/agent_based/mkeventd_status.py`): an *active check* on the monitoring core polls `mkeventd`'s `GET status` table and graphs received-messages-per-second, drops-per-second, rule_hits/sec, processing-time-per-message — see graph definitions at `cmk/plugins/checkmk/graphing/mkeventd_message_processing.py`, `mkeventd_rule_efficiency.py`, `mkeventd_standalone.py`. These are pipeline-health metrics, **not** trap content metrics.

### 8.2 Alerting / Notifications

Three paths:

1. **`@NOTIFY` action**: the rule lists `@NOTIFY` among its `actions`. When the event opens, `do_notify(...)` (`actions.py:325-440`) constructs a CheckMK notification context that *simulates* a service notification (`actions.py:333`: `We simulate a *service* notification.`), with the event's `host`, `application`, `text`, `state`, `id`, `phase` mapped into `HOSTNAME`, `SERVICEDESC`, `SERVICEOUTPUT`, `SERVICESTATEID`, etc., plus EC-specific `EC_ID`, `EC_CONTACT_GROUPS`, `EC_PHASE`. Then `cmk --notify` is invoked via `subprocess.run`, which dispatches through the standard notification rule engine — email, SMS, Slack, MS Teams, ServiceNow, OpsGenie, PagerDuty, JIRA, mobile push, etc. Per the EC docs, "the only fixed field is `NOTIFICATIONNUMBER=1` — escalation is not supported through the EC notification path" (`ec.asciidoc:1156`).
2. **`check_mkevents` active check**: a 398-line C++ binary (`active_checks/check_mkevents.cc`) plus a Python server-side-call wrapper (`cmk/plugins/checkmk/server_side_calls/mkevents.py:20-76`) implement a per-host (or per-host+application-regex) **active check** that queries the EC's status socket for open events on that host and converts the result to the standard Checkmk service state (OK / WARN / CRIT). The rule that creates this active check is "Check event state in Event Console" (`checkmk-docs/src/onprem/en/ec.asciidoc:1447-1456`). Operationally this means **every CheckMK-monitored host can have a service whose state mirrors the worst open EC event for that host** — and that service is then subject to *all* normal Checkmk alerting paths (notifications, BI aggregation, SLA reporting, dashboards). This is a distinct alerting pipeline from `@NOTIFY` and is the primary way EC events drive on-call notifications for organisations whose alerting is already wired through the standard CheckMK notification rules.
3. **Direct script/email action**: the rule lists `email` or `script` action(s). These are synchronous subprocess invocations with the full `CMK_*` env (`actions.py:189-205` `_do_script_action` → `actions.py:258-280` `_execute_script`). They bypass the standard notification rule routing — they fire whenever the rule does.

Acknowledgement and clear: operator clicks Acknowledge in the GUI → `UPDATE` command via the Unix socket → event's `phase` moves to `ack` → history `UPDATE` row written. Clearing happens either by `match_ok`-style cancelling rules (the "OK message clears the open event") or by livetime expiry (rule's `livetime` ages the event out automatically).

### 8.3 Topology

CheckMK has a *topology visualisation* (`cmk/gui/nodevis/topology.py` with permissions at `cmk/gui/default_permissions.py:313-320`; user-facing docs at `Checkmk/checkmk-docs :: src/common/en/hosts_structure.asciidoc:282-285`; no edition gate was identified in the reviewed evidence). Traps are **not** mapped onto the topology graph. `event.core_host` ties an event to a monitored host; from there the topology shows it as "this host has events". There is no L2/L3-aware suppression ("don't alert on downstream traps when the upstream is down"). Implementation-wise, this is verifiable: a grep for `topology` across `packages/cmk-ec/` returns zero matches.

### 8.4 Logs / Events

The EC IS the unified events store, but the schema is fixed (`packages/cmk-ec/cmk/ec/event.py:28-62` — `Event(TypedDict, total=False)` has ~33 fields, all flat, no nested varbind list). The "events" Livestatus table at the EC's Unix socket gives a searchable, sortable view; the "history" table gives the archive. The OpenAPI surface exposes facility 31 as the string `"snmptrap"` in JSON responses (`cmk/gui/mkeventd/_openapi/commands.py:92, :152`). Retention default is 365 days for sqlite/mongo or until the disk-space manager evicts (whichever is shorter, per `ec.asciidoc:1534-1537`). Search granularity is column-level filters (`Filter: event_text ~ <regex>`, `Filter: event_host = <name>`, …).

### 8.5 Northbound Forwarding

**No native outbound trap forwarding.** Verified by grep: no `northbound`, `send_trap`, `forward_trap`, `snmptrap.*send` references in `packages/cmk-ec/` or `cmk/gui/mkeventd/`. The `forwarder.py` `MessageForwarder` is *inbound* (Logwatch → EC, or central → remote site) — it ships syslog-style messages, not SNMP traps.

If an operator wants to forward processed events to an upstream NMS, they have three unsupported options:

1. **Community treasures script**: `doc/treasures/notifications/snmp_trap` (a 26-line bash script that builds an `snmptrap -v 1 -c <community>` invocation from the CheckMK notification env block, targeted at HP Service Center's MIB OIDs). This is a community-contributed example, not a maintained product feature.
2. Use a CheckMK Notification plugin that produces an `snmptrap` (Net-SNMP CLI) call, attached to a rule with `@NOTIFY`. The Notification → SNMP trap path exists generically through the CheckMK notification system, but no maintained `snmptrap` notification plugin is present in `packages/cmk-notification-plugins/notifications/`. (Note: there *is* an `mkeventd` notification plugin in that directory, but it serves the *opposite direction* — it forwards Checkmk service notifications *into* the EC via the `mkevent` CLI; not outbound to upstream SNMP.)
3. Use a rule's `script` action to invoke `snmptrap` (or any other CLI) on the event. This is what `doc/treasures/Event_Console/mail_mkevent.py` and `message_to_syslog.py` (`doc/treasures/Event_Console/`) do for email and syslog respectively.

This is a real product gap: **CheckMK EC is one-way for SNMP — it receives traps but does not emit them.** No SNMP MIB describing its own emitted traps is provided either (unlike OpenNMS, which ships `opennms.mib`).

## 9. Severity Model

There is **no vendor-supplied severity** in the SNMP trap protocol itself (v1/v2c/v3), so no decoder-side severity mapping exists. Severity is entirely a *rule outcome* in EC:

- The Event Console state has four levels: OK / WARN / CRIT / UNKNOWN (mapped to `0`/`1`/`2`/`3` in `event["state"]`).
- The rule sets the state via one of three mechanisms (`main.py:1475-1499`):
  - **Fixed**: `rule["state"] = 0|1|2|3` — assign that state for any match.
  - **Derived from priority**: `rule["state"] = -1` — `prio ≤ 3 → CRIT`; `prio == 4 → WARN`; `prio ≥ 5 → OK`. **Note: for traps, priority is hard-coded to 5**, so `state=-1` always gives OK. This mode is effectively useful only for syslog events, not traps.
  - **Text pattern**: `rule["state"] = ("text_pattern", {"2": pat, "1": pat, "0": pat})` — three regexes; first match wins; no match → UNKNOWN.

The operator's typical pattern for traps is "fixed state per rule": one rule per trap-OID (or trap-OID class), each rule pins the state to CRIT / WARN / OK based on operator judgment.

### Vendor severity is operator-supplied

If a vendor's trap carries a severity in a varbind (e.g., Cisco's `cefcModuleStatusReason` text or HP's `hpEvtLogTrap` numeric field), the operator must:
1. Upload the vendor MIB so the OID is named.
2. Write a rule with `match_application` matching the trap OID and `match` (regex on `text`) capturing the severity-bearing varbind value as a match group.
3. Use the `state = ("text_pattern", {…})` mode or write three separate rules (one per severity).

This is laborious. There is no built-in "severity translator" or vendor-pack equivalent. The end-user docs acknowledge the friction without offering a shortcut.

## 10. Storm / Volume Handling

### Per-source rate limits

**None at the listener layer.** The single-threaded `recvfrom(65535)` loop processes packets at whatever rate the rule engine + history writer can sustain.

### Dedup keys and windows

CheckMK does **not** implement per-trap dedup keys (no equivalent of OpenNMS's `reductionKey`). The closest is the **counting** mechanism (`main.py:1361-1402`):

- A rule with `"count"` set causes incoming matches to fold into an existing in-`counting`-phase event with matching `match_groups`. The fold compares matched group tuples (`main.py:1365-1370`).
- After N matches (`count.count` reached), the event transitions from `counting` to `open` and the rule's actions fire.
- After `count.algorithm["count_duration"]` seconds without reaching N, the event is auto-archived with `COUNTFAILED`.

This is volumetric aggregation ("alert me only if X happens 5 times in 10 minutes"), not dedup of duplicate traps. Two identical traps arriving in succession (same OID, same varbinds, same source) become **two open events** unless a rule explicitly counts them. The end-user docs warn about this and label it intentional: events are about state changes, not state.

### Absence detection via `expect`

A volumetric-adjacent feature: a rule with `"expect"` defines an *expected* arrival cadence ("an OK heartbeat from this device should arrive every N seconds"). The implementation walks active expect-rules periodically (`packages/cmk-ec/cmk/ec/main.py:967-1030`). For each rule, it keeps a counting-phase event tracking arrivals in the current interval. If the count fails to reach the expected threshold within the interval, `_handle_absent_event(...)` (`main.py:1038-1113`) fires:
- For a **newly opened** absent event: history records `NEW` via `EventStatus.new_event()` (`main.py:2782-2789`) followed by `COUNTFAILED` (`main.py:1112`).
- For a **merged existing** absent event (a continuation of the same expect-rule's missing-message condition): history records only `COUNTFAILED`.

This is the trap-monitoring counterpart of "tell me when this device *stops* sending traps" — useful for keepalive-trap and link-up/down patterns where silence is the failure signal. It is implemented as scheduled work, not packet-driven, so it does not add per-trap overhead.

### Storm detection / circuit breakers

`event_limit` (`config.py:60-69`, `defaults.py:75-87`):

- `by_host: limit=1000, action=stop_overflow_notify`
- `by_rule: limit=1000, action=stop_overflow_notify`
- `overall: limit=10000, action=stop_overflow_notify`

`stop_overflow_notify` means: when the limit is reached, stop creating new events for this host/rule/overall, create a single **synthetic "overflow event"**, and notify the configured contacts via the CheckMK notification system. The implementation is in `EventServer.new_event_respecting_limits` (`main.py:1578-1696`).

Three other action variants:
- `stop`: just stop new events; no notification.
- `stop_overflow`: stop + create overflow event but don't notify.
- `delete_oldest`: drop the oldest existing event to make room for the new one. Useful for high-churn environments where seeing the *latest* problems matters more than preserving the oldest.

The docs put this in context: "This will protect you in situations where, due to a systematic problem on your network, thousands of critical events are flooding in and the Event Console would 'shut down'." (`ec.asciidoc:1707`).

### Backpressure / queue management

There is no application-level queue. Kernel UDP buffer is the only buffer. If kernel buffer fills, packets are dropped silently by the kernel. CheckMK has no `netstat -su`-derived metric exposed in the EC perfcounters.

The drop counter `_perfcounters.count("drops")` (`main.py:1313`) increments only on rule-decision drops (`rule.get("drop") == True`), not on kernel buffer drops. This is a real visibility gap: an operator under a trap storm may see "drops/sec" stay at 0 and still be losing traps at the kernel.

### Listener-layer protection gap

`event_limit` is a **post-processing gate** — it limits the number of *open* events but does not rate-limit the listener. Under a trap storm, the EC thread spends CPU on *every* received packet:

1. `recvfrom(65535)` on the SNMP trap socket (`main.py:756`).
2. `SNMPTrapParser.parse(...)` — full BER decode + USM authentication via PySNMP (`snmp.py:170-185`).
3. `create_event_from_trap(...)` — varbind formatting (`main.py:1778-1794`).
4. `process_potential_event(...)` — rule matching against all candidate rules (`main.py:1266-1473`).
5. *Only then* does `new_event_respecting_limits(...)` (`main.py:1578-1696`) check the per-host / per-rule / overall caps and apply the overflow action.

So a million-trap storm hitting the UDP socket consumes CPU all the way through decode + rule-match for every packet, even when `event_limit.action="stop"` would discard the result. The pre-filter line of defence is the kernel UDP buffer (`SO_RCVBUF`); the only mitigation an operator has against sustained storms is OS-level firewall rate-limiting (e.g., `iptables -m hashlimit`), not anything inside CheckMK.

## 11. Security

### SNMPv3 USM

Configured per credential entry in `snmp_credentials` (`config.py:296-316`, GUI at `wato.py:4603-4647`):

- Auth: MD5, SHA, SHA-224, SHA-256, SHA-384, SHA-512.
- Privacy: DES, AES, 3DES-EDE, AES-192, AES-256, AES-192-Blumenthal, AES-256-Blumenthal.
- Engine IDs configured as hex strings (validated by `regex="^[A-Fa-f0-9]*$"`, `wato.py:4623`).

Credential records also support `noAuthNoPriv` and `authNoPriv` modes (`snmp.py:131-150`), so the full RFC 3414 USM matrix is supported (auth in {none, MD5/SHA}, priv in {none, DES/AES}).

### DTLS / TLSTM

Not implemented (see §3).

### Credential storage

SNMP credentials live in `etc/check_mk/mkeventd.d/wato/global.mk` as plaintext Python literals (`snmp_credentials = [{"description": "...", "credentials": "public", "engine_ids": [...]}]`). The OMD site filesystem permissions (Apache user owns its site, mode 0700 on the site root) are the protection. There is no built-in secret-encryption-at-rest for the credentials; an operator with site-filesystem read access has the SNMP credentials.

This is a real and signalled tradeoff in CheckMK's wider design — most secrets across the site (notification plugin tokens, agent secrets, etc.) follow the same pattern.

### Access control on the trap subsystem itself

- UDP 162 is open to *the world* once the SUID helper binds it. There is no source-IP allow-list at the EC layer. The operator must use OS-level firewall rules (iptables, nftables, ufw) to restrict.
- The end-user docs explicitly call this out: "After all, the syslog and SNMP ports are open to all hosts on the network." (`ec.asciidoc:1104`).
- The status socket (`tmp/run/mkeventd/status`) is mode 0660 owned by the OMD site user; remote TCP access via the optional `remote_status` config is documented as the way to allow trusted GUIs to talk to a remote EC.

### Audit logging

- WATO changes (rule pack edits, MIB uploads, settings) are tracked through the standard WATO change log (audit trail with user, timestamp, diff).
- EC operator actions on events (acknowledge, change state, delete, custom action) go to the history backend as `UPDATE`, `CHANGESTATE`, `DELETE`, `SCRIPT`, `EMAIL` rows with a `who` field.

For diagnosing rule-match issues without source-level instrumentation, three booleans in `defaults.py` control verbose output:

- `debug_rules` (default `False`, `defaults.py:40`) — every match attempt is traced into `mkeventd.log` showing which rule was tried, which fields matched, which did not match, and the reason. The docs warn this generates an "immense volume of logging and should never be used in productive operation" (`cmk/gui/mkeventd/wato.py:4659-4664`).
- `log_rulehits` (default `False`, `defaults.py:52`) — every successful rule match is logged with rule pack, rule id, facility, priority, message text.
- `log_messages` (default `False`, `defaults.py:53`) — every incoming message is written to a syslog-like text file before rule processing.

The first two are the primary operator-facing trap troubleshooting tools. Toggling `log_rulehits` for a few minutes during commissioning of a new trap type tells the operator immediately whether their rule fired.

## 14. User Customization Surface

### Custom rules

Three nested levels of control:

1. **Rule packs**: a named collection of rules with a shared `id`, `title`, `disabled` flag. The pack's rules are evaluated in order until one matches (or a `drop:"skip_pack"` rule fires). Drag-and-drop reordering of packs from the GUI.
2. **Rules within a pack**: each rule has all the matching fields (text, host, IP CIDR, facility, priority, application, service level, time period, site) and outcome fields (state, set_text, set_host, set_application, set_contact, contact_groups, actions, count, expect, livetime, drop, drop=skip_pack, invert_matching).
3. **Match-group substitution**: capture groups in regex matches can be referenced via `\1`, `\2` in `set_*` fields (`packages/cmk-ec/cmk/ec/main.py:318-340` `replace_groups`).

### Custom OID handlers

There is no plugin model for "custom OID parsers." The PySNMP-based decoder always returns `(oid_or_name, formatted_value)` pairs; the operator's customisation is at the rule layer, not the decoder layer. To extract a numeric severity from a varbind, the operator writes a regex with a capture group on `event.text`.

### Custom MIBs

Already covered in §4: upload ASN.1 source via WATO; PySMI compiles to Python; PySNMP loads on next start.

### Custom severity rules

No first-class "severity mapping" surface. Severity is per-rule (see §9).

### Custom dedup

No first-class dedup; the operator uses `count` with a regex that captures the dedup key into `match_groups` (per-event match-group identity determines the fold target in `count_event`).

### Plugin / extension model

- **Notification plugins** (`packages/cmk-notification-plugins/`): generic CheckMK notification scripts, available to EC events via `@NOTIFY`. Operator can author new ones in any language; they are CLI scripts that receive a `NOTIFY_*` env block.
- **MKP (CheckMK Extension Package)**: a tarball-based packaging mechanism that can ship rule packs (`packages/cmk-ec/cmk/ec/mkp.py:1-69`). The `MkpRulePackProxy` (`config.py:204-272`) lets an installed MKP override or augment a rule pack. Operators / vendors publish MKPs on the CheckMK exchange.
- **Rule actions** (within EC): `email` and `script` are the two action types (`config.py:60-92` — `Action`, `EMailActionConfig`, `ScriptActionConfig`). Scripts run as `/bin/bash` with the event fields injected as `CMK_*` env vars; arbitrary shell.

### API surface for automation

REST / OpenAPI for **events only** — two endpoint families under `cmk/gui/mkeventd/_openapi/`: `current_events/` (list, show, archive, change_state, change_multiple_states, update_and_acknowledge, update_and_acknowledge_multiple) and `historical_events/` (list, show). There is **no REST endpoint for rule CRUD, MIB upload, or SNMP credentials management**; those go through the WATO change-activation pipeline only. Notification rule administration is a separate Checkmk API surface (outside the EC namespace) and is not part of the EC REST. For one-shot event injection, the named pipe `tmp/run/mkeventd/events` is reachable via any process that can write to it.

## 15. End-User Value Analysis

### What an operator gets day-1 with default config

After enabling `MKEVENTD_SNMPTRAP`:
- An UDP/162 listener that accepts v1, v2c (community `public`), and v3 (no users pre-configured) PDUs.
- A pipeline that decodes each trap, untranslated, and creates an `Event` per trap with `application = <trap OID>` (numeric) and `text = "<oid>: <value>, <oid>: <value>, ..."`.
- A rule list that is **empty**, so by default `archive_orphans=False` means **unmatched traps are silently dropped** — no archived row, no event in the GUI.

To see a single trap as an event in the GUI on day-1, the operator must either (a) set `archive_orphans=True`, or (b) write a catch-all rule.

### What requires customisation

- MIB upload (per-vendor) for human-readable OID names.
- Per-trap rules to define severity and any rewriting.
- Notification routing (via `@NOTIFY` + standard CheckMK notification rules).
- Counting / suppression rules (`count`, `livetime`) for spammy traps.
- Operator host-mapping enrichment via the monitoring core (i.e., add the trap-emitting device as a monitored CheckMK host so `core_host` resolves).

### Learning curve

Medium. The rule editor in WATO is dense (all matching fields, all outcome fields, all action types in one form) but consistent across syslog/pipe/trap events. The PySNMP/PySMI MIB workflow is straightforward but iteratively painful — operators upload MIBs, hit "missing dependency", chase the dependency, upload that, repeat.

### Operational toil

- **MIB curation**: substantial manual effort per vendor (multi-hour effort common; days possible for vendors with deep dependency trees). There is no shared community catalogue.
- **Rule curation**: one rule per trap OID class. Cisco's catalog alone has hundreds of trap types; mapping each to a severity is a long-running maintenance task.
- **Trap-storm response**: the `event_limit` protects from RAM exhaustion but requires post-mortem cleanup (operator deletes archive files manually if archive grows out of hand — `ec.asciidoc:1672-1675`).
- **Cross-site coordination** (Distributed EC): retargeting device trap destinations to remote sites is "the most laborious task" (`distributed_monitoring.asciidoc:668-669`).

### Visibility into the pipeline's own health

Surfaced metrics and log streams (verified):
- `mkeventd_status` active check graphs received-messages-per-second, drops-per-second, rule_hits/sec, rule_tries/sec, rule_hit_ratio, processing-time-per-message (`cmk/plugins/checkmk/graphing/mkeventd_*.py`).
- WATO Status page (`ModeEventConsoleStatus`) shows event-limit-active-{hosts,rules,overall}, open-events count, virtual memory size, replication slave mode.
- `~/var/log/mkeventd.log` logs trap arrival, acceptance/drop reason, and the rule-optimizer hash distribution at every reload (`ec.asciidoc:1754-1781`).

What is **not** visible:
- Kernel UDP buffer drops (no counter).
- Per-source rate at the listener (no histogram).
- Per-rule processing-time breakdown (only aggregate `processing` time).
- Trap PDUs that failed authentication: VERBOSE-only log; no counter.

## 16. Strengths

1. **Architectural simplicity through unification**: traps, syslog, named-pipe events, and Unix-socket events share one record (`Event`), one rule engine, one history store, one GUI. The operator learns one mental model and applies it to four signal classes (`packages/cmk-ec/cmk/ec/main.py:1266-1473`, `rule_matcher.py:216-228`).
2. **In-process PySNMP / PySMI**: no `snmptrapd` to manage, no UDP-forward gymnastics. The operator just toggles `MKEVENTD_SNMPTRAP=on`. The privileged-port helper (`bin/mkeventd_open514.cc`) is small (162 LoC), single-purpose, and **uses Linux capabilities by default** (`cap_net_bind_service+ep`) — SUID-root only as a packaging fallback. See `omd/omd.spec.in:71` and `omd/debian/postinst.in:76-87`.
3. **Rule-optimizer hash by `(facility, priority)`**: precomputed bucketing reduces per-event rule evaluations when many rules carry explicit `match_facility` (`packages/cmk-ec/cmk/ec/main.py:1142-1244`). The post-restart log dump shows per-bucket rule counts — operator-facing diagnostic that exposes the optimizer's view of the rule base.
4. **Three pluggable history backends** (file / sqlite / mongodb) with explicit performance characterisation in the end-user docs (`ec.asciidoc:1528, :1746-1749`). Default is sqlite in WAL mode with indexes on `time`/`id`/`host` (`history_sqlite.py:62-72`).
5. **Cancelling/clear semantics in the rule grammar**: `match_ok`, `cancel_application`, `cancel_priority` (`rule_matcher.py:222-282`) let the operator express "this OK message closes open events whose text matches X" without writing imperative cleanup code. Useful pattern for power-cycle traps.
6. **Counting + expect** semantics (volumetric + absence detection): a rule with `count` aggregates incoming matches; a rule with `expect` triggers when an *expected* message fails to arrive in an interval (`main.py:976-1030`). This covers both noisy and silent-failure patterns.
7. **Candid documentation**: the end-user manual (`checkmk-docs/src/onprem/en/ec.asciidoc`, 1,854 lines) discloses scale limits ("As soon as the number of current events exceeds about 5 000, the performance begins to deteriorate noticeably", line 1667), MIB coverage gaps ("not able to deliver vendor specific MIB files", line 1357), and the trap-storm risk ("the receiver has no chance of slowing them down", line 1567). No marketing language.
8. **CheckMK notifications as a first-class action target**: `@NOTIFY` (`actions.py:43-50, :325-440`) routes EC events through the same notification system used for service alerts, picking up the operator's existing escalation/contact/quiet-time rules.
9. **MKP packaging**: rule packs can be exported as Checkmk Extension Packages (`packages/cmk-ec/cmk/ec/mkp.py`, `rule_packs.py:37-77`) and shared via the CheckMK exchange. Operators / vendors can ship their curated trap rule sets.
10. **Hostname mapping by IP / address / alias**: `HostConfig` (`host_config.py:1-82`) maps trap-sender IP to a monitored host using *any* of the host's configured addresses or aliases (`host_config.py:76-80`), with cache invalidation tied to the monitoring core's `status_program_start` timestamp (`host_config.py:52-66`).

## 17. Weaknesses / Gaps

1. **No SNMPv3 INFORM, no SNMPv2c INFORM** (`packages/cmk-ec/cmk/ec/snmp.py:48-53`). The code comment is explicit: `# Disable receiving of SNMPv3 INFORM messages. We do not support them (yet)`. Devices that emit INFORM with retry-on-no-ack will produce no EC event. The exact PySNMP handling of an unsupported-PDU-type at the receiver layer (whether it triggers the `_handle_unauthenticated_snmptrap` observer, drops silently inside PySNMP's MessageDispatcher, or returns a SNMPv2c "wrongValue" response) is not source-verified in this pass — verifiable only with a live trap-source test, which is out of scope here.
2. **No DTLS / TLSTM**. The UDP socket is plaintext SNMPv1/v2c/v3 only. For shops modernising onto RFC 5953/6353 transports, CheckMK has nothing to offer.
3. **Single-threaded reception**. The whole pipeline runs in one `EventServer` thread. Scaling beyond a few hundred traps/sec is design-limited. The end-user docs acknowledge the 5,000-open-events ceiling for GUI/processing health; the kernel UDP buffer is the only line of defence against bursts.
4. **No vendor MIBs / no bundled rules / no vendor packs OOTB**. CheckMK ships zero vendor knowledge for SNMP traps. Operators build the knowledge base themselves. Compare to OpenNMS's hundreds of bundled event-definition XML files; CheckMK is a blank canvas. The end-user docs disclose this gap directly (`ec.asciidoc:1357`). For the exact OpenNMS corpus counts see `.agents/sow/specs/snmp-traps/opennms.md`.
5. **SNMPv3 engine-IDs are effectively required but the GUI allows empty lists** (`packages/cmk-ec/cmk/ec/snmp.py:152-168`, `cmk/gui/mkeventd/wato.py:4615-4636`). A WATO operator can save an SNMPv3 credential entry with an empty `engine_ids` list (the field is `optional_keys=["engine_ids"]`); at runtime, no v3 user is registered for that entry, so matching SNMPv3 traps will fail authentication and be silently dropped (only logged at VERBOSE). This is a real footgun — the GUI should treat `engine_ids` as required for v3 entries or warn at save time.
6. **Trap-storm mitigation is at the event-count layer, not at the listener layer**. The `event_limit` (`defaults.py:75-87`) caps *open events* but does not rate-limit the listener. Under a sustained storm, every UDP packet is `recvfrom`-ed, BER-decoded by PySNMP, USM-authenticated (if v3), formatted into varbinds, and rule-matched **before** `new_event_respecting_limits(...)` (`main.py:1578-1696`) gets to discard it. This means trap storms burn EC CPU even when the limit-action is `stop`. There is no `iptables hashlimit`-style pre-filter built into EC; the docs recommend OS-level firewall hardening instead.
7. **Kernel UDP buffer drops are invisible**. `_perfcounters.count("drops")` (`main.py:1313`) increments only on rule-decision drops. The kernel's `netstat -su` `RcvbufErrors` is not surfaced; operators do not know when packets are being lost upstream of EC.
8. **No northbound forwarding**. CheckMK can receive traps but does not emit them (no built-in trap-forward action; no SNMP-trap notification plugin in `packages/cmk-notification-plugins/notifications/`). Operators wanting to feed an upstream NMS must script `snmptrap` calls from rule `script` actions.
9. **No raw trap-bytes archive**. The decoded `Event` keeps the formatted OID/varbind text in `application` and `text` but discards the original PDU bytes. A future re-translation after uploading a missing MIB is impossible — the original traps cannot be replayed against the new MIB knowledge.
10. **Severity is fully operator-defined**. There is no built-in mapping of well-known vendor severity varbinds (Cisco `cefcModuleStatusReason`, HP `hpEvtLogTrap`, NetApp ONTAP alert severity) to the EC's OK/WARN/CRIT/UNKNOWN. Every operator re-derives this. By contrast, OpenNMS bundles thousands of pre-derived severity mappings.
11. **No topology-aware suppression**. Trap from a downstream switch doesn't get squashed by an upstream-router-down trap. The operator can simulate this via `cancel_application` / `match_ok` patterns, but it is manual.
12. **`translate_snmptraps` add_description mode is "more detailed -- and more confusing"** (`ec.asciidoc:1349-1350`). The MIB description text is appended verbatim to the value with `({description})`, often producing very long, hard-to-read event lines. Many operators turn it off after trying it.
13. **Plain-text storage of SNMP credentials in `global.mk`**. No encryption at rest. Filesystem permissions are the only protection. In multi-admin sites this is a real exposure.
14. **Cloud SaaS edition has no EC**. CheckMK Cloud (CSE) explicitly disables `mkeventd` (`cmk/gui/general_config.py:578`, `omd/packages/check_mk/MKEVENTD:13-15`). For customers moving from on-prem to CSE, the EC feature must be migrated to *something else* (rule-based notification or external SIEM); no migration path is documented in source.
15. **Day-1 emptiness is a UX cliff**. After enabling traps, an operator sees nothing in the GUI until they (a) write a rule or (b) toggle `archive_orphans=True`. There is no nudge UI ("you've received 14 unmatched traps — want to see them?"). Net-SNMP's `snmptrap` test command in the docs is the standard way to verify the daemon is alive, not the GUI.
16. **No native time-window-based dedup**. Two identical traps within seconds produce two events unless a rule has `count`. This is the OpenNMS-`reductionKey` gap.
17. **No SNMP unit tests, and integration tests only exercise SNMPv1**. The `packages/cmk-ec/tests/` suite has 20 files covering rule matching, history backends, syslog parsing, perfcounters, forwarder, host_config — but **no `test_ec_snmp.py`** and no PySNMP parser tests. The only SNMP-pipeline coverage is the integration test at `tests/integration/cmk/ec/test_ec.py:353-402`, which shells out to the Net-SNMP `snmptrap` CLI **with `-v 1 -c public`** (SNMPv1 only — see `_get_snmp_trap_cmd` at `test_ec.py:194-209`). SNMPv2c and SNMPv3 trap reception are not exercised by any test. Negative-path coverage — malformed BER, oversized varbinds, BER fuzz, malformed engineId, mis-encoded varbinds — does not exist. This is a coverage gap in a security-relevant decoder path.

## 18. Notable Code or Configuration Examples

### 18.1 The whole trap-to-event normalisation in one function

`packages/cmk-ec/cmk/ec/main.py:1778-1794` — `create_event_from_trap`:

```
def create_event_from_trap(trap, ipaddress_):
    """New event with the trap OID as the application."""
    trapOIDs, other = partition(
        lambda binding: binding[0] in {"1.3.6.1.6.3.1.1.4.1.0", "SNMPv2-MIB::snmpTrapOID.0"}, trap
    )
    return Event(
        time=time.time(),
        host=HostAddress(scrub_string(ipaddress_)),
        ipaddress=scrub_string(ipaddress_),
        priority=5,    # notice
        facility=31,   # not used by syslog -> we use this for all traps
        application=scrub_string(trapOIDs[0][1] if trapOIDs else ""),
        text=scrub_string(", ".join(f"{oid}: {value}" for oid, value in other)),
        core_host=None,
        host_in_downtime=False,
    )
```

This is the **entire** trap-to-internal normalisation. Sixteen lines. The design choice — fold a trap into a syslog-shaped record by stashing the trap OID into `application` and the varbinds into `text` — defines what the rule grammar can express. It also defines the limits: any cross-varbind reasoning ("if varbind A equals X and varbind B is greater than Y") becomes a single regex over the joined `text`, not a structured query.

```
        f'"{event_message}"',
    ]
```

Tests don't construct raw PDU bytes — they shell out to Net-SNMP's `snmptrap` (which is bundled in the OMD site). This is pragmatic: round-trip through a real SNMP client and server, on a real UDP socket, exercises the *whole* path including PySNMP's BER decoder. The downside: negative-path tests (malformed BER, oversized varbinds, BER fuzz) are not in scope. Trap-fuzzing of the parser is not part of the CI gate.
