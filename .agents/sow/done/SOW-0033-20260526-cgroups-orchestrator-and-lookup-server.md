Sub-state (implementation-correction 2026-05-27): R13D-6's "send partial response on lookup overflow" resolution was rejected during implementation because it contradicts the vendored netipc lookup contract. Evidence: the C dispatcher rejects `builder.item_count != request.item_count` with `NIPC_ERR_BAD_ITEM_COUNT` (`src/libnetdata/netipc/src/protocol/netipc_protocol.c:2328-2329`), the Rust dispatcher does the same (`src/crates/netipc/src/protocol/lookup.rs:1423-1424`), the Go dispatcher has the same exact-count rule (`src/go/pkg/netipc/protocol/lookup.go`, `DispatchCgroupsLookup`), and the C client also rejects response item-count mismatch (`src/libnetdata/netipc/src/service/netipc_service.c:757-758`). Implementation decision: preserve the exact-count lookup contract. On `NIPC_ERR_OVERFLOW`, the lookup handler records a request error, logs once per `cgroup_discovery_generation`, and returns the builder error; it does NOT send a partial lookup response. This is a SOW correction to align with the primary source code, not a wire-format redesign.

## Requirements

### Purpose

Make `cgroups.plugin` the authoritative source of cgroup → orchestrator/name/labels metadata that other plugins can query over netipc. Two changes:

1. **Broaden the structured orchestrator detection** in cgroups.plugin from the current 3-value enum (UNSET/UNKNOWN/K8S) to match the netipc wire enum (UNKNOWN/SYSTEMD/DOCKER/K8S/KVM/LXC/PODMAN/NSPAWN). 2. **Add a CGROUPS_LOOKUP netipc server thread** to cgroups.plugin alongside the existing CGROUPS_SNAPSHOT server. The new server answers per-path queries returning the cgroup's friendly name, structured orchestrator, and labels.

This SOW is the foundation. After it ships, no consumer exists yet — but the server is exercised end-to-end via a small test client in validation.

**Server lifecycle and request handling:**

- A new netipc service named `"cgroups-lookup"` is registered, listening on its own UDS socket. The existing `"cgroups-snapshot"` service is unchanged in protocol; only its generation source changes (per above).
- The CGROUPS_LOOKUP handler returns:
  - `KNOWN` for cgroups present in `cgroup_root` AND with `processed == 1` AND `pending_renames == 0`.
  - `UNKNOWN_RETRY_LATER` for: (a) cgroups present in `cgroup_root` with `processed == 0` or `pending_renames > 0`, (b) ALL queries received before the first `discovery_find_all_cgroups()` completion (detected via `cgroup_discovery_generation == 0`), and (c) cgroups not in `cgroup_root` and not in the reaped-set (see next bullet).
  - `UNKNOWN_PERMANENT` for cgroups whose path is in the bounded reaped-set.
- A bounded reaped-set is added (size cap configurable, default 4096 entries, LRU eviction on full): when discovery removes a cgroup (i.e., the corresponding `/sys/fs/cgroup` path is gone), the path is added to this set. Concretely, the insertion happens inside `discovery_cleanup_all_cgroups()` (`cgroup-discovery.c:731-768`), immediately before each `cgroup_free(cg)` call at line 756, so that `cg->id` is captured before the cgroup is freed. **String ownership**: the reaped-set DEEP-COPIES `cg->id` via `strdupz()` before `cgroup_free(cg)` is invoked. The set OWNS each duplicated string and releases it via `freez()` on LRU eviction or on set destruction at plugin shutdown. The lookup handler reads the set under `cgroup_root_mutex` for membership checks ONLY — it does NOT take ownership of, free, or retain pointers into the set's string storage past the mutex release (it only needs the boolean "in set" result to choose `UNKNOWN_PERMANENT` vs `UNKNOWN_RETRY_LATER`). This eliminates any use-after-free risk between the writer's free at `cgroup_free(cg)` and the reader's membership probe. The set is shared between discovery (writer) and the lookup handler (reader) under `cgroup_root_mutex` — `discovery_cleanup_all_cgroups()` already runs inside the lock window at lines 1133-1141, so no additional locking is required for the writer side; the reader (lookup handler) MUST hold `cgroup_root_mutex` while consulting the set. **Shutdown cleanup**: the reaped-set MUST be destroyed in `cgroup_netipc_lookup_cleanup()` — every retained `strdupz`'d path string is `freez()`'d and the set's backing storage (hash table or whichever container is chosen) is released — to prevent a memory leak at plugin shutdown. Without this, ASan and valgrind runs against a graceful-shutdown harness report 4096-entry leaks (`4096 × strlen(cg->id)` bytes plus container overhead).

**Labels:**

- All labels from `cg->chart_labels` are serialised into the response per the netipc label table format using `rrdlabels_walkthrough_read`. Order is **stable for an unchanged label set** across responses — the walkthrough iterates the JudyL index via `JudyLFirstThenNext` (`src/database/rrdlabels.c:51-58`), which orders by the integer Index value (the label name's encoded key), NOT by insertion order (per CDX-NB-1). The order is deterministic per JudyL iteration semantics and therefore reproducible across repeated queries on the same label set, but it is NOT the order in which labels were added. Validation: a cgroup with K8s labels yields the labels in the SAME order across repeated queries; the order is not asserted to be insertion-order.

**Mutex discipline + label memory ownership:**

- The handler acquires `cgroup_root_mutex`, looks up the matching cgroup by path using a **linear scan** of the `cgroup_root` linked list (searches by `cg->id` until a match is found or the list is exhausted, matching the iterator discipline of the snapshot server at `cgroup-netipc.c:56` but with search-and-stop semantics instead of iterate-all — the snapshot handler walks every cgroup to build a complete snapshot, whereas the lookup handler stops at the first id match per request; see Decision L12 and Risk register), and performs ALL data extraction under the lock:
  1. Copies scalar fields (`container_orchestrator`, plus the response's `generation` value loaded via `__atomic_load_n(&cgroup_discovery_generation, __ATOMIC_ACQUIRE)`) into local variables.
  2. **Deep-copies strings**: allocates new string memory and `memcpy`s the contents of `cg->id` (path), `cg->name` into local buffers (using `mallocz`/`strdupz`). The handler does NOT retain pointers into cgroup-owned memory.
  3. **Deep-copies labels**: walks `cg->chart_labels` via `rrdlabels_walkthrough_read`; for each label, allocates a `nipc_lookup_label_view_t` entry in a local `growable_array` with deep-copied key+value strings. The handler does NOT retain `RRDLABELS` internal pointers.
- After the deep-copy, the handler releases `cgroup_root_mutex`.
- The handler then calls `nipc_cgroups_lookup_builder_add` with the local copies. The builder copies the data into the wire response buffer (per netipc design).
- **Buffer-overflow handling (corrected from R13D-6 on 2026-05-27)**: the handler MUST check the return code on EVERY `nipc_cgroups_lookup_builder_add()` call. On `NIPC_ERR_OVERFLOW` (wire response buffer full), the handler MUST leave the builder error intact, return handler failure so the dispatcher returns the protocol error, record a request error, and log at `collector_error()` level ONCE PER `cgroup_discovery_generation` value (not once per request). It MUST NOT send a partial lookup response because the vendored netipc lookup contract requires `builder.item_count == request.item_count` in C, Rust, Go, and the C client. Without this guard, a single buffer-full event would either be silently flattened into a generic handler failure or spam logs. Dedupe per generation keeps the signal informative while bounded.
- After `builder_add` returns, the handler frees ALL locally-allocated string memory and the label-array memory.
- **Label ownership rule**: `cg->chart_labels` is owned by the cgroup and may be freed/mutated by the discovery thread after the handler unlocks the mutex. The handler MUST NOT retain any pointer into RRDLABELS-owned memory across the unlock.
- **Defensive NULL-guard on `cg->chart_labels`**: the handler MUST check `cg->chart_labels != NULL` before calling `rrdlabels_walkthrough_read`. If NULL, the handler emits zero labels rather than relying on library-internal behaviour. Note: `rrdlabels_walkthrough_read` IS NULL-safe today (`src/database/rrdlabels.c:591` returns 0 when `!labels`), but relying on a library-internal NULL-guard at an IPC boundary is fragile — a future tightening of the contract or an inlined fast-path could remove it. The explicit caller-side guard makes the IPC handler's NULL behaviour independent of libnetdata internals. **`chart_labels` is NOT guaranteed non-NULL for every cgroup that reaches the handler (per CDX-NB-2)**: `processed = 1` is set at `cgroup-discovery.c:1057` BEFORE the early-return branches at `:1061` (RRD_ID_LENGTH_MAX), `:1068` (systemd duplicate), `:1078` (`CGROUP_OPTIONS_DISABLED_EXCLUDED`), `:1083-1084` (`!matches_enabled_cgroup_names`), `:1088-1089` (`!matches_enabled_cgroup_paths`), and `:1094-1095` (non-systemd duplicate); the label-creation calls are downstream at `:1071` (systemd branch) and `:1098-1099` (non-systemd branch). Disabled/excluded cgroups CAN therefore reach the lookup handler with `processed==1 && pending_renames==0` AND `chart_labels==NULL`. The defensive NULL-guard makes the handler safe in that case — it emits zero labels for the cgroup and the response remains well-formed. Zero-cost runtime guard; non-negotiable in the implementation.
- **Rename-race mitigation**: `cgroup_root_mutex` does NOT synchronize against `discovery_rename_cgroup` (`cgroup-discovery.c:226-278`) or `cgroup_parse_resolved_name_and_labels` (`cgroup-discovery.c:171-224`), both of which mutate `cg->name`, `cg->chart_id`, and `cg->chart_labels` WITHOUT holding `cgroup_root_mutex`. Those functions run inside `discovery_process_cgroup` at `cgroup-discovery.c:1124-1127`, which executes BEFORE the mutex is acquired at line 1133. The handler mitigates this race by only deep-copying mutable fields (`name`, `chart_id`, `chart_labels`) for cgroups that satisfy `processed == 1 && pending_renames == 0` — at which point discovery has finished mutating them for this discovery cycle. For `UNKNOWN_RETRY_LATER` and `UNKNOWN_PERMANENT` responses the handler MUST NOT read `cg->name`/`cg->chart_id`/`cg->chart_labels`; only the path key supplied by the requester and the status code are returned.
- **Immutable field invariant**: `cg->id` (the filesystem path) is set once in `discovery_cgroup_add` and is never mutated afterward — `discovery_rename_cgroup` rewrites `cg->name` and `cg->chart_id` but never `cg->id`. Orchestrator classification is therefore based on the stable `cg->id` and cannot be invalidated by the rename script.
- No long-held locks. No nested locking. The handler MUST NOT call any libnetdata function that may itself acquire `cgroup_root_mutex` while holding it.

## Analysis

Sources checked: see Master Plan SOW-0032.

Current state:

- `cgroups.plugin` already runs a netipc server (CGROUPS_SNAPSHOT). The new lookup server is a sibling, not a replacement.
- The orchestrator enum is barely used today (one detection site, no consumers). Broadening it is low-risk.

Risks:

- **Generation counter refactor breaks existing CGROUPS_SNAPSHOT consumer (`ebpf.plugin`)**: the existing per-request generation is replaced with a discovery-driven one. The wire format is unchanged (still a uint64); the semantic meaning changes from "monotonic across snapshot requests" to "monotonic across discovery cycles". Mitigation: ebpf.plugin already treats the generation as opaque per-snapshot version data, not as a per-request token; verify by inspecting ebpf.plugin's snapshot consumer code before implementation.
- Contention on `cgroup_root_mutex`: the lookup handler will compete with the discovery thread for the mutex. Discovery iterations can be long. Mitigation: per Contract 9, the handler holds the mutex only for its read window. Worst case: a discovery iteration delays lookup responses by the discovery duration. Acceptable per D6. **Three-way contention note (R11D-5)**: `cgroup_root_mutex` actually has THREE contenders, not two — the discovery thread, the lookup handler (new), AND `cgroups_main` (the collection thread) at `sys_fs_cgroup.c:1461-1482`, which holds the mutex for the entire read-all + chart-update window per collection cycle. On large hosts (>1000 cgroups) `cgroups_main`'s critical section can be the dominant holder. Adding the lookup handler as a fourth caller could in theory delay collection cadence under burst load. Mitigation: the lookup handler's critical section is read-only (linear scan + deep-copy + unlock), sub-millisecond for the default `cgroup_root_max=1000`; worst case is negligible against `cgroups_main`'s much larger window. If real-world load testing (post SOW-0034 / SOW-0035 consumer landing) shows measurable regression in collection latency or metric freshness, mutex fairness, a reader-writer lock, or finer-grained per-cgroup locking belong in a follow-up SOW — strict performance change, no wire-format or correctness impact. **Bidirectional contention note (R13D-5)**: contention is symmetric — conversely, discovery holding the mutex for seconds during `discovery_find_all_cgroups()` (the worst-case discovery iteration on a busy host) may delay lookup responses by the same duration. This is acceptable per D6: the lookup server is opportunistic enrichment for downstream consumers (apps.plugin, network-viewer), not a time-critical real-time API. Consumers MUST treat lookup-handler latency as best-effort and MUST NOT block their collection cycle on a lookup response. A consumer that cannot tolerate seconds-scale latency belongs in a different architecture, not a tighter lookup-handler SLA.
- **Detection-pattern correctness across distros / runtimes**: heuristic on cgroup path strings. Validation must include synthetic fixtures covering the full matrix listed in Acceptance Criteria (including the libvirt LXC and KVM patterns ported from `cgroup-name.sh.in:693-701`). A missed pattern silently classifies the cgroup as UNKNOWN — visible regression for that orchestrator's containers.
- **Enum value mapping risk**: if the plugin enum values do not match the wire enum exactly, every CGROUPS_LOOKUP response emits the wrong orchestrator. Mitigation: enforced by acceptance criterion (direct cast must work) and verified by a compile-time `_Static_assert((int)CGROUPS_ORCHESTRATOR_SYSTEMD == NIPC_ORCHESTRATOR_SYSTEMD)` per value.
- **NULL `cgroup_root` window**: the lookup server starts at `cgroup-discovery.c:1173`, before the first discovery at line 1185. Without the explicit "generation == 0 → RETRY_LATER" rule, a fast consumer could receive PERMANENT for every cgroup it asks about during startup, permanently blacklisting them per Contract 2. Mitigated by AC.
- **Reaped-set memory bound**: bounded at 4096 entries default. On hosts with extreme container churn (e.g., short-lived K8s init containers) the LRU eviction could cause a previously-reaped cgroup to drop out of the set, then return RETRY_LATER instead of PERMANENT on a subsequent query — minor degradation: consumer wastes one round-trip retrying for a vanished cgroup. Acceptable.
- Forward-compat: the netipc wire enum may grow in future. If we add a new orchestrator to `nipc_orchestrator_t` without updating the plugin enum, the plugin emits 0 (UNKNOWN). Consumer must treat unknown enum values as UNKNOWN. (This is already the netipc decoder forward-compat rule.)
- **Proxmox file-I/O in detection path**: the Proxmox LXC branch (`cgroup-name.sh.in:715-721`) and the Proxmox KVM branch (`cgroup-name.sh.in:707-714`) require reading config files under `${NETDATA_HOST_PREFIX}/etc/pve/lxc/<id>.conf` and `${NETDATA_HOST_PREFIX}/etc/pve/qemu-server/<id>.conf` to confirm the cgroup name. For structured orchestrator classification, the C port only needs to **confirm the Proxmox path pattern matches AND the `${NETDATA_HOST_PREFIX}/etc/pve` directory exists** (a single `stat()` per detection call, cacheable across all cgroups within one discovery cycle). It does NOT need to open the per-container/per-VM config file — friendly-name resolution from those files remains the responsibility of `cgroup-name.sh.in`. Mitigation: cache the `/etc/pve` directory-existence result for the duration of one `discovery_find_all_cgroups()` iteration (single `stat` per cycle, not per cgroup). Discovery runs in a separate thread (Contract 9); even an uncached worst case (one `stat` per cgroup) is `stat`-class latency, not `docker inspect`-class. Acceptable per D6.
- **Linear-scan batch-lookup latency**: per Decision L12, the lookup handler uses linear scan over `cgroup_root` (matching the existing snapshot server). For a batch of N paths against M cgroups, complexity is O(N·M). With `cgroup_root_max` default 1000 (`sys_fs_cgroup.c:34`) and N bounded by netipc per-request capacity (already constrained by `NIPC_*` size limits at `netipc_protocol.h:458-466`), worst-case batch latency is bounded by `M_max × N_max` element comparisons, each a path-string compare. Mitigation: (a) the cgroup count is capped at `cgroup_root_max`, so M is bounded; (b) the handler holds `cgroup_root_mutex` only for the read window, so worst-case impact on `cgroups_main` is one extra mutex wait per cycle; (c) if measured latency becomes user-visible, replace linear scan with a hash table keyed by `cg->hash` (see Followup) — this is a strict performance change, not a correctness change. Acceptable for the initial implementation per D6.

Ordered this way because container orchestrators may live under systemd-managed paths (e.g. `/machine.slice/*.service` for nspawn, `/system.slice/podman-*.scope` for rootful Podman) and must be classified by their container pattern FIRST, before the generic SYSTEMD fall-through. Note: `is_cgroup_systemd_service()` flag is set only for `.service` units under `/system.slice/`, so Docker rootful `.scope` units don't collide with SYSTEMD even without ordering, but ordering remains correct discipline.
- **L6 RETRY_LATER vs PERMANENT for in-cgroup_root-but-not-yet-processed**: RETRY_LATER. Both `processed == 0` and `pending_renames > 0` yield RETRY_LATER. PERMANENT is reserved exclusively for the reaped-set membership case.
- **L7 Reaped-set policy**: bounded LRU. Default size 4096. Configurable via `[plugin:cgroups] cgroup lookup reaped set size`. On overflow: drop the oldest entry. Trade-off (overflow → entry returns to RETRY_LATER state, consumer wastes one round-trip) is acceptable per the Risk register.
- **L8 NULL cgroup_root handling**: handler returns RETRY_LATER for ALL queries when `cgroup_discovery_generation == 0` (before first discovery completion). Detected via atomic load.
- **L9 Discovery generation storage**: `_Atomic uint64_t cgroup_discovery_generation` at file scope in `sys_fs_cgroup.c`. Acquire/release semantics for the load/store. Bump inside `discovery_find_all_cgroups` **while holding `cgroup_root_mutex`** (just before the unlock at `cgroup-discovery.c:1141`), using `__atomic_add_fetch(&cgroup_discovery_generation, 1, __ATOMIC_RELEASE)`. Under-mutex bump guarantees that any handler that subsequently acquires the mutex observes a consistent `(generation, cgroup_root)` snapshot.
- **L10 Existing snapshot server refactor**: in-scope for this SOW. The per-request `static uint64_t generation` at `cgroup-netipc.c:39,49` is removed and replaced with reads of the shared `cgroup_discovery_generation`. This is a behavioural change in the snapshot response semantics (generation now bumps with discovery, not with requests) — **verified safe** for the existing `ebpf.plugin` consumer by grepping the ebpf.plugin source: `ebpf_cgroup.c:347` reads `ebpf_cgroup_cache.generation` and uses it for LOGGING ONLY, not for cache eviction logic. The refactor is non-breaking. Verification step (acceptance criterion): include a smoke test that runs ebpf.plugin against a refactored cgroups.plugin and confirms no regression in cgroup tracking.
- **L11 Test client location**: `src/collectors/cgroups.plugin/cgroup-lookup-test/cgroup-lookup-test.c` (small C program, linked against libnetdata + netipc client helpers). Build wired into CMakeLists.txt as an OPTIONAL target (default OFF), not installed in production packaging.
- **L12 Lookup algorithm**: **linear scan** over the `cgroup_root` linked list inside the handler. Matches the existing snapshot server pattern at `cgroup-netipc.c:56`. Rationale: (a) zero new data-structure surface — no parallel hash table to keep in sync with `cgroup_root` mutations; (b) `cgroup_root_max` default 1000 caps the per-call work; (c) implementation cost is one nested `for` loop, not a new container with its own lifecycle. Trade-off accepted in the Risk register entry "Linear-scan batch-lookup latency". If real-world batches expose this as a bottleneck, swap to a `cg->hash`-keyed hash table in a follow-up (see Followup) — strict performance change, not a correctness change. No wire-format implication.
- **L13 Proxmox detection — directory check caching**: the Proxmox KVM and Proxmox LXC branches BOTH require `${NETDATA_HOST_PREFIX}/etc/pve` directory-existence. To keep detection cheap, cache the result of this `stat()` for the duration of one `discovery_find_all_cgroups()` iteration (single function-static or thread-local flag refreshed once per discovery cycle, not per cgroup). The cache invalidation happens implicitly at the start of each discovery cycle. No persistent cache — Proxmox install/uninstall is reflected within one cycle.

## Lessons Extracted

- Lookup-style netipc responses are exact-count contracts. Snapshot-style truncation is not valid for lookup services because C/Rust/Go dispatchers and clients all reject item-count mismatch.
- Test-only helpers should be guarded in both declaration and implementation. Guarding only the implementation leaves a misleading production header surface.
- Normal info logs should clear stale `errno` before logging when no error is being reported; otherwise prior syscall failures can appear attached to successful startup messages.
