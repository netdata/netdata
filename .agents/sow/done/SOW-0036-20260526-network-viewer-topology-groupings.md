## Requirements

### Purpose

Use the per-PID enriched metadata (cgroup path, orchestrator, container name, labels, pod identity) now available in network-viewer.plugin (via the cache populated by SOW-0035) to add new topology groupings to the network-viewer Function output. This is the first user-visible payoff of the topology-containers integration.

Concretely, network-viewer topology will gain the ability to group network nodes by:

- Container (cgroup_name).
- Orchestrator (Docker / Kubernetes / systemd-service / LXC / podman / nspawn / KVM).
- For Kubernetes: pod / namespace / workload (when labels expose this).
- For Docker: container name / image (when labels expose this).
- For systemd: service unit name.

The existing per-PID view remains. The **Agent Function output** advertises the new columns, the 7 new `aggregation_scopes`, and the 9-element `group_by` selector list, so the wire format is ready for UI/Cloud consumers to drive grouping. **User-visible switching in the Cloud frontend is NOT in scope of this SOW** (verified round-9 — reviewer CRITICAL-2): the Cloud frontend at `cloud-frontend/src/domains/functions/topology/v1/normalizeTopologyV1.js:458-460` only surfaces `aggregationScopes` as metadata "for PR-3 to consume" today, and the topology service at `netdata/cloud-topology-service/internal/topology/aggregate/aggregate.go:855-899` still merges actors by `identity`/`merge_identity`, NOT by a requested `group_by`. SOW-0036 ships the producer half (Agent emits the right payload) and is gated on the consumer half (Cloud frontend + topology service implement `group_by`-driven scope selection) shipping in a separate Cloud-side SOW before operators see the switching UI. This split keeps the Agent ready and unblocked while the Cloud team independently lands the consumer work.

For `self` and `endpoint` actor rows the fields remain zero-initialised → emitted as JSON `null`. The columns themselves exist in `topology_v1_emit_actor_columns()` for ALL rows (table columns are global to the actor table); the values are simply null for non-process rows.

## Analysis

Sources checked: Master Plan SOW-0032; SOW-0021/0028/0031 in `.agents/sow/done/`; `.agents/skills/project-create-topology/SKILL.md`; `network-viewer.c` (specific line citations in Facts above); `<repo>/src/plugins.d/FUNCTION_TOPOLOGY_SCHEMA.json`; `<repo>/src/plugins.d/FUNCTION_TOPOLOGY_DEVELOPER_GUIDE.md`.

Current state: network-viewer renders per-PID topology only; no container/orchestrator grouping. The Function output structure is well-established (20 actor columns, 3 aggregation scopes, 3 actor types). Adding new columns is a natural extension via the dynamic `actors.columns[]` array — the topology v1 schema does not maintain a per-actor-type column registry; producers declare columns at runtime.

Risks:

- **Schema correctness against topology v1 contract**: any deviation breaks Cloud rendering. Mitigation: schema-validation harness from SOW-0021/0031 is run; the new columns/scopes are added to the schema spec in the same commit as the producer changes.
- **Cloud aggregator dependency**: the new columns must round-trip through the aggregator unchanged. Per SOW-0031 the aggregator passes through unknown columns; verify with a test query against a development backend before declaring complete. If the aggregator drops or rejects them, this becomes a coordinated cross-team SOW.
- **UX of "too many dimensions"**: adding cgroup/pod/orchestrator to PID can produce overwhelming output. Mitigation: groupings are selectable views via `group_by`, NOT always-on rows. Default `group_by` remains `process_name` (existing default).
- **Label exposure**: real labels from production K8s clusters may contain customer information. Mitigation: producer-side **Function argument** `labels:<simple-pattern>` (round-9 — reviewer MAJOR-4 wording fix: previous revision called this a "config knob", but it is a per-Function-call argument parsed by `topology_parse_options()`, NOT a netdata.conf knob), DENY-BY-DEFAULT for free-form labels (empty whitelist → emit nothing); operators must opt-in or set `*` to pass-all. - **`docker_image` registry-hostname PII (added round-10 — reviewer minor; mirrors the sensitive-data plan qualifier)**: the `docker_image` column emits unconditionally when present in the cache and is NOT subject to the `labels:<pattern>` whitelist. - **Transient state during container teardown**: PID still has connection, container metadata says "gone". Mitigation: per Contract 2, SOW-0035's cache evicts on UNKNOWN_PERMANENT; SOW-0036 emits empty/null for the new columns; topology renders the PID without container affiliation until the PID itself exits.
- **Payload size increase**: 9 nullable string columns per process actor. JSON cost per populated string ≈ `len(string) + 2` bytes (no dictionary encoding for these columns). On a host with 500 process actors, 50% container coverage, average populated-string length ≈ 32 chars across the 9 columns → ~75 KB additional JSON per Function call worst-case (most columns unpopulated except cgroup_name, orchestrator, one orchestrator-specific column). On hosts with no containers → ~0 KB extra (all null). Negligible relative to overall actor metadata payload. In-memory struct cost is ~1152 bytes per actor (see Memory model decision); for 500 actors that is ~560 KB extra resident memory per Function execution. **High-churn worst-case (quantified round-6)**: 10000 actors × ~1152 bytes ≈ 11.5 MiB extra resident memory per Function execution. ## Lessons Extracted

- Integration markdown is generated from `metadata.yaml`; direct edits to generated integration docs should be replaced by a metadata update plus `gen_integrations.py` / `gen_docs_integrations.py`.
- Public Function metadata must satisfy the collector metadata schema even for optional parameters: parameter objects require `id`, `name`, `description`, `type`, `required`, `default`, and `options`; return columns require at least `name`, `type`, and `unit`.
- Whitelisted cgroup labels should remain distinguishable from network-viewer-owned labels; emitting `source="cgroups"` and `kind="label"` makes that provenance machine-checkable.
