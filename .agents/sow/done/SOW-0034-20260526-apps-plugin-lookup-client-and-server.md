## Requirements

### Purpose

Make `apps.plugin` the bridge between cgroups metadata and per-PID metadata:

1. **CGROUPS_LOOKUP client.** apps.plugin already reads `/proc/<pid>/cgroup` to know each PID's cgroup path. It will additionally query cgroups.plugin (via the CGROUPS_LOOKUP server from SOW-0033) for the structured orchestrator, friendly name, and labels associated with that cgroup. Results are cached per Contract 1; cache keyed by cgroup path.
2. **APPS_LOOKUP server.** apps.plugin exposes per-PID enriched metadata to downstream consumers (network-viewer.plugin in SOW-0035). Each lookup returns the PID's `comm`, `ppid`, `uid`, `starttime`, plus the cgroup data it learned via CGROUPS_LOOKUP.

apps.plugin becomes both a consumer (of cgroups.plugin) and a producer (for network-viewer.plugin). This SOW is the first end-to-end exercise of an IPC chain in the topology integration.

### User Request

See master plan SOW-0032 for the full directive. This SOW implements step 2 of six.

- The CGROUPS_LOOKUP client runs on a **dedicated async cache-updater thread** (added by this SOW, R6-final-reviewer-C1). The main collection loop NEVER issues IPC; it enqueues unresolved cgroup paths onto a request queue and reads only from the local cache. The async worker dequeues paths, batches, sends CGROUPS_LOOKUP requests, and commits responses under `apps_pids_mutex`. The APPS_LOOKUP server runs on its own NEW dedicated thread parallel to the main collection loop.
- A new mutex `apps_pids_mutex` is added to protect the PID table (`pids.all_pids` + per-PID `struct pid_stat` fields used by APPS_LOOKUP). Main collection loop acquires it during PID-table mutations; APPS_LOOKUP handler acquires it during the read window. Stdout protection remains on `apps_and_stdout_mutex`; these two mutexes are NOT nested in the same direction (the new mutex never wraps the existing one).
- Cache key for CGROUPS_LOOKUP results is the cgroup path string. Cache value is the lookup response (orchestrator, name, labels). Memory bound: number of unique active cgroups at any time × per-entry size, bounded by total active cgroups on the host (a few thousand typical, ~10000 extreme).

- **Cache architecture (two tiers + async worker, R6-final-reviewer-C1)**:
  1. **Global `cgroups_lookup_cache` hashtable** (file-scope in the new `apps-cgroups-lookup-client.c`, key: `STRING *cgroup_path`, value: `struct cgroup_lookup_entry *`). Each entry holds: `STRING *key` (the cache's OWN reference to the keying cgroup path STRING — see ownership rule below), `cgroup_status`, `orchestrator`, `STRING *cgroup_name`, a cache-owned copied label array (`struct cgroup_lookup_label { STRING *key; STRING *value; }`), `uint64_t generation`, `uint64_t last_used_iteration` (for LRU), and `uint32_t refcount` (number of pid_stats whose `cgroup_cache` points at this entry). Refcount is incremented when a pid_stat links to the entry and decremented when the pid_stat is unlinked (path change, PID exit, or generation invalidation), all under `apps_pids_mutex`. One entry per unique cgroup path → labels and name are stored once even when many PIDs share a cgroup. Bounded LRU at 4096 entries (per L5).
     - **Key STRING ownership (R8k-M2, mandatory)**: each cache entry holds its OWN reference to the keying `STRING *cgroup_path`, distinct from the references held by linked pid_stats. On entry allocation: `entry->key = string_dup(path);` (where `path` is the STRING currently held by `p->cgroup_path` — `string_dup()` bumps the libnetdata interned-string refcount, returning the same pointer). On entry eviction (Trigger 3 only — Triggers 1 and 2 KEEP the entry alive and never free the key): `string_freez(entry->key);` releases the entry's reference. pid_stats hold INDEPENDENT references to the same interned STRING (acquired via `string_strdupz()` in chunk 3 when the cgroup file is parsed) and free them INDEPENDENTLY in `del_pid_entry()` (chunk 1). This double-reference pattern is the libnetdata interned-STRING idiom: every distinct owner takes its own reference; the underlying storage is freed when the last reference drops. Without this rule, evicting a cache entry whose key matches a still-live pid_stat's `cgroup_path` would risk underflow / use-after-free if the cache borrowed the pid_stat's reference.
  2. **Per-PID pointer into the cache**. `struct pid_stat` gains the fields below; the lookup-result fields are NOT duplicated on the pid_stat.
  3. **Request queue + async cache-updater thread**: a bounded FIFO `cgroups_lookup_request_queue` of pending `STRING *cgroup_path` entries (capacity `APPS_CGROUPS_LOOKUP_QUEUE_MAX = 4096`, equal to the LRU bound — same upper bound on unique unresolved paths in flight). Operations are protected by a dedicated `netdata_mutex_t cgroups_lookup_queue_mutex` with a `netdata_cond_t cgroups_lookup_queue_cond`. **Queue entry STRING ownership invariant (R10k-MAJOR — mandatory)**: every entry on the queue owns its OWN independent reference to the `STRING *cgroup_path` it carries; the queue's lifecycle for that reference is fully independent of the originating `pid_stat`'s lifetime. The async worker may pop, perform IPC, and commit milliseconds-to-seconds after the main loop enqueued, by which time the originating pid_stat may have exited and `del_pid_entry()` may have `string_freez`'d the pid_stat's reference. Without an independent queue reference, the worker would dereference freed STRING storage. Enqueue/dequeue/drain rules: (a) **enqueue** — caller computes `STRING *dup = string_dup(p->cgroup_path);` (bumps the interned-string refcount) and pushes `dup` onto the queue; (b) **dequeue** — worker pops the entry, uses the STRING through IPC and commit, then calls `string_freez(path);` AFTER the response has been committed OR dropped (release the queue's reference); (c) **capacity drop** — if the queue is full at enqueue time, the caller MUST NOT have called `string_dup` yet (preferred — test capacity first), OR if `string_dup` was already called and the push fails, the caller MUST `string_freez(dup)` immediately on the drop path so the reference is not leaked; (d) **cleanup** — `apps_cgroups_lookup_cleanup()` (chunk 6) drains every remaining entry on the queue and calls `string_freez(path)` for each before destroying the queue mutex/condvar and the queue storage. The main loop's scan step (under `apps_pids_mutex`) enqueues each unresolved path **once** (dedupe via a "pending" bit on the cache entry — set on enqueue, cleared when the response arrives or the entry is evicted) and signals the condvar; if the queue is at capacity the path is dropped silently and re-tried on the next iteration (this is rate-limiting under pathological miss rates, not data loss — the PID's status remains UNKNOWN_RETRY_LATER, which is the correct semantic). The async worker drains the queue, batches paths up to `NIPC_MAX_PAYLOAD_CAP`, issues CGROUPS_LOOKUP requests (no per-request timeout exists in netipc — verified `src/libnetdata/netipc/src/service/netipc_service.c:573-642`; this is acceptable here because the worker is the ONLY thread that blocks on IPC), and commits responses to the cache under `apps_pids_mutex`. Main loop never blocks on IPC. Invariant restated: **queue entries own their `STRING *` reference; their lifecycle is independent from pid_stat lifetime, and the queue is fully responsible for `string_freez` on every entry it ever held.**

**Client side (CGROUPS_LOOKUP consumer — async worker model, R6-final-reviewer-C1):**

- The CGROUPS_LOOKUP **async cache-updater thread** owns the netipc client socket and is the ONLY thread that issues `CGROUPS_LOOKUP` requests. It connects to cgroups.plugin's CGROUPS_LOOKUP service at startup. On connection failure: logs once at INFO, retries connect every 30s. While the peer is absent the cache stays as-is; any PID whose cache entry never populated reports `UNKNOWN_RETRY_LATER` via the APPS_LOOKUP handler (Contract 5 fallback). The async worker does NOT touch per-PID state on connection failure — it only touches cache entries, and only via the response-commit path.
- **Main collection loop's role** (no IPC): once per iteration, while holding `apps_pids_mutex`, the main loop scans `pids.all_pids` and for each PID with a non-empty `cgroup_path` AND non-HOST_ROOT path: looks up the cgroup_path STRING in `cgroups_lookup_cache`. If a hit, links `p->cgroup_cache = entry` (and bumps `entry->refcount` if newly linked); if a miss, allocates a placeholder cache entry with `cgroup_status = UNKNOWN_RETRY_LATER` and `generation = 0`, links the pid_stat to it, and **enqueues the cgroup_path STRING onto `cgroups_lookup_request_queue`** (unless the entry's `pending` bit is already set, in which case the request is already in flight or already queued). This main-loop scan acquires only `apps_pids_mutex` plus a brief `cgroups_lookup_queue_mutex` window around the enqueue+signal — no IPC.
- **Async worker thread**: dequeues paths (blocking on the condvar when the queue is empty), batches into one or a few CGROUPS_LOOKUP requests bounded by `NIPC_MAX_PAYLOAD_CAP`, issues the request to cgroups.plugin, waits for the response (no timeout — verified `src/libnetdata/netipc/src/service/netipc_service.c:573-642`), then under `apps_pids_mutex` finds each path's cache entry and writes the response fields (deep-copying label key/value strings into a cache-owned copied label array), updates `last_used_iteration`, sets `generation` to the response's generation, and clears the `pending` bit. The pid_stat's `cgroup_cache` pointer is **unchanged** (already linked by the main loop). Refcounts are NOT touched on response (the link was made on enqueue).
- **Generation bump from cgroups.plugin** (per Contract 4, R6-final-reviewer-M3): when the async worker observes a higher `generation` field on any response, it resets every cache entry's payload (`cgroup_status = UNKNOWN_RETRY_LATER`, frees the old `cgroup_name` STRING, frees the copied label strings, clears `orchestrator`) under `apps_pids_mutex` AND clears each entry's `generation` so the next time the main loop scans, every linked pid_stat will trigger a re-enqueue. Per-pid_stat `cgroup_cache` pointers and refcounts are left intact. Because APPS_LOOKUP derives status only from the (now-reset) cache entry, no stale data can leak — the handler will observe `UNKNOWN_RETRY_LATER` for those entries until the worker repopulates them.
- Cgroup-path-changed mid-life: detected in the cgroup-reading step (chunk 3) by comparing the newly-parsed `cgroup_path` STRING identity against the previously-stored one on the same `struct pid_stat`. On mismatch: replace `p->cgroup_path` with the new STRING; under `apps_pids_mutex`, if `p->cgroup_cache != NULL` decrement `p->cgroup_cache->refcount`; set `p->cgroup_cache = NULL`. The next main-loop scan will re-link the pid_stat to (a new or existing) cache entry keyed by the new path and enqueue if needed. The old cache entry is NOT freed here — it may still be referenced by other pid_stats, and will be evicted by L5 trigger 3 if it loses all referencers and ages out.
- **Request-validation note**: the wire protocol is defined by SOW-0033's CGROUPS_LOOKUP server; the async worker uses the existing netipc decoders which already enforce `NIPC_MAX_PAYLOAD_CAP` and per-field length bounds.

- **Lock-ordering invariant** (mandatory, enforced by reviewer):
  - **Three-mutex ordering rule (R8k-M3)**: when more than one of `{apps_and_stdout_mutex, apps_pids_mutex, cgroups_lookup_queue_mutex}` is held simultaneously, the order MUST be `apps_and_stdout_mutex` (outermost) → `apps_pids_mutex` (middle) → `cgroups_lookup_queue_mutex` (innermost). This is a total order; any partial subset of the three must be acquired in this order and released in the reverse order.

  - `apps_pids_mutex` is the **middle** lock.
  - `cgroups_lookup_queue_mutex` is the **innermost** lock.   - CGROUPS_LOOKUP async cache-updater thread holds the innermost when popping from the queue, releases it before any other lock or IPC; later holds **only** the middle lock during commit windows. Never touches the outer.

## Analysis

Sources checked: see Master Plan SOW-0032; plus `src/collectors/apps.plugin/apps_plugin.{c,h}` (mutex declaration, main loop), `apps_pid.c` (PID table, collect_data_for_all_pids), `apps_os_linux.c` (per-PID `/proc` reads), `apps_incremental_collection.c` (incremental reader), `apps_targets.c`, `apps_functions.c` (the existing functions thread that takes `apps_and_stdout_mutex`), `apps_aggregations.c`, `apps_output.c`. Plus `netipc_protocol.h`, `netipc_service.h`, and `cgroup-netipc.c` (template for the new server).

Current state:

- apps.plugin has zero cgroup awareness today. The premise of this SOW requires substantial new code, NOT just IPC wiring.
- The main collection loop is single-threaded, but the process already has evloop worker threads; after this SOW there are also the CGROUPS_LOOKUP async cache-updater thread and the APPS_LOOKUP server thread. Only `apps_and_stdout_mutex` exists today, and it protects stdout, not the PID table.
- `struct pid_stat` lacks the fields needed for both the client cache and the server response.

Risks:

- **New code volume**: this SOW is substantially larger than originally framed. The CGROUPS_LOOKUP client wiring is straightforward once the prerequisites are in place; the new cgroup-reading code, struct extensions, mutex addition, two-tier cache, and threading infrastructure are the real work.
- **Performance regression from per-PID `/proc/<pid>/cgroup` reads**: on a 10k-PID host with update_every=1, this is ~30k extra syscalls/sec (open + read + close per PID per iteration). The existing `p->read` flag (set in `pid_collection_started`, cleared per-iteration) prevents double-reads in the same iteration but does NOT skip iterations. Measurement required as part of Pre-Implementation Gate. Mitigation if measurement exceeds target: two-pass scheme (read outside mutex, commit briefly under mutex).
- **Mutex contention window is the ENTIRE collect+aggregate phase**, not just the hashtable resize moments (per round-3 reviewer-d/reviewer-e review). The APPS_LOOKUP handler waits across this whole window. Mitigation: the collection phase is bounded by the cost of `/proc` iteration; on a 10k-PID host this is hundreds of ms but well below 1s. Measurement target: handler p99 wait <50ms (set conservatively; allow re-evaluation if real-world workloads show acceptable bursts up to ~200ms during the aggregate phase). **Sub-step contributors inside the critical section (R11q-MINOR-4)**: the `apps_pids_mutex` window opened at `apps_plugin.c:843` covers (in order) `collect_data_for_all_pids` → which calls `apps_os_collect_all_pids_linux` (`apps_pid.c:914-916`) → which calls `collect_parents_before_children()` at `apps_os_linux.c:1018` (a Linux-specific pre-pass that ensures parent PIDs are read before children for correct uptime/CPU accounting) → then the main `/proc` directory walk at `apps_os_linux.c:1029+` → then `link_all_processes_to_their_parents()` at `apps_pid.c:937` → then `process_exited_pids()` at line 941 → returns to main loop → `aggregate_processes_to_targets()` → generation bump → cache-link / request-enqueue scan. Every one of these runs inside `apps_pids_mutex` and contributes to critical-section duration. The measurement gate at chunk 11 implicitly covers all sub-steps because per-iteration wall-clock is measured end-to-end across the mutex window, not split per function — but listing them explicitly here so future maintainers know the full set of contributors before considering optimizations.
- **`function_processes` blocked on `apps_pids_mutex`**: when an operator runs the `processes` function while collection is in progress, the function waits for the mutex during its PID-iteration phase (lines 1082-1295). The JSON emission phase (lines 1296-1383) no longer holds the inner mutex (R7q-M3), so the APPS_LOOKUP handler is not blocked across JSON serialization. The `processes` function can also be called programmatically (dashboards, API consumers), so this is not exclusively user-driven; narrowing the window reduces the worst-case stall accordingly. Remaining contention shape is the same as collect+aggregate. Acceptable; further narrowing would require the rejected buffer-copy refactor.
- **CGROUPS_LOOKUP IPC has no per-request timeout** (round-4, reviewer-b-#2 / reviewer-c-#1; mitigation upgraded R6-final-reviewer-C1): the netipc library does not support a per-request timeout (verified `src/libnetdata/netipc/src/service/netipc_service.c:573-642` and `src/libnetdata/netipc/include/netipc/netipc_uds.h:59-68`). Mitigation: all CGROUPS_LOOKUP IPC runs on the dedicated **async cache-updater thread**, which holds NEITHER `apps_and_stdout_mutex` NOR `apps_pids_mutex` during the round-trip. An unbounded stall on the IPC does NOT delay the main collection loop's `heartbeat_next()`, does NOT block `function_processes`, and does NOT block the APPS_LOOKUP server thread. The only visible effect is that PIDs whose cgroup info is pending stay at `cgroup_status = UNKNOWN_RETRY_LATER` until the worker recovers — which is the Contract 5 behavior the consumer already handles. Follow-up: if real-world experience shows persistent IPC stalls, a `timeout_ms` field can be added to `nipc_uds_client_config_t` and threaded into `call_with_retry` — separate SOW. Risk shifted from "main loop blocks on IPC" to "request queue may saturate under pathological miss rates" — the bounded FIFO (`APPS_CGROUPS_LOOKUP_QUEUE_MAX = 4096`) caps the worst case; dropped paths re-try next iteration.
- **Threat-surface change from new inbound UDS server** (R6-final-reviewer-M5): adding the APPS_LOOKUP UDS socket to a capability-bearing binary (`cap_dac_read_search,cap_sys_ptrace`, see `packaging/cmake/pkg-files/deb/plugin-apps/postinst:7-10`) widens the local attack surface. Mitigations: (a) UDS socket mode 0600, owned by `netdata` user — only same-uid processes can connect; (b) request validation hardened per AC (`NIPC_MAX_PAYLOAD_CAP`, `APPS_LOOKUP_MAX_PIDS_PER_REQUEST = 8192`, existing netipc primitive decoders enforce per-field bounds); (c) handler exits on malformed request and closes connection — never crashes the server thread; (d) responses contain only per-PID metadata already collected by apps.plugin and forwarded labels — no new sensitive fields are exposed beyond what `function_processes` already returns to the local netdata daemon. README Security section updated to describe the new bidirectional model (see Artifact impact plan).
- **Memory bound**: cache size grows with unique cgroups (NOT PIDs, since labels are deduplicated per cgroup in the two-tier model). Bounded LRU at 4096 unique cgroup paths (per Contract 1 + L5). Typical host: a few hundred to a few thousand cgroups; the bound is generous.
- **OS coverage**: CGROUPS_LOOKUP and APPS_LOOKUP server are both `#ifdef OS_LINUX` guarded (revised per round-3). Justification: network-viewer.plugin only runs on Linux today; the APPS_LOOKUP server has no non-Linux consumer, so building it on non-Linux would be dead code. The cgroup parsing inside `apps_os_linux.c` is naturally Linux-only.
- **Hash table resize while APPS_LOOKUP handler iterates**: if the handler walks `pids.all_pids.ht` and the main loop triggers a resize concurrently, UB results. Mutex prevents this — handler holds the lock for the iteration. The PID hashtable size is `pids.all_pids.count` which on a steady-state host changes by ±few per iteration; resizes are bounded and rare.
- **Cross-process socket permissions** (per round-3 reviewer-c observation): the apps-lookup UDS socket is created by apps.plugin; clients (network-viewer in SOW-0035) connect from the netdata daemon side. Both run as the `netdata` user in production. Socket permissions follow the existing netipc convention (0600 owner-only). Documented for the SOW-0035 author.
- **Cold-start nested mutex acquisitions (R10q-MINOR)**: the main-loop cache-link scan (chunk 7) acquires `cgroups_lookup_queue_mutex` per enqueue, nested inside `apps_pids_mutex`. On a cold-start iteration with N unresolved non-HOST_ROOT PIDs (e.g. N = 10000 on a high-PID host where the cache is empty), this is N brief lock/unlock cycles of `cgroups_lookup_queue_mutex`, each consisting of a FIFO push plus a `pthread_cond_signal()` — measured in microseconds per cycle on commodity hardware. Total overhead is negligible compared to the N hashtable lookups against `cgroups_lookup_cache` performed in the same scan, but is worth recording so future maintainers do not redesign for a problem that does not exist at expected scales. If a future host pushes N into the tens of thousands and profiling shows this becoming a bottleneck, the scan can be reshaped to enqueue in batches under a single queue-mutex window — flagged for follow-up but not addressed here.
- **`apps_lookup_netipc_cleanup()` idempotency (R8k-m7)**: the cleanup function may be called multiple times (process shutdown after the fallback test already invoked it, repeated SIGTERM handlers, paranoid defensive code). Mechanism: NULL-check on a file-scope `static ND_THREAD *apps_lookup_netipc_thread = NULL;` handle; second-and-subsequent calls return at the NULL-check. Mirrors `cgroup_netipc_cleanup()` at `src/collectors/cgroups.plugin/cgroup-netipc.c:171-184`. No atomic flag is needed because the function is only invoked from single-threaded contexts (process shutdown and the fallback-test entry point). If a future caller invokes `apps_lookup_netipc_cleanup()` concurrently from multiple threads, an atomic stop-flag would be required — flagged for SOW-0035 / future work but not addressed here because no such caller exists.

Decisions inherited from SOW-0032 (D1-D11).

Local decisions (resolved 2026-05-26 after round-1; L3, L5, L7, L8, L9, L10 revised after round-3 reviewer feedback; L4, L5, L7, L9 further revised and L12 added after round-6 final-reviewer review):

- **L1 Service name**: `"apps-lookup"` (hard-coded constant). Per D11.
- **L2 Worker thread count**: `APPS_NETIPC_LOOKUP_WORKER_COUNT = 2`.
- **L3 New mutex**: `apps_pids_mutex` (dedicated) — NOT reuse of `apps_and_stdout_mutex`. Lock-ordering (revised per round-3 review): `apps_and_stdout_mutex` is the outer lock; `apps_pids_mutex` is the inner lock. Main loop and `function_processes` both hold outer then acquire inner. APPS_LOOKUP handler holds ONLY inner. The forbidden inversion (acquire outer while holding inner) is the deadlock cycle; APPS_LOOKUP must never write to stdout to keep this invariant intact.
- **L4 Lookup frequency**: cache-first, async. Global cgroup_path cache (per L5); per-PID `cgroup_cache` pointer reused across iterations; main loop NEVER issues IPC — it only links pid_stats to cache entries and enqueues unresolved paths onto the request queue (per L12). The async cache-updater worker is the only thread that performs CGROUPS_LOOKUP IPC, on misses or after a generation bump.
- **L5 Cache eviction** (revised per round-3 review — two-tier model; round-4 added `refcount` to make eviction O(1) per candidate):
  - **Cache structure**: a single global hashtable `cgroups_lookup_cache` keyed by `STRING *cgroup_path` (interned-string identity), with each entry holding its OWN `STRING *key` reference (R8k-M2 — acquired via `string_dup()` on allocation, released via `string_freez()` ONLY on Trigger 3 eviction) plus the CGROUPS_LOOKUP result fields once (orchestrator, cgroup_name, cache-owned copied label array, generation, `last_used_iteration`, `refcount`). Each `pid_stat` carries a borrowed `struct cgroup_lookup_entry *cgroup_cache` pointer AND its own independent `STRING *cgroup_path` reference; both reference the same interned underlying storage but are reference-counted independently.
  - **LRU bound**: max 4096 unique cgroup paths. Counter: `size_t cgroups_lookup_cache_entries` (plain `size_t`, not atomic — always read/written under `apps_pids_mutex`).
  - **Reference counting** (round-4 addition, resolves reviewer-b-#1 / reviewer-c-LOW): each cache entry has `uint32_t refcount` — the number of `pid_stat`s whose `cgroup_cache` points at this entry. Maintained under `apps_pids_mutex` (the only mutex protecting both the cache and `p->cgroup_cache`). Increment sites: (a) CGROUPS_LOOKUP response handler links a pid_stat to an entry; (b) cache-hit fast path re-links a pid_stat that had `cgroup_cache == NULL`. Decrement sites: (a) path-changed mid-life (`p->cgroup_cache` cleared); (b) `del_pid_entry()` for a PID whose `cgroup_cache != NULL` — decrement on the pointee before freeing the pid_stat; (c) Trigger 1 (UNKNOWN_PERMANENT eviction) when a pid_stat is force-unlinked from the evicted entry. Refcount must reach 0 before an entry can be freed by LRU pressure (Trigger 3 below).
  - **Eviction triggers** (per Contract 1):
    1. Producer signals `UNKNOWN_PERMANENT` for a path → **keep the entry alive** and set `entry->cgroup_status = UNKNOWN_PERMANENT` under `apps_pids_mutex`; free the entry's `cgroup_name` STRING, free copied label strings, and clear `orchestrator` (no payload to surface). Linked pid_stats keep their `cgroup_cache` pointer; the APPS_LOOKUP handler then naturally returns `UNKNOWN_PERMANENT` derived from `p->cgroup_cache->cgroup_status` (R6-final-reviewer-M3 single-source-of-truth). Refcounts are NOT decremented and the entry is NOT freed — symmetric with Trigger 2's keep-alive policy. Critical: if the entry were freed and `p->cgroup_cache` were NULLed, the handler would map `cgroup_cache == NULL` to `UNKNOWN_RETRY_LATER` (per AC handler rule) and the next main-loop cache-link scan would re-enqueue the same path, defeating the point of `UNKNOWN_PERMANENT`. The entry can still be evicted later by Trigger 3 once all referencing pid_stats exit and its `refcount` drops to 0; the next main-loop scan will then either re-allocate a fresh placeholder (if a live PID still parses this path) and re-enqueue — which is acceptable because, by then, the previously affected pid_stats are gone.
    2. Producer generation bump (Contract 4) → reset every cache entry's payload to UNKNOWN_RETRY_LATER (clear `cgroup_status`, free `cgroup_name` STRING, free copied label strings, clear `orchestrator`, set `generation = 0`); keep entries in place so they can be repopulated by the async worker without losing the pid_stat pointers or perturbing refcounts. Because APPS_LOOKUP derives status only from `p->cgroup_cache` (R6-final-reviewer-M3), zeroing the entry's payload immediately stops stale-label leakage; per-pid_stat fields require no walk.
    3. LRU pressure: when inserting a new entry would exceed 4096, scan the cache (O(N) at N=4096, not a hot path) to find the candidate with `refcount == 0` AND the smallest `last_used_iteration`. The `refcount == 0` check is O(1) per candidate (a field read on the entry) — no nested scan of pid_stats is needed. **On eviction (R8k-M2)**: `string_freez(entry->key);` releases the cache's reference to the keying cgroup path STRING (any still-live pid_stat's `p->cgroup_path` is unaffected — pid_stats hold their own independent references); `string_freez(entry->cgroup_name);` releases the cache's name STRING; free copied label strings and array; remove from hashtable; free the entry struct. If no entry has `refcount == 0`, log a once-per-iteration WARNING and skip insertion (the new path stays `UNKNOWN_RETRY_LATER` until next iteration when some referenced pid_stat may have exited and dropped refcount to 0). Note: entries whose `refcount` reaches 0 (via PID exit in `del_pid_entry`, path-change mid-life, or producer-signaled invalidation) are **reclaimed lazily by Trigger 3 only** — they are never freed eagerly at the moment refcount drops to 0, so the cache may transiently hold up to 4096 `refcount == 0` "zombie" entries between LRU pressure events. This is bounded and intentional (eager free would require additional bookkeeping for no measurable benefit).
  - **`last_used_iteration` update**: bumped to a dedicated `static uint64_t cgroups_lookup_cache_iteration` counter (file-scope in `apps-cgroups-lookup-client.c`, plain `uint64_t` — only ever read/written under `apps_pids_mutex`) every time a pid_stat is re-linked to the entry during the CGROUPS_LOOKUP client's response handling. The counter is incremented once per CGROUPS_LOOKUP client pass, also under `apps_pids_mutex`. This avoids a data race with the existing `global_iterations_counter` (`apps_plugin.h:183` is a plain `size_t` declared without atomic; incremented at `apps_plugin.c:818` without holding `apps_pids_mutex`) and keeps the LRU ordering counter inside the same mutex window as the cache itself.
  - **`cgroup_cache_size` rename**: replaced by the descriptive `cgroups_lookup_cache_entries` to avoid the ambiguity of round-3.
- **L6 Thread model**: dedicated new thread for APPS_LOOKUP server. NOT reuse of `apps.plugin/apps_functions.c` thread.
- **L7 Cgroup-path-changed mid-life**: replace `p->cgroup_path` STRING, under `apps_pids_mutex` decrement the old `p->cgroup_cache->refcount` (if non-NULL), set `p->cgroup_cache = NULL`. No mirror field to reset (R6-final-reviewer-M3). The main-loop cache-link scan on the next iteration will re-link the pid_stat to (a new or existing) cache entry keyed by the new path and enqueue if needed. The old global cache entry remains (it may still be referenced by other pid_stats); the per-PID pointer just stops referencing it.
- **L8 HOST_ROOT detection** (revised per round-3 review — narrowed intent):

This is NOT a "PID is on the host versus in a container" classification — that distinction requires the orchestrator field returned by CGROUPS_LOOKUP. `/system.slice/sshd.service` is a systemd service, NOT a HOST_ROOT PID; apps.plugin must still query CGROUPS_LOOKUP for it (the response will set `orchestrator = SYSTEMD`).
  - **Rule**: HOST_ROOT iff (cgroup_path == `/` exactly) OR (cgroup_path == `/init.scope` exactly, the systemd v1 root scope where PID 1 lives). The empty-path case is removed (`/proc/<pid>/cgroup` always has at least one line on Linux; an empty parsed path indicates a parsing bug, which should result in `UNKNOWN_RETRY_LATER`, not HOST_ROOT — fail-closed).
  - **v1 multi-hierarchy confirmation (R7q-m5)**: on a standard cgroup v1 systemd host, PID 1's `/proc/1/cgroup` lists multiple lines (e.g. `1:name=systemd:/init.scope`, `2:cpu,cpuacct:/`, `3:memory:/`). The v1 precedence rule (cpuacct → blkio → memory → name=systemd → first) at AC line 93 selects the cpuacct line, whose path is canonically `/`. So PID 1 → cgroup_path == `/` → HOST_ROOT. Correct. The `/init.scope` rule remains as a safety net for hosts where the v1 cpuacct line for PID 1 reads `/init.scope` instead of `/`, and for hosts where cpuacct/blkio/memory are all disabled and the rule falls through to `name=systemd` (whose path for PID 1 is `/init.scope`).
  - HOST_ROOT PIDs skip the CGROUPS_LOOKUP query entirely (the path `/` is well-known and meaningless to lookup).
- **L9 Generation counter**: `_Atomic uint64_t apps_collection_generation`, bumped AFTER `aggregate_processes_to_targets()` (which includes `cleanup_exited_pids()`) — see AC for rationale (revised per round-3 review: bumping pre-cleanup would publish a generation that doesn't reflect the post-cleanup PID table, violating Contract 4). Concrete location: immediately after `aggregate_processes_to_targets()` returns at `apps_plugin.c:850` (new statement inserted on line 851, before the `#if (ALL_PIDS_ARE_READ_INSTANTLY == 0)` block at line 852), while `apps_pids_mutex` is still held. Verified 2026-05-26: line 850 is the call itself; the bump is the statement that follows. **Co-located with the deferred SERVER init (R11q-MAJOR)**: the new statement on line 851 is followed by the `if (!apps_lookup_server_started) { apps_lookup_netipc_init(); apps_lookup_server_started = true; }` guard block — both the generation bump and the first-time SERVER startup happen in the same window. This co-location is intentional: it guarantees the SERVER cannot publish a response with `generation=0` and an empty PID table (the bump precedes the init by a few CPU cycles within the same critical section). The CLIENT (CGROUPS_LOOKUP async cache-updater) init is NOT here — it lives at the early site (between lines 799 and 805) because it has no server-state issue. Returned in every APPS_LOOKUP response via `__atomic_load_n(.., __ATOMIC_ACQUIRE)`. Overflow: generation is monotonic within a single plugin session. `uint64_t` at `update_every=1` overflows after ~584 billion years — acceptable for production. The wire format already carries `uint64_t` with no sentinel; if a future need arises, the counter can be replaced with a monotonic clock-based value without a wire-format change.
- **L10 OS compilation** (revised per round-3 review): cgroup reading + CGROUPS_LOOKUP client + APPS_LOOKUP server are ALL `#ifdef OS_LINUX`. Reason: network-viewer.plugin only runs on Linux, so a non-Linux APPS_LOOKUP server has no consumer; the prior compromise of "compile non-Linux server that always returns UNKNOWN_PERMANENT" was dead code. If a future consumer is ported to BSD/macOS, the guards can be widened.
- **L11 No reaped-PID-set in apps.plugin**: per Contract 3 the consumer (network-viewer) refreshes its working set every iteration; PIDs that exit naturally fall out as the consumer stops asking. Simpler than the cgroups.plugin reaped-set (which exists because cgroup discovery is lazier than PID enumeration).
- **L12 CGROUPS_LOOKUP async cache-updater thread** (R6-final-reviewer-C1): dedicated thread spawned at apps.plugin startup, owns the CGROUPS_LOOKUP client socket, drains a bounded FIFO `cgroups_lookup_request_queue` (capacity `APPS_CGROUPS_LOOKUP_QUEUE_MAX = 4096`, protected by `cgroups_lookup_queue_mutex` + `cgroups_lookup_queue_cond`). Main collection loop NEVER issues IPC — it only enqueues unresolved cgroup paths during the cache-link scan (under `apps_pids_mutex`, with a brief queue-mutex window around the push+signal). Worker batches paths up to `NIPC_MAX_PAYLOAD_CAP`, sends CGROUPS_LOOKUP requests with NO mutex held during the round-trip (netipc has no per-request timeout, but this thread is the only blocker — main loop stays responsive), then commits responses to the cache under `apps_pids_mutex`. Dedupe of in-flight enqueues uses a `bool pending` bit on each cache entry. Queue saturation under pathological miss rates drops the path silently and retries next iteration (the PID's status remains UNKNOWN_RETRY_LATER, which is correct Contract 5 semantic). Lifecycle: `apps_cgroups_lookup_init()` / `apps_cgroups_lookup_cleanup()` symmetric to APPS_LOOKUP server hooks. The previous design (synchronous call from the main loop with mutexes released around IPC) was rejected in R6 because it still allowed unbounded stalls to delay the main loop's `heartbeat_next()` between iterations.
  - **Queue STRING ownership (R10k-MAJOR)**: queue entries own their `STRING *cgroup_path` reference; the lifecycle is fully independent from pid_stat lifetime. Enqueue calls `string_dup()` (bumps the libnetdata interned-string refcount); dequeue/drop/cleanup calls `string_freez()` (releases the queue's reference). This is mandatory because the worker may pop and dereference seconds after the enqueue, by which time the originating pid_stat may have exited and `del_pid_entry()` may have released the pid_stat's reference — without the queue's own reference, the worker would access freed storage. The cache entry (when one exists) also holds an independent reference per R8k-M2; the queue's reference is additive on top of both, and is always released by whoever holds the entry next (worker on commit/drop, or `apps_cgroups_lookup_cleanup` on shutdown drain).

## Lessons Extracted

- External plugins should not casually reuse main-binary RRD/database helpers. The apps-side label cache now stores copied strings directly, keeping the external plugin link boundary clean.
- APPS_LOOKUP needed one transient-state protocol relaxation: a live PID may be known before cgroup path attribution is available. Encoding that as `KNOWN + UNKNOWN_RETRY_LATER + empty path` is more truthful than returning outer `UNKNOWN`.
- Exact-count response checks are needed on clients even when producers promise exact-count replies; otherwise malformed peer responses can strand cache entries in `pending=true`.
