# Palo Alto Networks PAN-OS

Plugin: go.d.plugin
Module: panos

## Overview

This collector monitors Palo Alto Networks PAN-OS firewalls through the PAN-OS XML API.

It collects read-only telemetry for BGP, system status, HA, environment sensors, licenses, and active IPsec SAs. It does not expose arbitrary XML API commands or PAN-OS configuration, commit, log, report, import, export, or user-id operations.

## Setup

You can configure the **panos** collector in two ways:

| Method | Best for | How to |
|:--|:--|:--|
| UI | Fast setup without editing files | Go to **Nodes -> Configure this node -> Collectors -> Jobs**, search for **panos**, then add a job. |
| File | Configuration management or repeatable deployments | Edit `go.d/panos.conf` and add a job. |

### Prerequisites

- Network access from the Netdata Agent to the firewall management interface.
- PAN-OS XML API access enabled.
- A PAN-OS administrator role that can run XML API operational requests for the enabled metricsets.
- Either an existing XML API key, or username/password credentials that can generate one.

PAN-OS management planes have a low XML API concurrency budget. This collector uses serial requests and caps the SDK transport to 2 connections per firewall job.

The collector uses PAN-OS XML API operational commands. Palo Alto Networks documents this API under [Run Operational Mode Commands](https://docs.paloaltonetworks.com/pan-os/11-0/pan-os-panorama-api/pan-os-xml-api-request-types/run-operational-mode-commands-api).

#### File

```yaml
jobs:
  - name: firewall
    url: https://192.0.2.1
    api_key: YOUR_PANOS_XML_API_KEY
    update_every: 60
    timeout: 10
    collect_bgp: yes
    collect_system: yes
    collect_ha: yes
    collect_environment: yes
    collect_licenses: yes
    collect_ipsec: yes
    max_bgp_peers: 512
    max_ipsec_tunnels: 1024
```

Username/password key generation:

```yaml
jobs:
  - name: firewall
    url: https://192.0.2.1
    username: netdata
    password: YOUR_PASSWORD
    update_every: 60
    timeout: 10
```

Self-signed certificate:

```yaml
jobs:
  - name: firewall
    url: https://192.0.2.1
    api_key: YOUR_PANOS_XML_API_KEY
    tls_skip_verify: yes
```

Use `tls_skip_verify: yes` only for lab or temporary validation. Production firewall jobs should use a certificate trusted by the Netdata Agent host, or set `tls_ca`.

Cardinality selector example:

```yaml
jobs:
  - name: firewall
    url: https://192.0.2.1
    api_key: YOUR_PANOS_XML_API_KEY
    bgp_peers:
      includes:
        - "* default/*"
      excludes:
        - "* default/203.0.113.*"
```

## Troubleshooting

- `config: api_key or username/password must be set`: configure `api_key`, or both `username` and `password`.
- `force_http2 is not supported`: PAN-OS XML API requests use the pango SDK HTTP/1.x transport.
- `url path must be empty, /, or /api`: configure the management interface base URL, not another path.
- `no PAN-OS BGP peers found`: the collector authenticated successfully, but none of the legacy or Advanced Routing Engine BGP peer commands returned peers. Verify BGP is configured, the account can run operational commands, and the PAN-OS XML response shape is recognized by this collector version.
- A metricset fails but other charts work: the collector keeps successful metricsets running and logs the failing metricset name and XML command context. Disable unsupported metricsets with `collect_bgp`, `collect_system`, `collect_ha`, `collect_environment`, `collect_licenses`, or `collect_ipsec`.
- `success response has no recognized telemetry payload`: PAN-OS accepted the operational command, but the XML result did not contain the expected section for that metricset, such as `<system>`, `<licenses>`, or `<entries>`. This usually means the command is unsupported for that platform/account, the XML shape differs from the fixtures covered by this collector, or the metricset should be disabled for that firewall.
- `missing integer`, `missing decimal`, `missing duration`, `missing status`, `invalid integer`, `invalid decimal`, `invalid duration`, `invalid status`, or `invalid expiration date`: PAN-OS omitted or returned a value the collector cannot safely turn into a metric. The error includes the metricset, field, entity name, and raw value when present, so the fixture can be captured and the parser can be corrected.
- `IPsec active tunnel count mismatch`: PAN-OS reported an active tunnel count that differs from the number of tunnel entries returned. The summary count is still emitted from `<ntun>`, but per-tunnel lifetime charts may be incomplete.
- `PAN-OS XML API response error code ...`: PAN-OS rejected the operational command. The collector preserves the PAN-OS code and message, including nested `<msg><line>...` and `<result><msg>...` response forms.
- Some discovered entities are not charted: check the `*.collection` charts. If `omitted_by_limit` is non-zero, raise the matching `max_*` option or configure the matching selector.
- No charts after a previously working job: check the Netdata Agent error log for the detected routing engine and the BGP command that failed.
